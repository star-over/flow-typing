## Вердикт

Ядро решений (ADR 0001–0009, 0011) — **целостное и сильное**: заголовки-утверждения,
рассмотренные альтернативы почти везде, честные последствия (включая признания вроде
«на 501 drill выигрыша нет»), решения записываются в момент принятия, связаны с
планами и кодом. Линия «тонкий клиент — server-authoritative» (0005 → 0006 → 0008 →
0009 → 0011) — одна последовательная траектория без взаимных перекрытий: каждый
следующий ADR сужает контракт, не переигрывая предыдущий. Это выше средней
индустриальной практики.

Проблемы — в трёх других местах:

1. **Практика ведения**: нет статусов/дат, принятые ADR правятся задним числом
   (анти-паттерн «silent editing»), истина о решениях размазана по 5+ домам —
   и дрейф уже случился (см. §3).
2. **Асимметрия покрытия**: адаптивная петля задокументирована до зубов
   (10 из 11 ADR), а решения того же веса — платформа Convex, гостевой доступ,
   «провайдер = аккаунт», «cloud wins» sync — живут в планах и CLAUDE.md без
   статуса решения (см. §4).
3. **Три содержательные дыры**, где канон противоречит сам себе или коду:
   гость (три несовместимые позиции), обещанный outbox против неидемпотентного
   `drillRecord`, дефолтная раскладка qwerty без серверных данных (см. §4).

## 4. Дыры: решения, которых нет (или которые противоречат друг другу)

### 4.1. Гостевой доступ — три несовместимые позиции ⚑ главная развилка

- `docs/plans/auth.md`: «Гостевой режим — **поддерживается** (soft-progression)» —
  зафиксированное решение.
- ADR 0005 / auto-flow: гость = **анонимная identity**, создаётся лениво;
  регистрация = перенос профиля (запланированный этап).
- ADR 0011 (Consequences) + рабочие обсуждения: «пользователь склоняется
  **убрать** неавторизованный доступ — отдельное решение».

Три позиции одновременно числятся действующими. Пока висит развилка, гость живёт в
худшем из миров: печатает, но профиль не растёт (`drillRecord` бросает
'Not authenticated', ошибка молча гасится — `session-impl.ts:60-72`), то есть
«soft-progression» из auth.md уже фактически нарушен. Это первый кандидат на
grilling-сессию и ADR 0012 — любое из трёх направлений всё упрощает, текущий
разнобой трёх позиций сложнее любого из них.

### 4.2. Обещанный outbox × неидемпотентный `drillRecord` — скрытый конфликт

ADR 0005 обещает «маленький localStorage-outbox для неотправленных сводок
(at-least-once)». Реальность: обе записи — fire-and-forget с молчаливым гашением
(`session-impl.ts:69-85`), при сбое сводка теряется (at-most-once). При этом
`foldSummaryIntoCells` строго аддитивен, защиты от повторного учёта нет — **если outbox
однажды сделают по букве ADR, повторная доставка удвоит предъявления** и исказит
Readiness/термостат. Нужно решение: либо отменить обещание (поправка к 0005 —
«официально at-most-once, потери приемлемы»), либо проектировать outbox вместе с
идемпотентностью (checkpoint-id в payload). Сейчас же обещание просто мёртвое —
его нет ни в коде, ни в backlog.

### 4.3. Дефолтная раскладка qwerty не имеет серверных данных

`DEFAULT_USER_SETTINGS.symbolLayoutId = 'qwerty'`, MenuScreen предлагает
en→qwerty, но `convex/layoutData.ts` знает только `йцукен` (лестница одна —
`jcuken.ts`). По коду путь нового пользователя с дефолтами: `drillNext` →
`getLayoutData('qwerty') === null` → `{ drills: [] }` → `loading.onError`/пустой
поток → session сразу `done` → **пустой экран результатов**. ADR 0011 называет
раскладку без данных «config-багом, не нормой» — но этот config-баг сидит в
дефолтном пути продукта. Нужно решение: либо qwerty-лестница + корпус (контентная
работа), либо дефолт/выбор ограничивается раскладками с серверными данными
(реестр «поддержанных» раскладок как источник для UI). Проверить фактическое
состояние dev-базы стоит, но код детерминирован.

### 4.4. Решения ADR-веса, живущие вне лога решений

По критерию значимости (дорого обратить / lock-in / cross-cutting / явный
trade-off) статуса ADR заслуживают, но живут в планах и CLAUDE.md:

- **Convex как платформа** (профиль, корпус, auth, агрегаты — всё Convex-shaped).
  Самое дорогое для отката решение проекта вообще не имеет записи с альтернативами:
  оно просочилось через Phase-1 план. Не зафиксированы: cost-model на масштабе
  (каждый чекпоинт — мутация), задержки EU-West-1 для целевой аудитории, exit-стратегия.
- **«Провайдер = аккаунт»** (NO link-by-email) — инвариант с тестом-замком и
  UX-последствиями, живёт в таблице auth.md.
- **«Cloud wins при login»** + one-shot sync-стратегия настроек — живёт в CLAUDE.md.
- **SPA-only / adapter-static / никаких `*.server.ts`** — «правила игры» auth.md.

Ретро-ADR для стека (SvelteKit, XState) не нужны — решения стабильны и не
оспариваются; фиксировать стоит то, что ещё будет передумываться или дорого
обратить.

### 4.5. Данные печати: приватность и retention — решения нет

Проект копит: пер-символьные профили навыка, журнал сессий с путаницами, а для
настройки чисел — сырые нажатия («захват attempts», файлы на персону). Хостинг —
EU (GDPR-юрисдикция), аудитория — РФ (152-ФЗ). Ни в одном доке нет решения о
сроках хранения, удалении по запросу, анонимизации захвата. На текущей стадии
(dev, свои данные) не горит, но до «Фазы 2: рост базы» из docs/01 нужен хотя бы
однострочный ADR о том, что собираем и когда удаляем.

### 4.6. Контент ранних шагов — вопрос без хозяина (мягкая дыра)

KeyLadder открывает шаг 0 из 13 клавиш; корпус — реальные тексты. Есть контентный
радар (`ladderReport`) и серверный дефолт (0011), но нет решения, **что считается
достаточным покрытием** шага и что делать, если реальных слов на первые шаги
структурно мало (добавлять руками? генератор искусственных слов — «интерфейс на вырост»
из ADR 0003?). Радар — диагностика без порогов и без владельца процесса.

## 5. Практика ведения против лучших практик 2024–2026

Сверено с MADR 4.0, AWS Prescriptive Guidance, Microsoft Well-Architected,
arc42 §9 и публикациями 2025–2026 об ADR для AI-агентов.

**Что проект делает лучше индустрии:**

- Заголовки-утверждения, considered options, честные trade-offs — три из семи
  канонических анти-паттернов уверенно обойдены.
- ADR пишутся в момент решения (git-история это подтверждает), лежат в repo
  (docs-as-code), связаны с планами внедрения.
- CONTEXT.md с полями `_Avoid_` — это ровно рекомендованный для AI-агентов паттерн
  «DO-NOT-директивы из отвергнутых опций»: свежее исследование (arXiv:2605.08112, 2026)
  показало рост соблюдения агентами записанных решений с 46% до 95% при доступности
  такого контекста. Проект интуитивно построил то, что литература советует.

**Чем практика расходится с консенсусом:**

1. **Нет метаданных.** Ни в одном ADR нет Status/Date. Консенсус (Nygard, MADR,
   Microsoft): статус и дата — минимально обязательные поля даже для соло-проекта;
   остальное (deciders и пр.) для соло справедливо опущено.
2. **Правки задним числом («silent editing»).** ADR 0007 переписан на месте:
   первоначальное «Решение 3» с фазой `draining` заменено противоположным
   («истечение → сразу done») без следа в тексте — след только в git
   (`fa41b0b` → `0bf4d49`). Именно поэтому четыре дока (§3.1) до сих пор описывают
   похороненный дизайн: тихая правка не создала события «решение изменилось»,
   которое заставило бы пройтись по потребителям. Поправки «ADR 0008 правит 0006»
   сделаны лучше — двунаправленные пометки в обоих текстах — но и они вписаны в
   тело принятых записей. Консенсус: тело принятого ADR неизменно; смена курса =
   новая запись со ссылками supersede/amend в обе стороны.
3. **Нет индекса.** `docs/adr/` без README-оглавления; сводная таблица в
   auto-flow.md выполняет его роль, но принадлежит одному подпроекту и уже отстала.
4. **Нет шаблона** (`_template.md`) — формат держится на подражании соседям.
   Пока автор один — работает; для агентов шаблон дешевле, чем вывод формата
   из примеров.
5. **Текущая истина контракта собирается из цепочки.** Сигнатура `drillNext`
   эволюционировала через 0005 → 0006 → 0008 → 0009; актуальна только сумма
   четырёх документов (сведённая в CONTEXT.md). Это законный паттерн — если в
   старых ADR стоит машинно-заметная пометка «текущее состояние см. CONTEXT.md /
   ADR N». Сейчас пометки есть, но нерегулярные.

## 6. Необходимость и достаточность

- **Лишних ADR нет.** Каждый из 11 фиксирует реальную развилку с отвергнутой
  альтернативой. ADR 0010 — на грани порога значимости (UX-инвариант навигации),
  но он записывает неочевидное следствие ADR 0007 — оправдан.
- **Взаимных перекрытий нет.** 0001+0002+0008 (префикс + ручная лестница +
  монотонность) — согласованная связка; 0003+0004+0009 (заявка + термостат +
  двухслойный отбор) — согласованная; линия 0005→0011 последовательно хоронит
  local-first, и каждый шаг честно помечает, что именно уточняет.
- **Недостаточность** — вся в §4: гость, outbox/идемпотентность, платформа,
  auth-инварианты, приватность. Плюс два ADR (0003 частично, 0004 полностью)
  описывают ещё не построенное — это законно (решение раньше кода), но у них
  нет статуса, который бы это показывал: читатель не отличит «сделано» от
  «спроектировано» без похода в код. Статус `Accepted (not yet implemented)` /
  этап-ссылка снял бы вопрос.

## 7. Оценка решений по существу (с высоты)

- **0005/0011 (тонкий клиент, тотальный сервер)** — самая смелая линия; довод
  «сеть всё равно обязательна» честный, устранение merge-проблемы реально.
  Цена принята сознательно (офлайн мёртв), но UX сетевого сбоя не решён нигде:
  сегодня обрыв → молча пустой экран результатов (`loading.onError → done`).
  Плюс платформенные риски, о которых канон молчит (§4.4).
- **0001/0002/0008 (репертуар)** — крепко; «не выкидывать забытое из пула, а
  прорабатывать» соответствует и педагогике, и механике префикса. Ручная лестница
  не масштабируется на много раскладок — но раскладок две, и это правильный
  сегодняшний trade-off. Миграция лестницы описана, но ни разу не исполнялась —
  первый реальный `version: 2` станет проверкой.
- **0003/0004/0009 (адаптивный мозг)** — спроектирован до реализации; риск
  «ADR-впрок» смягчён валидацией на 100 реальных прохождениях (итог которой уже
  скорректировал 0004 — ритм понижен до вторичного сигнала). Явная политика
  «ADR фиксирует структуру, не значения» — зрелая; но тогда числам нужен один
  дом (сейчас: таблица в auto-flow.md с устаревшими значениями + правда в
  `shared/*/config.ts`).
- **0006/0007/0010 (клиентская петля)** — 0007 — образцовый ADR (глубина разбора
  альтернатив уровня учебника); портит его только способ, каким он менялся (§5.2).

## 8. Рекомендации

По убыванию (цена ↑, ценность решается драйвером):

1. **Санитарный проход дрейфа** (механика, ~час): §3 целиком — draining в
   4 доках, числа (90с→15с или пометка «см. код»), статусы этапов auto-flow,
   backlog (Phase 6/7, health, блок Session-vs-drill), known-issues #1 → закрыт,
   CLAUDE.md (/stats, draining), docs/04 (роут меню), docs/05 (гость, домашний
   ряд), ADR 0009 (atBatch → count/at).
2. **Каркас ведения** (разово, ~час): `docs/adr/README.md`-индекс (номер +
   заголовок + статус + одна строка сути); `_template.md` с полями Status/Date/
   Amended-by; проставить статусы задним числом по git-датам; зафиксировать
   политику: *тело принятого ADR не переписываем — смена курса = новый ADR или
   явная Amended-пометка*. Для 0003/0004 — статус «принято, не реализовано
   (этап 3/4)».
3. **Закрыть три содержательные дыры решениями** (каждая — короткая
   grilling-сессия + ADR):
   - **Гость** (§4.1) — снять разнобой трёх позиций: анонимный профиль / гость без
     петли / вход обязателен.
   - **Outbox × идемпотентность** (§4.2) — подтвердить at-most-once (поправить
     0005 новым решением) или спроектировать защиту от повторов.
   - **qwerty по умолчанию** (§4.3) — реестр поддержанных раскладок против
     контентной работы.
4. **Перенести в ADR решения ADR-веса** (по одному, когда трогается зона):
   платформа Convex (ретро, с exit-стратегией), «провайдер = аккаунт»,
   «cloud wins» sync. Не оптом — по драйверу «зона снова в работе».
5. **Один дом для чисел-настроек**: таблица auto-flow.md перестаёт хранить
   значения, хранит имя ↔ где в коде (`shared/*/config.ts`) ↔ как уточняем.
6. **Приватность данных печати** (§4.5) — до «Фазы 2» из docs/01 хватит
   однострочного решения о retention и анонимизации захвата.
