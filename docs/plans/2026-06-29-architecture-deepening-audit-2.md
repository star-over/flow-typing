## Очередь этапов

### Этап 1 — Слой чтения cloud-данных: шов реактивного запроса + вывод accuracy — MEDIUM–HIGH · ✅ СДЕЛАНО

Две части в одной зоне (сторы чтения + статистика), трогают пересекающиеся файлы
(`sessions-store`) — логично делать вместе. Часть 1A — основная (чистый
shallow→deep), 1B — попутная уборка.

#### 1A. Lifecycle реактивного Convex-запроса продублирован по трём сторам · ✅ СДЕЛАНО

**Файлы:** `src/lib/sessions/sessions-store.svelte.ts:47–76` ·
`src/lib/repertoire/repertoire-store.svelte.ts:28–51` ·
`src/lib/repertoire/progression-store.svelte.ts:15–37`

**Проблема.** Три стора несут дословно один и тот же жизненный цикл подписки:

```
let data = $state(unauthValue);
$effect(() => {
  if (authStore.state.status !== 'authenticated') { data = unauthValue; return; }
  const unsubscribe = convex.onUpdate(api.X, { symbolLayoutId: symbolLayoutId() }, r => { data = r; });
  return () => unsubscribe();
});
```

Каждый стор сам по себе **shallow** — интерфейс (`{ get value() }`) почти равен
реализации, поверх `convex.onUpdate` ничего не добавляет, кроме доменных довесков
(`didStepGrow`/`markSessionStart` у репертуара, `formatSessionRow` у сессий). Но
тонкий **инвариант корректности** — «гость сбрасывается на unauth-значение;
подписка чистится; пере-подписка при смене `symbolLayoutId`» — не имеет одного
дома: скопирован трижды. Правка в одном (обработать `loading`-статус; сбрасывать
не в `[]`, а иначе) молча не доедет до двух других. Различия между сторами лишь
параметрические: unauth-значение (`[]` у сессий vs `null` у снимков) и доменные
довески. Deletion-тест шва: удали гипотетический общий шов — auth-gate + onUpdate
+ cleanup-танец всплывает у трёх вызывающих → шов зарабатывал бы хлеб.

**Решение (направление, не интерфейс).** Поднять lifecycle за один глубокий шов
«auth-gated reactive Convex query»: владеет инвариантом подписки / сброса для
гостя / очистки / пере-подписки при смене аргументов. Каждый стор становится
тонким: шов + только его доменное. `Repertoire`, снимок прогресса и детальный
прогресс ступени остаются разными концептами — общим становится лишь механика
«cloud-данные → auth-gated реактивное значение».

**Выгода.** Locality: инвариант auth-gate + cleanup в одном месте и впервые
становится тестовой поверхностью (сейчас покрыты только чистые
`formatSessionRow`/`didStepGrow` в `*.test.ts`, сама подписка — никак). Leverage:
четвёртый такой стор тривиален; знание «как живая Convex-подписка становится
реактивным Svelte-значением для гостя/юзера» в одном доме, а не в трёх.

**Перед исполнением (grilling).** Куда селится шов (отдельный модуль vs вспомогательный модуль
рядом с `convex.ts`); реактивны ли аргументы через геттер (как сейчас
`symbolLayoutId()`); как сторы с доп-состоянием (`startStep` у репертуара)
строятся поверх шва; что именно покрывает новый тест (guest-reset, cleanup,
re-subscribe при смене раскладки).

**Сделано (2026-06-29).** Шов разбит на два дома по образцу `auth-state.ts` /
`auth-store.svelte.ts`: чистое ядро инварианта `runAuthGate` (`src/lib/auth-gate.ts`,
без runes — тестируется в node) + тонкая runes-привязка `createAuthGatedQuery`
(`src/lib/auth-gated-query.svelte.ts`, владеет `$state`/`$effect`). Источник подписки
внедряется (DI) как `subscribe: (onResult) => convex.onUpdate(query, args(), onResult)`
— ядро не знает про Convex (тот же приём, что `attachCloudSync` с `pullCloud`/`pushCloud`);
аргументы реактивны, потому что `subscribe` читает `symbolLayoutId()` внутри `$effect`
(пере-подписка при смене раскладки сохранена). Три стора стали тонкими: шов + только
доменное (`startStep`/`grew`/`markSessionStart` у репертуара поверх шва). Тест
`auth-gate.test.ts` впервые покрывает инвариант (guest/loading-reset, одна подписка при
authenticated, доставка значения через подписку, cleanup=отписка). Поведение-сохраняющий;
прежние 505 тестов зелёные. **Решение по тестируемости:** `$effect` в node-режиме vitest
не исполняется (Svelte компилируется в server-mode, эффекты вырезаются), а jsdom в
проекте нет — поэтому инвариант вынесен в чистую функцию, а не тестируется через рунную
обёртку (jsdom + `resolve.conditions:['browser']` сочли раздуванием scope). Ветка
`refactor/auth-gated-query-seam`.

**Статус:** ✅ СДЕЛАНО.

#### 1B. Вывод accuracy дублируется между карточкой результатов и строкой /stats — LOW · ✅ СДЕЛАНО

**Файлы:** `src/lib/stats-calculator.ts:21` ·
`src/lib/sessions/sessions-store.svelte.ts:29`

**Проблема.** Формула `(clean / exposures) * 100` живёт дважды и держится в
синхроне руками — комментарий в `stats-calculator.ts` буквально гласит «ровно как
formatSessionRow». Это НЕ нарушение ADR 0007 (канонические числа
`SessionSummaryPayload` считаются один раз на `done`), а дублирование
**презентационного вывода** поверх канона. Deletion-тест слабый: вывод тривиален,
сложность почти не концентрируется — потому LOW и «заодно», а не отдельная сессия.

**Решение.** Один чистый вывод производных процентов (accuracy) поверх
канонической сводки; карточка и строка `/stats` читают одну формулу, а не
синхронизируются комментарием.

**Выгода.** Скромная: одна формула — один дом.

**Сделано (2026-06-29).** Формула вынесена в чистую `accuracyPercent({ exposures, clean })`
в `src/lib/stats-calculator.ts` — единственный дом. `lessonStatsFromSummary` (карточка
результатов) и `formatSessionRow` (строка /stats) читают её, а не синхронизируются
комментарием; header-комментарий `stats-calculator.ts` обновлён. Прямые тесты на
`accuracyPercent` добавлены; прежние тесты обоих потребителей зелёные. Ветка
`refactor/auth-gated-query-seam`.

**Статус:** ✅ СДЕЛАНО.

### Этап 2 — `contentGap`: один дом политики пустого пула + склейка пробелами — MEDIUM · ✅ СДЕЛАНО

Подтверждение и обогащение кандидата 7 первого прохода. Зона —
сервер↔клиент↔конвейер потока, шире и горячее этапа 1 → делать после него.

**Файлы:** `convex/drill.ts:92–96,121` (сигнал `contentGap`) ·
`src/machines/session-impl.ts:55–66` (решение клиента) ·
`src/lib/drill-stream.ts:14–26,53–65` (`glueDrillsIntoStream` / `glueServerDrills`) ·
`src/lib/drill-selection.ts:31–34` (`selectRandomDrill`) ·
`src/machines/session.machine.ts:172–176` (сепаратор на стыке порций)

**Проблема.** Сервер **сигналит** «пул пуст» успешным ответом
`{ contentGap: true, drills: [] }` (дважды: при `count === 0` и при сплошь битых
ссылках), а клиент (`fetchDrills`) **решает** деградировать на локальный корпус.
Политика «контентного радара» (ADR 0002) не имеет одного дома: половина на
сервере, половина в `session-impl`. Вокруг — стопка pass-through
(`glueServerDrills`, `glueDrillsIntoStream`, `selectRandomDrill`), в основном
переименовывающих поля. **Новая грань (не было в #7):** правило «как drill'ы
склеиваются пробелом в `TypingStream`» само расщеплено надвое — **внутри** порции
пробелы ставит `drill-stream.ts:21`, а **на стыке** порций отдельный сепаратор
клеит `session.machine.ts:172`. Два дома у одного правила «границы drill'ов
невидимы, но разделены пробелом»; смена политики (двойной пробел, no-space на
стыке) требует правки обоих.

**Решение.** Свести политику пустого пула в одно место (сервер гарантирует
дефолтный домашний-ряд drill **или** клиентский fallback оформлен одним явным швом
«источник drill'ов»); склейку-пробелами слить в одну точку истины, потребляемую и
refill'ом, и стыком порций.

**Выгода.** Один ответ на «откуда берётся drill, когда подбор пуст» и «где
ставятся пробелы между drill'ами»; меньше shallow-слоёв между ответом сервера и
`TypingStream`. Связан с зоной сводок (#1 первого прохода).

**Перед исполнением (grilling).** Чей дом политики — сервер (гарантирует дефолт,
снимает ветку с клиента) или клиент (явный шов «источник drill'ов», сервер только
сигналит); сохраняем ли `contentGap`-флаг в контракте или заменяем
полиморфным ответом; куда селится склейка-пробелами (один модуль, потребляемый и
`glueDrillsIntoStream`, и `appendFetched`). НЕ пересматривать ADR 0009 (двухслойный
отбор, `drillNext` = query+seed) и CQRS-границу (drillRecord — единственный
писатель) — целы.

**Сделано (2026-06-29).** Grilling вскрыл, что код противоречил доменной модели
глубже, чем «два дома политики»: на клиенте жил **второй** корпус drill'ов
(`DRILL_CORPUS`, встроенный из `drills.jsonl?raw`), на который шла деградация
при contentGap и недоступности сервера. По решению пользователя выбран не «клиентский шов», а
**удаление клиентского корпуса совсем** + **тотальный сервер**: `drillNext` всегда
возвращает непустой `drills` для раскладки с серверными данными, на контентный сбой
(пустой пул / битые ссылки) сам строит дефолтный drill из символов открытых шагов
(`buildDefaultDrills` через `symbolsAtStep`, источник истины KeyLadder). Контракт
ужат до `{ drills: [{ text }] }` (убраны булев `contentGap` и per-drill `id`/`length`).
Удалены `drill-corpus.ts`, `drill-selection.ts`, `integrity.ts` (load-time guard
покрытия — его повод поглощён серверным дефолтом), `fetchLocalDrillStream` и обе
ветки деградации в `session-impl.ts`. Склейка-пробелом сведена в один дом
`drillSeparatorStream` (+ `joinBatchToStream` для стыка порций), потребляемый и
внутри порции, и `appendFetched`. Офлайн на этом этапе не поддержан (сервер
недоступен → `done`). Гостевой доступ НЕ тронут — пользователь склоняется убрать и
его, но это отдельная крупная задача (auth, ADR 0005). Зафиксировано в **ADR 0011**;
`CONTEXT.md` обновлён. Поведение меняется только в contentGap-пути (был богатый
локальный корпус → стал дефолт открытых шагов) — сознательно, к букве ADR 0002.
Прежние тесты зелёные (496); развёртывание проверено `npx convex dev --once`. Ветка
`refactor/drill-source-server-total`.

**Статус:** ✅ СДЕЛАНО.

## Проверено и счёл здоровым / отклонил

Чтобы будущие проходы не пере-предлагали:

- **Ритм-канал (`src/lib/rhythm-channel.ts`, `RhythmChannel.svelte`) — здоров, не
  трогать.** Explore поднял «трещину паузы» (`PAUSE_FLOOR_MS=4000` vs
  `exposure-reading` `PAUSE_CAP_MS=1500`) и «UI считает интервалы через
  `performance.now()`». Прочитан целиком: это **два законно разных концепта** —
  живой визуальный метроном (когда нажатие ещё «в ритме» для падающей шкалы)
  против латентности профиля (что вычесть из числа для термостата). Объединять —
  ошибка. Физика уже вынесена в чистый модуль (источник истины —
  `docs/research/rhythm-visualization.md`), полностью покрыта `*.test.ts`;
  компонент владеет rAF — правильно для живой анимации.
- **`areKeyCapIdArraysEqual` в 6 местах — leverage, не трещина.** Единый чистый
  предикат «нажатый аккорд == цель», вызываемый из 6 мест, — это глубина, а не
  «low locality». У него один дом.
- **Классификация Space «в 3 местах»** — задокументированный gotcha в CLAUDE.md
  (клавиатурная сцена делит пробел надвое); Space-логика `hands-scene` уже
  отложена в `docs/backlog.md` первым проходом (#3). Известно/отложено.
- **`symbol-utils.ts` cohesion, `inState`, `integrity.ts` (side-effect на import),
  Zod-схемы в `types.ts`** — всё уже в разделе «Также замечено / микро-обёртки»
  первого прохода; вопрос cohesion, не глубины. Подтверждаю открытым, заново не
  выношу.
