# Auto-Flow — план внедрения адаптивной выдачи drill

Umbrella-план системы **Auto-Flow**: подбор упражнений под уровень пользователя, вся логика на сервере (Convex). **Зачем** и **каким языком** — в [`CONTEXT.md`](../../CONTEXT.md) (глоссарий) и [`docs/adr/0001..0011`](../adr); здесь — **что и в каком порядке строить**.

При расхождении плана с ADR — доверять ADR.

## Этапы сборки

Сборка идёт от простого к умному. Каждый этап — отдельная ветка и отдельный merge; детальный план каждого пишем перед стартом (`writing-plans`).

| Этап | Что даёт | Статус | Ветка |
| --- | --- | --- | --- |
| **1. Рабочая петля** | сквозная тренировка end-to-end + реальный корпус в базе + сбор данных печати | ✅ реализовано ([`docs/plans/2026-06-17-session-machine.md`](2026-06-17-session-machine.md)): `sessionMachine` (таймер, очередь, refill, чекпоинт-запись), непрерывный `TypingStream` | `docs/auto-flow` |
| **2. Рост набора букв** | система решает, *сколько* букв давать | ✅ реализовано ([`2026-06-22-auto-flow-repertoire.md`](2026-06-22-auto-flow-repertoire.md), ADR 0008): Readiness + монотонный рост `openedSteps` у писателя `drillRecord` | `feat/auto-flow-repertoire` |
| **3. Фокус на слабых местах** | система решает, *какие* упражнения давать | 📋 запланировано | `feat/auto-flow-focus` |
| **4. Авто-подстройка трудности** | система решает *трудность* | 📋 запланировано | `feat/auto-flow-thermostat` |
| **Анонимный профиль** | единственный путь возврата гостя (ADR 0012): ленивая анонимная учётная запись + перенос профиля при регистрации (ADR 0005). До него тренировка **только для авторизованных**; драйвер — публичная альфа | 📋 запланировано (отдельно) | `feat/auto-flow-anon-profile` |
| **Уборка** | снос старого кода (docs/05 уже переписан) | 📋 запланировано | `chore/auto-flow-cleanup` |

Сквозь все этапы идёт **сбор реальных данных о печати** и настройка по ним чисел (см. «Числа-настройки»).

## Зафиксированные решения (→ ADR)

| Решение | Где | Суть |
| --- | --- | --- |
| Репертуар = число открытых шагов KeyLadder (`openedSteps`) | ADR 0001 | одно число, индексируемое сравнение |
| KeyLadder — по геометрии раскладки (пальцам), ручные шаги | ADR 0002 | не по частотности; версия |
| Подбор — заявкой требований, не оценкой сложности | ADR 0003 | сложность нигде не хранится числом |
| Зона потока — лексикографически, приоритет точности | ADR 0004 | без сводного «балла успеваемости» |
| Профиль и логика — на сервере, клиент тонкий | ADR 0005 | один писатель, одна истина |
| Контракт выдачи — в символах, cpm на клиенте | ADR 0006 | размер + фильтрация в одном слое |
| Таймерная сессия — отдельной машиной; таймер охватывает running+refilling | ADR 0007 | три машины по времени жизни + один писатель; `timing`-обёртка |
| Конвейер параметризован раскладкой | ADR 0001 (уточнение) | проверка «символы ⊆ набор раскладки», не определение языка |
| Рост репертуара — монотонный, у писателя (CQRS) | ADR 0008 | `openedSteps` только вверх; читается из профиля, не с провода |
| Случайный отбор — агрегат (hard) + soft-shaping; `drillNext` — query | ADR 0009 | свежесть через `seed`; soft-слой отложен до реального сигнала |
| `drillNext` тотален; клиентского корпуса нет | ADR 0011 | на контентный сбой сервер сам строит дефолтный drill из открытых шагов |
| Доля фокусных упражнений — цель, не квота | CONTEXT.md | при нехватке кандидатов добор обычными |

## Правила игры (инварианты, держатся во всех этапах)

1. **Раскладка — параметр, не вывод.** Конвейер и подбор опираются на `symbolLayoutId`; язык не определяется автоматически — упражнение совместимо с раскладкой ⟺ его символы входят в набор символов раскладки.
2. **Сложность не материализуется.** Нигде нет хранимого числа «сложность» (ADR 0003).
3. **На сервер уходят только сводки.** Сырые нажатия (attempts) — данные для офлайн-анализа и тестов, не часть рабочего обмена с сервером (ADR 0005).
4. **Единственное жёсткое требование подбора — набор введённых букв.** Фокус, длина слова, анти-повтор — мягкие, решаются ранжированием.
5. **Старый код сносим только когда есть замена** — раньше нельзя, на него ссылается работающее приложение, сборка сломается.

| **Чистые функции** | сводка по завершённому упражнению; критерий готовности буквы; карта слабостей; определение зоны (фрустрация/поток/скука); сборка заявки на следующую порцию | Справочники классификации | src |

## Порядок зависимостей

```
Схемы базы → KeyLadder → { Конвейер корпуса, Справочники классификации }
Чистые функции          — почти независимы, начинаем рано
Серверные функции       — после: Схемы базы + KeyLadder + Чистые функции
Клиент                  — после: Серверные функции
Сбор данных             — как только Клиент пишет нажатия → питает настройку чисел
Уборка                  — в любой момент, по мере замен
```

## Этапы подробно

### Этап 1 — Рабочая петля

**Цель.** Завести сквозную тренировку: вход → упражнения из базы → буфер → тренировка по таймеру → сводка → запись в профиль → профиль сохраняется. И **заодно получить реальный корпус в базе и реальные данные печати**, чтобы проверить наши прикидки до постройки умных частей.

**Что собираем:** Схемы базы + KeyLadder (рукотворная лестница) + Конвейер корпуса (прогнать на текущих текстах → реальная мета в базе) + Серверные функции в минимуме (анонимный вход; выдача = фильтр по набору букв + **случайный выбор**; приём сводки) + сводка по упражнению + Клиент в минимуме (таймер + буфер) + захват нажатий.

**Что заглушено:** набор букв фиксированный; ничего умного — нет ранжирования, фокуса, подстройки трудности; профиль наполняется, но пока ничего не выбирает.

**Готово, когда:** тренировка идёт минутами на упражнениях из базы; профиль растёт в базе; нажатия выгружаются в набор данных.

### Этап 2 — Рост набора букв

**Что собираем:** критерий готовности буквы (Чистые функции) + рост набора букв на сервере (выдача динамически учитывает, сколько букв уже введено).

Система начинает решать, *сколько* букв давать. Все новички стартуют одинаково — ручки трудности на минимуме, старт с домашнего ряда; опытный быстро проходит начало (ускоренная лестница).

**Границы (зафиксированные решения, см. ADR 0008):**
- **Рост только вверх.** `openedSteps` монотонно неубывающий; регресс навыка не откатывает репертуар (его гасят фокус + термостат на других осях). Снижение репертуара исключено как механизм.
- **CQRS.** Рост живёт у писателя: `drillRecord` после `apply` сводки проверяет Readiness и поднимает `openedSteps`. `drillNext` — чистый читатель: берёт `openedSteps` **из профиля** (не с провода — уходит из аргументов, правка ADR 0006), фильтрует пул. Смешение ответственности запрещено.
- **Предохранитель — долговой лимит, без прицельной подачи.** Рост опирается на случайную выдачу + долговой лимит (1–2 недозревших не блокируют шаг). Прицельная подача недозревших (фокус на неподтверждённых) — этап 3, не здесь.
- **Только для авторизованных.** Гость без профиля: `drillNext` отдаёт cold-start ступень, запись пропускается. Анонимный профиль — отдельный этап.

### Этап 3 — Фокус на слабых местах

**Что собираем:** Справочники классификации + карта слабостей (Чистые функции) + ранжирование на сервере (доля фокусных упражнений, анти-повтор).

Система решает, *какие* упражнения давать. При нехватке подходящих упражнений деградация плавная (доля фокусных — цель, не квота); совсем пустой результат = дыра в корпусе + запасное упражнение.

### Этап 4 — Авто-подстройка трудности

**Что собираем:** определение зоны (Чистые функции) + ручки трудности на сервере (пока одна — целевая длина слова) + «свежее окно» в профиле.

Система решает *трудность*. Зона определяется по правилам с приоритетом точности; вниз быстро, вверх медленно, с задержкой на несколько окон (ADR 0004).

## Числа-настройки

Адаптивной системе нужно ~десяток чисел. Хорошие значения узнаём по реальным данным печати (участок «Сбор данных»). Значения, уже зафиксированные в коде, помечены ссылкой на константу — **истина о числе всегда в коде**; остальные задаются на своём этапе, уже с накопленными данными.

| Число | Где используется | Временное значение | Как уточним |
| --- | --- | --- | --- |
| Сколько упражнений слать в браузер за раз | размер порции | окно ≈ **15 секунд** печати (`BATCH_WINDOW_SECONDS`, `src/lib/batch-budget.ts`) | эргономика, правим по ощущению |
| Когда подгружать следующую порцию | дозагрузка буфера | хвост < **40 символов** (`REFILL_THRESHOLD_SYMBOLS`, `src/lib/session-config.ts`) | то же |
| Сколько раз набрать букву, чтобы поверить замеру | готовность буквы | **20** (`minExposures`, `shared/repertoire/config.ts`) | средняя — смешаны люди; уточнить на чистом сборе |
| Порог first-try accuracy для готовности | готовность буквы | **0.9** (`minFirstTryAccuracy`, там же) | средняя |
| Множитель латентности для готовности | готовность буквы | **1.5× собственной медианы** (`latencyK`, там же) | сред.-высокая (метод), средняя (k) |
| Сколько «несозревших» букв терпим (долговой лимит) | расширение набора | **2** (`REPERTOIRE_DEBT_LIMIT`, там же) | то же |
| Коэффициент затухания ячеек профиля | приём сводки | латентность: EWMA **α=0.3** (`LATENCY_EWMA_ALPHA`, `convex/drill.ts`); затухание счётчиков не реализовано | то же |
| Доля упражнений, целящих в слабые места | подбор | — | этап «Фокус», по данным |
| Сколько слабостей брать в фокус | подбор | — | то же |
| Пороги входа/выхода слабости | карта слабостей | — | то же |
| Порог точности «мучается / комфортно» | зона / подстройка | **<0.85 / ≥0.92** (общая ~0.94) | средняя |
| Доля ошибок = проскок (вычесть до метрик) | сводка | **~45%** | высокая |
| Сколько окон подряд держать зону перед шагом | подстройка трудности | — | то же |
| Размер «свежего окна» | подстройка трудности | — | то же |
| Шаг ручки трудности | подстройка трудности | — | то же |

### Валидация на реальных данных (100 прохождений, смешаны люди/раскладки)

Четыре независимых анализа сошлись на выводах:

- **Подтвердилось:** относительная латентность — надёжный сигнал (ранжирование символов воспроизводится, split-half ≈0.78), абсолютная — нет (скачет ×2.5–3 между людьми); точность — сигнал зоны на агрегате; разреженность пар букв требует более грубого объединения; слабости layout-специфичны.
- **Пересмотрено:** per-символьная слабость берётся по **латентности** (точность per-символ ненадёжна без 50–80 предъявлений); **ритм вторичен** (шумный, сильно зависит от человека).
- **Новое:** **проскок** ≈ 45% всех ошибочных нажатий — `summarize` обязан вычитать (настоящая ошибка = первое ошибочное нажатие); **интерференция раскладок** — самый воспроизводимый сигнал слабости, кандидат на ранний учёт (см. `CONTEXT.md`).

Числа в таблице — **провизорные**; финал — на чистом сборе по одному человеку: **отдельный файл на персону, ~30 предъявлений/символ, раскладки раздельно, дедупликация текстов** (`userId` в захвате не нужен — разделяем файлами).

## Открытые решения

- ✅ **`textLanguage` / индекс `by_language`** (закрыто) — **убираем и поле, и индекс из `drills`**: подбор идёт по раскладке, язык выводим из символов и дублирует карту «раскладка → язык» в `src/lib/layouts`. Определение языка тоже убираем — его заменяет проверка «символы ⊆ набор раскладки». Настройку пользователя `userSettings.textLanguage` **не трогаем** (это вход в UI: язык → дефолтная раскладка → на сервер уходит `symbolLayoutId`).
- ✅ **Числа-настройки** (закрыто) — сейчас фиксируем только два значения первого этапа (см. таблицу выше), остальные задаём на своём этапе по реальным данным.
- ✅ **Порядок открытия клавиш** (закрыто) — KeyLadder по геометрии/пальцам ручными шагами (ADR 0002); частотный словарь не нужен.
- **Отложено на потом:** ручка частотности слова, ручка моторной нагрузки (доля движений одним пальцем), учёт путаниц «букву X набрал как Y».

## Как стартует каждый этап

Перед стартом этап детализируется через `writing-plans` в отдельный `docs/plans/<дата>-auto-flow-<этап>.md`:

- **Цель** — одно предложение
- **Границы** — что входит, что нет
- **Ветка** — `feat/...` или `chore/...`
- **Изменения файлов** — что создаётся / правится / удаляется
- **Шаги** — по порядку, с привязкой к коду
- **Тесты** — какой проект (`src` / `convex`), что покрываем
- **Готово, когда** — проверяемые критерии
- **Проверка** — `make check-all` обязательно + ручная
- **Merge** — `git switch master && git merge --no-ff feat/<этап>`
