# Глава 6: Темы и компонентные роли

Этот документ описывает архитектуру визуальной системы FlowTyping: как устроена тема, к чему привязываются компоненты и какой инвариант поддерживает контракт-тест.

Модель — **двухслойная** (ADR 0029). Прежний третий слой — компонентные контрактные токены `--<компонент>-*` и файлы `*.contract.ts` рядом с компонентами — был транзитными лесами и **растворён**: компоненты теперь ссылаются на роли напрямую. Слово «контракт» в главе живёт в двух смыслах — историческом (растворённый слой L3) и живом (контракт-тест `contract.test.ts`, который держит инвариант тем). Оба помечены явно.

## 6.1. Принцип

Тема в FlowTyping — это **приватная палитра плюс её раскладка по семантическим ролям**. Устроена в два слоя:

> **Ядро (L1)** — закрытая палитра темы: абсолютные значения `oklch(…)` под голыми именами (`--parchment`, `--ink`, `--amber`). Единственное место, где в теме появляется новый цвет.
> **Роли (L2)** — семантика: фиксированный словарь имён `--color-*`, общий для всех тем. Каждая роль — маппинг или калькуляция от ядра либо от другой роли; новых абсолютов на этом слое нет.

Компоненты привязываются к ролям L2 **напрямую** — `var(--color-*)` в собственном `<style>`. Отдельного слоя «значений компонента» больше нет.

Из этого следуют три практических правила:

1. **Имя роли — это «какая роль в UI», а не «какого цвета».** `--color-keycap-group-2-background` — это «заливка клавиш под указательным пальцем». `--amber` — палитровая переменная ядра темы, в словарь ролей она не входит.
2. **Геометрия живёт в компоненте, цвет — в роли.** Роль несёт только цвет (`var(…)` или `oklch(from var(…) …)`). Структурные решения — толщина рамки, отступ кольца, геометрия тени — записаны в CSS компонента как литералы: `border: 1px solid var(--color-keycap-group-1-border)`, `box-shadow: inset 0 0 0 0.14rem var(--color-route-1), 0 0 0 0.15rem var(--color-gap)`. Тема красит, компонент формирует.
3. **Темы независимы между собой.** Каждая тема — самостоятельный snapshot: ядро приватно (свои абсолютные цвета), роли — общий интерфейс. Что внутри ядра — выбор темы; контракт-тест проверяет, что весь словарь ролей объявлен, а не какие в нём значения.

## 6.2. Где живёт темизация

Двухслойная модель раскладывается на два места в коде:

```
src/themes/roles.ts        ← ROLE_DICTIONARY: единый интерфейс L2 (73 роли)
src/themes/sepia.css        ┐
src/themes/light.css        │  значения ядра (L1) + ролей (L2) на тему;
src/themes/dark.css         │  селектор :root[data-theme="<id>"]
src/themes/nord.css         ┘
src/themes/_template.css    ← скелет новой темы: каждая роль как `unset`
```

- **`src/themes/roles.ts`** — источник истины словаря ролей. Массив имён `--color-*` с литеральными типами; каждая тема обязана объявить каждое имя из него:

  ```ts
  export const ROLE_DICTIONARY = [
    '--color-background',
    '--color-text-primary',
    '--color-keycap-group-2-border',
    // …73 роли
  ] as const satisfies readonly `--color-${string}`[];
  ```

- **`src/themes/<id>.css`** — конкретные значения для одной темы: сперва ядро (абсолюты), затем роли (маппинг/калькуляция от ядра). Эталон структуры — `sepia.css`.

Файлов `*.contract.ts` рядом с компонентами **больше нет** — они удалены вместе с агрегатором `src/themes/contract.ts` (`THEME_CONTRACT`). Компонент объявляет свои цветовые нужды тем, что ссылается на роли: `var(--color-*)` прямо в `<style>`. Никакого промежуточного файла-декларации.

## 6.3. Принципы именования ролей

Роль именуется по **семантике** — что она значит в UI, а не в каком элементе появляется и не каким CSS-свойством применяется.

| Шаблон | Когда используется | Пример |
|---|---|---|
| `--color-<роль>` | базовая семантическая роль | `--color-background`, `--color-text-primary`, `--color-error` |
| `--color-<домен>-<роль>` | роль внутри доменной группы | `--color-keycap-group-2-background`, `--color-symbol-one-error`, `--color-finger-3` |
| `--color-<роль>-<суффикс>` | вариация базовой роли (см. словарь суффиксов) | `--color-surface-hover`, `--color-error-dim`, `--color-border-accent` |

Словарь суффиксов (источник истины именования, ADR 0029):

- `-foreground` — текст/символ поверх плотной заливки;
- `-hover` — hover-состояние роли;
- `-dim` — приглушённая полупрозрачная поверхность (альфа над фоном);
- `-strong` / `-muted` — усиленная / приглушённая ступень;
- `-raised` — приподнятая поверхность;
- `-accent` — тёплый / акцентный вариант;
- `-border` — граница.

Аббревиатуры раскрыты по конвенции проекта: `bg` → `background`, `fg` → `foreground`. Роль под префиксом `--color-` не называет текст CSS-свойством (`--color-*-color` — масло масляное), поэтому текст на плотной заливке несёт семантический суффикс `-foreground`, а не `-color`.

## 6.4. Два слоя, читаются по префиксу

Тема — самостоятельный CSS-файл `src/themes/<id>.css` с селектором `:root[data-theme="<id>"]`. Слой каждого токена **читается из префикса имени** (инвариант, ADR 0029):

- **голое имя = ядро (L1)** — приватная палитра, абсолютные `oklch(…)`;
- **`--color-*` = роль (L2)** — семантика, фиксированный словарь, значение только `var(…)` или `oklch(from var(…) …)`.

Третьего префикса больше нет: слой компонентных контрактов `--<компонент>-*` (L3) **растворён** (ADR 0029). Компоненты, которым раньше отвечал контракт, теперь ссылаются на роли L2 напрямую.

Форма темы (фрагмент `sepia.css`):

```css
:root[data-theme="sepia"] {
  color-scheme: light;

  /* Слой 1 · Ядро — приватная палитра sepia, только абсолюты */
  --parchment : oklch(0.93 0.04 80);
  --ink       : oklch(0.30 0.04 60);
  --green     : oklch(0.60 0.13 145);
  /* … */

  /* Слой 2 · Роли — маппинг на ядро или калькуляция от него */
  --color-background : var(--parchment);
  --color-text-primary : var(--ink);
  --color-success   : var(--green);
  --color-success-dim: oklch(from var(--green) l c h / 0.16);
  /* … */
}
```

Внутри темы могут жить любые `--*` под голым именем — это ядро, приватная кухня темы. Тест не диктует, сколько их и как их звать; он держит **дисциплину слоёв**: в ядре нет `var()`, роли — только ссылка или калькуляция от ссылки (§6.6).

## 6.5. Глобальный `:root` и body

`src/app.css :root` содержит **только то, что не тематизируется** — primitives:

- typography (`--font-sans`, `--font-mono`, `--font-size-*`, `--line-height-*`, `--font-weight-*`),
- radius (`--radius`, `--radius-N`),
- spacing (`--spacing`, `--spacing-N`),
- elevation / shadow (`--shadow-*`),
- motion (`--motion-duration-*`, `--motion-ease-*`).

Эти токены общие для всех тем и от продуктовой палитры не зависят.

**Body-fallback больше нет.** Раньше `:root` держал `--body-background` / `--body-color` как литералы — на тонкое окно между paint и bootstrap-скриптом. Теперь `body {}` ссылается на роли напрямую:

```css
body {
  font-family: var(--font-sans);
  background: var(--color-background);
  color: var(--color-text-primary);
}
```

Это безопасно, потому что синхронный head-скрипт в `src/app.html` выставляет `data-theme` **до** первого применения CSS — окна без темы не существует, страховать нечем. Роли `--color-*` определены только внутри выбранной темы; вне темы они не заданы, но этого состояния пользователь не видит.

## 6.6. Контракт-тест

`src/themes/contract.test.ts` — единая точка проверки инварианта тем. После растворения L3 он держит именно двухслойную модель. Что проверяется:

1. **Registry ↔ файлы.** Список `*.css` в `src/themes/` (кроме `_template.css`) совпадает с `THEMES` из `registry.ts`; обязательные темы `light`/`dark` объявляют свой `color-scheme`.
2. **Объявление словаря ролей (L2).** Каждая тема **и** `_template.css` объявляет **каждую** роль из `ROLE_DICTIONARY`. Значения свободны (в шаблоне допустим `unset`). Плюс: заявленный в registry `color-scheme` присутствует в файле темы.
3. **Дисциплина слоёв** (на реальных темах `sepia`/`light`/`dark`/`nord`):
   - **L1:** любой не-`--color-*` токен (ядро) **не содержит `var()`**. Заодно ловит оставшийся остаток L3 — ядерное имя со значением-`var()` было бы не растворённым до конца контрактом.
   - **L2:** каждая `--color-*` начинается с `var(` либо `oklch(from var(` — только ссылка или калькуляция от ссылки, никаких абсолютов на слое ролей.
4. **Ссылки на роли разрешаются.** Всякий `var(--color-*)`, встреченный внутри значений темы, объявлен в этой же теме — висячих ссылок нет.
5. **Нет дублей и циклов.** В теме нет повторных объявлений одного `--*` имени и нет циклов в графе `var()`-ссылок (обход в глубину с тремя цветами).
6. **Компоненты ссылаются только на валидные роли, остатка L3 нет.** Скан `src/**/*.svelte` + `src/app.css`:
   - ни один `var(--color-*)` не указывает на роль вне `ROLE_DICTIONARY` (динамические шаблоны `var(--color-finger-${…})` / `var(--color-route-${…})` пропускаются);
   - не осталось ссылок на растворённые контрактные токены `--<компонент>-*` (легаси-префиксы `keycap`, `finger`, `footer-actions`, … перечислены в тесте). Два документированных исключения — локальные размерные переменные компонентов, делящие легаси-префикс, но не являющиеся токенами темы: `--keycap-unit` и `--avatar-size`.
7. **Bootstrap в `app.html`.** Инлайн-скрипт синхронно ставит `document.documentElement.dataset.theme`, читает mirror-ключ `flow-typing-theme` и падает в `prefers-color-scheme: dark`.

Тест НЕ проверяет:
- Значения ролей между темами (они и должны различаться).
- Геометрию в CSS компонентов (`1px solid` vs `2px dashed` — вне зоны теста; это дело компонента).

## 6.7. Добавление компонента с темизируемыми свойствами

1. **Перечислить визуальные элементы и состояния** компонента.
2. **Для каждого** — определить, есть ли уже подходящая роль в `ROLE_DICTIONARY`.
   - **Есть** — сослаться на неё из `<style>` компонента: `var(--color-<роль>)`. Геометрию (рамка, кольцо, тень) записать литералом прямо здесь.
   - **Нет** — завести новую роль:
     1. добавить имя в `ROLE_DICTIONARY` (`src/themes/roles.ts`) по соглашениям §6.3;
     2. определить её значение **в каждой теме** (`src/themes/light|dark|sepia|nord.css`) — маппингом/калькуляцией от ядра темы;
     3. добавить роль в `_template.css` (значение `unset`);
     4. сослаться на неё из компонента.
3. **`make check-all`** — контракт-тест должен пройти (все темы и шаблон объявили новую роль, компонент не ссылается на неизвестную роль).

Прежнего шага «создать `*.contract.ts` и вписать в `THEME_CONTRACT`» больше нет — файлов контракта не существует.

## 6.8. Архитектурные решения

| Решение | Альтернатива | Почему так |
|---|---|---|
| Компоненты привязаны к ролям L2 напрямую (`var(--color-*)` в `<style>`) | Слой компонентных контрактных токенов `--<компонент>-*` | Целевая модель ADR 0029: роли — постоянный интерфейс темы. Контрактный слой был транзитными лесами (промежуточный слой значений на компонент) и растворён — прямой `var(--color-*)` не протечка, а норма. |
| Геометрия (рамки, кольца, тени) — литералами в CSS компонента | Полное CSS-свойство как значение токена темы (`1px solid oklch(…)` целиком) | Роль несёт только цвет; форма — решение компонента, не палитры. Тема перекрашивается, не трогая структуру. |
| Единый словарь ролей `ROLE_DICTIONARY` для всех тем | Свой набор имён у каждой темы | Роль — общий интерфейс: компонент ссылается на имя, любая тема обязана его закрасить. Тест держит полноту словаря в каждой теме и шаблоне. |
| Роль именуется по семантике (`--color-text-primary`, `-foreground`) | По CSS-свойству элемента (`--color-*-color`) | Семантика переживает перекладку разметки; `--color-*-color` — тавтология. Суффикс-словарь (§6.3) фиксирует смысл вариаций. |
| Темы независимы: нет общего `:root` для цветов, ядро приватно | Общий `:root` с переопределениями в темах | Ядро каждой темы — свои абсолютные цвета (`sepia` — пергамент/чернила/янтарь, `nord` — своя палитра). Общий `:root` навязал бы единую палитру; роли уже дают общий интерфейс без этого. |
| Примитивы в `app.css :root`, без body-fallback | Держать `--body-*` литералами на окно до bootstrap | Head-скрипт ставит `data-theme` до применения CSS — окна без темы нет, страховать нечем. `body` ссылается на роли напрямую. |
