# Глава 6: Компонентные контракты и темы

Этот документ описывает архитектуру визуальной системы FlowTyping: как компоненты декларируют свои стилизуемые свойства, как темы их заполняют, и какой инвариант поддерживает контракт-тест.

## 6.1. Принцип

Тема в FlowTyping — это **набор значений CSS-свойств для визуальных элементов компонентов**, не палитра цветов. Контракт между компонентами и темами выстроен по правилу:

> **Компонент** диктует, какие визуальные свойства у его элементов есть и как они называются. **Тема** заполняет каждое свойство значением — литералом, ссылкой на свой внутренний токен или формулой `oklch(from …)`.

Из этого следуют три практических правила:

1. **Имя токена — это «где в UI», а не «какого цвета».** `--keycap-l2-background` — это «заливка клавиш левого указательного пальца». `--key-blue` — это палитровая переменная темы, в контракт она не входит.
2. **Токен несёт полное значение CSS-свойства, не фрагмент.** `--keycap-l2-border` — это `1px solid oklch(…)` целиком; тема может задать `2px dashed …`, если захочет. `--keycap-home-ring` — полное `box-shadow: 0 0 0 0.125rem oklch(…)`. Геометрия — тоже визуальное решение темы.
3. **Темы независимы между собой.** Каждая тема — самостоятельный snapshot значений; что внутри (литералы, формулы, ссылки на внутренние ключи) — её выбор. Контракт-тест проверяет только наличие токенов, не их содержимое.

## 6.2. Где живёт контракт компонента

У каждого компонента, у которого есть стилизуемые элементы, рядом лежит файл `*.contract.ts`:

```
src/components/key-cap/KeyCap.svelte
src/components/key-cap/KeyCap.contract.ts     ← 30 токенов
src/components/hands-scene/Finger.svelte
src/components/hands-scene/Finger.contract.ts ← 9 токенов
src/components/app/FooterActions.svelte
src/components/app/FooterActions.contract.ts  ← 16 токенов
…
src/Root.contract.ts                          ← body fallback (2 токена)
```

Контракт-файл — это массив имён токенов с литеральными типами:

```ts
export const FLOW_LINE_CONTRACT = [
  '--flow-line-border',
  '--flow-line-correct-background',
  '--flow-line-error-background',
] as const satisfies readonly `--${string}`[];

export type FlowLineContractToken = (typeof FLOW_LINE_CONTRACT)[number];
```

Все контракты агрегируются в `src/themes/contract.ts → THEME_CONTRACT` — это единый источник истины для контракт-теста.

## 6.3. Принципы именования токенов

| Шаблон | Когда используется | Пример |
|---|---|---|
| `--<component>-<element>-<property>` | свойство конкретного элемента компонента | `--keycap-l2-background`, `--cursor-symbol-color`, `--lesson-stats-display-item-background` |
| `--<component>-<element>-<state>-<property>` | свойство элемента в конкретном состоянии | `--keycap-l2-target-background`, `--footer-actions-btn-hover-background` |
| `--<component>-<semantic-role>` | декоративная роль, без явного CSS-свойства | `--keycap-home-ring`, `--keycap-path-ring`, `--nav-arrow-outline` |

- Для `background`, `border`, `color` имя совпадает с CSS-свойством — однозначно, читается естественно.
- Для `box-shadow` имя описывает визуальную роль (`ring`, `outline`), потому что одно CSS-свойство используется в разных смыслах — «обводка home-клавиши» vs «обводка path-клавиши».
- Префикс компонента (`--footer-actions-btn-*`, а не просто `--btn-*`) обязателен — это даёт scope-изоляцию между компонентами с похожей семантикой.

## 6.4. Что считается контрактом, а что — кухней темы

Тема — самостоятельный CSS-файл `src/themes/<id>.css` с селектором `:root[data-theme="<id>"]`. Внутри неё могут жить любые `--*` переменные — это не нарушение контракта. Тест проверяет только, что **все токены из `THEME_CONTRACT` декларированы**.

Сейчас все 4 темы внутри себя держат «палитру» с префиксом `--color-*` (legacy):

```css
:root[data-theme="light"] {
  /* Внутренняя палитра темы — не контракт */
  --color-bg     : oklch(1 0 0);
  --color-surface: oklch(0.95 0 0);
  --color-success: oklch(0.72 0.19 150);
  /* … */

  /* Контракт — ссылки на палитру */
  --body-background          : var(--color-bg);
  --keycap-l2-background     : var(--color-keycap-group-yellow-bg);
  --keycap-l2-border         : 1px solid var(--color-keycap-group-yellow-border);
  --footer-actions-btn-success-background: var(--color-success);
  /* … */
}
```

Эта структура — не обязательная. Тема может быть и литералами без промежуточных переменных, и через формулы `oklch(from var(--key) …)` из 4 ключевых цветов. См. `light.css` для «плоской» формы и потенциальный пример с формулами в комментарии к `_template.css`.

## 6.5. Глобальный `:root` и body fallback

`src/app.css :root` содержит **только то, что не тематизируется**:

- **Primitives** — typography (`--font-sans`, `--font-size-*`), radius (`--radius`, `--radius-N`), spacing (`--spacing`, `--spacing-N`), shadows (`--shadow-*`), motion (`--motion-duration-*`, `--motion-ease-*`). Эти токены общие для всех тем.
- **Body fallback** — `--body-background` и `--body-color` как литералы. Это нужно для тонкого окна между paint и inline-bootstrap-script в `src/app.html`, который синхронно ставит `data-theme`. Без fallback body выглядел бы голым в этом окне.

Все остальные контрактные токены (`--keycap-*`, `--finger-*`, и т.д.) живут **только в темах**. Когда `data-theme` выставлен — токены берутся из выбранной темы; вне темы они не определены, и компоненты, которые их используют, не отрисуются. Это допустимо, потому что inline-script гарантирует, что `data-theme` всегда выставлен до того, как сцена прорисовывается.

## 6.6. Контракт-тест

`src/themes/contract.test.ts` — единая точка проверки инварианта тем. Он:

1. **Парсит CSS-блок** каждой темы и `_template.css` через регулярку (без CSS-парсера; формат `<selector> { … }` плоский, парсинг тривиален).
2. **Проверяет**, что каждая тема декларирует **каждый** токен из `THEME_CONTRACT`. Значения свободные.
3. **Проверяет** `_template.css` тоже — там значения могут быть `unset`, важно только наличие имён.
4. **Sanity-checks для `THEME_CONTRACT`** — нет дубликатов, минимум 100 токенов, все начинаются с `--`.
5. **Проверяет registry-консистентность** и inline-bootstrap-script в `app.html`.

Тест НЕ проверяет:
- Значения токенов между темами (они и должны различаться).
- Корректность форматирования значений (`1px solid oklch(…)` vs `1px solid red` — обе валидны).
- Соответствие компонентов и контракта (если компонент использует `var(--несуществующий-токен)`, это поймает ESLint/svelte-check, не contract-тест).

## 6.7. Добавление нового компонента с темизируемыми свойствами

1. **Перечислить визуальные элементы и состояния** компонента.
2. **Для каждого** — придумать имя по соглашениям §6.3 и решить, какое полное CSS-значение принимает.
3. **Создать `<Component>.contract.ts`** рядом с компонентом, по образцу `KeyCap.contract.ts`.
4. **Добавить в `src/themes/contract.ts`** — импорт и spread в `THEME_CONTRACT`.
5. **Добавить токены в каждую тему** (`light/dark/sepia/nord/_template`) с осмысленными значениями. Можно через ссылку на палитру темы или литералом.
6. **Использовать в компоненте** через `var(--<имя>)`.
7. **`make check-all`** — contract-тест должен пройти.

## 6.8. Архитектурные решения

| Решение | Альтернатива | Почему так |
|---|---|---|
| Контракт-файл рядом с компонентом (`*.contract.ts`) | Один общий `src/themes/all-contracts.ts` | Co-location: правка компонента и контракта живёт в одном месте; снижает риск рассинхрона. |
| Длинные префиксы по имени компонента (`--footer-actions-btn-*`) | Короткие (`--btn-*`) | Scope-изоляция: если два компонента имеют «кнопки» с разным стилем, темы могут различать их без хаков. |
| Per-position для KeyCap (10 × `--keycap-l1..r5-*`) | Per-finger-num (5 × `--keycap-1..5-*`) | Возможность разделить L и R руки в теме (например, левая в одной тональности, правая — в другой). |
| Общий target/correct/error в KeyCap не per-position | Полностью per-position | Состояние перекрывает позицию: «правильно нажато» важнее, на каком пальце. Per-position только для `target` — поведение цели читается как «эта L2 — цель», и тема может выделять L2 уникальным glow. |
| `--keycap-home-ring` несёт полное `box-shadow` | Только цвет | Геометрия рамок — визуальное решение, должна управляться темой. Тема «sharp» может иметь узкие ring, «glow» — широкие. |
| Темы независимы (нет общего production-ready `:root` для цветов) | Один общий `:root` с переопределениями в темах | Темы могут радикально различаться по структуре (light — 4 ключа + формулы, nord — 50 литералов). Общий шаблон ограничивал бы свободу. |
