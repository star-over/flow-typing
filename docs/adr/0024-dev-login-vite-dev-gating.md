# Dev-вход — гейт `import.meta.env.DEV` (клиент) и одиночный fail-closed `DEPLOY_ENV` (сервер)

> Статус: принят · 2026-07-11 · Поправляет ADR 0012 (механизм dev-двери) и ADR 0023 (снят второй предохранитель Password) · Реализация: в коде (`SignInScreen.svelte`, `convex/auth.ts`)

Инструментальный dev-вход (ADR 0012) держался на **пяти** переключателях: клиентские `PUBLIC_DEV_LOGIN` / `PUBLIC_DEV_LOGIN_EMAIL` / `PUBLIC_DEV_LOGIN_PASSWORD` в gitignored `.env.local` и серверные `AUTH_DEV_LOGIN_ENABLED` + `DEPLOY_ENV`. Клиентские три — recurring-боль: `.env.local` не в репозитории, их надо заводить заново на каждой машине / клоне / сессии агента, иначе кнопки dev-входа нет. При этом собратья dev-входа — клиентские dev-helper'ы `window.__resetProfile` / `window.__setLadderStep` (ADR 0012) — **уже** за одним лишь `import.meta.env.DEV` (`appActor.ts`), без env-флагов. Dev-вход с env-флагами был выбивающимся частным случаем.

Решение — привести dev-вход к тому же паттерну и свести пять ручек к одной:
- **Клиент:** кнопка dev-входа и зашитый throwaway-аккаунт за `import.meta.env.DEV` — встроенным признаком Vite (`true` под `vite dev`, `false` в любой `vite build`). Ветка статически заменяется на сборке и вырезается dead-code elimination → prod-сборка её не содержит. `PUBLIC_DEV_LOGIN*` сняты.
- **Сервер:** Password-провайдер регистрируется при `!isProduction()` — единственный fail-closed гейт `DEPLOY_ENV` (ADR 0023). `AUTH_DEV_LOGIN_ENABLED` снят: Password встаёт в один ряд с dev-мутациями (`assertNonProd`), которые и так под этим единственным флагом.
- **Итог:** 5 → 1. Остаётся `DEPLOY_ENV=development`, задаётся один раз на dev-deployment (не recurring).

## Considered options

- **Сохранить двойной предохранитель Password (статус-кво ADR 0023)** — отвергнут. Второй флаг защищал **только** Password от одной ошибки — активного `DEPLOY_ENV=development` на prod; но та же ошибка уже открывает dev-мутации (`setMyLadderStep` = дыра целостности), которые сидят под одним `DEPLOY_ENV` без второго флага. Belt-and-suspenders на одном гейте при незащищённом соседе — кривой приоритет, а не defense-in-depth. В recurring-боль второй флаг не входит (set-once), т.е. проблему не решал, но оставался лишней ручкой.
- **Оставить `PUBLIC_DEV_LOGIN*`, но показывать кнопку только за `import.meta.env.DEV`** — отвергнут: вход всё равно требует аккаунта из `.env.local` → recurring-конфигурация выживает, смысла нет.
- **Гейт по `import.meta.env.MODE !== 'production'` + сборка `vite build --mode staging`** — отложен. Нужен только если E2E/агент ходит на собранный артефакт (`make preview`), где `DEV === false`. Текущий агент/ручной прогон идёт через `make dev` → `import.meta.env.DEV` достаточно. Ввести `MODE` точечно, если понадобится prod-подобная сборка с dev-входом.
- **Зашить аккаунт vs держать в env** — выбран hardcode: zero-config и есть цель; остаточный риск — ниже.
- **`import.meta.env.DEV` + одиночный `DEPLOY_ENV` (выбран)** — единообразно с уже существующими dev-helper'ами; клиент без env вовсе (строго безопаснее — нет флага, который можно ошибочно запечь в prod); сервер одним fail-closed флагом, как dev-мутации.

## Consequences

- **Клиентская дверь единообразна** с `window.__*` dev-helper'ами (оба на `import.meta.env.DEV`). Гарантия «нет кнопки на prod» теперь на компиляторе (статическая замена + DCE), а не на env-дисциплине — крепче: нет флага, который можно забыть/ошибочно выставить.
- **Сервер: `DEPLOY_ENV` — единый гейт** и для Password, и для dev-мутаций. Fail-closed без изменений: забыл на prod → закрыто; забыл на dev → dev-инструмент шумно ломается.
- **Остаточный риск (репозиторий публичный).** Аккаунт зашит в открытый репозиторий → войти им можно в **dev-deployment** как тестовый аккаунт `dev@flowtyping.local`. Границы: (a) работает лишь где поднят Password — т.е. `DEPLOY_ENV=development` (dev-песочница); (b) тестовый аккаунт, без эскалации, видит только свои данные; (c) писательские мутации под rate-limiting (P0-10); (d) на prod нужен активный misconfig `DEPLOY_ENV=development`, который и без этого флага уже открывает dev-мутации → снятый второй предохранитель **не создаёт нового класса prod-отказа**, лишь убирает belt-and-suspenders на prod, уже пробитом тем же misconfig. Принято ради zero-config для несенситивной песочницы.
- **`AUTH_DEV_LOGIN_ENABLED` больше не читается.** На dev-deployment безвреден, если оставить; опциональная уборка — `npx convex env unset AUTH_DEV_LOGIN_ENABLED`. Prod-bootstrap больше не должен «не ставить» его — пункт снят.
- **Тесты:** convex-проект vitest по-прежнему ставит `DEPLOY_ENV=development` (`vitest.config.ts`); `buildProviders` проверяется как чистая функция по `production`-параметру, env-флаг ему не нужен.
- **ADR 0023** сохраняет тело; его следствие «Password за двумя предохранителями» помечено здесь как поправленное (датированная сноска в 0023).
