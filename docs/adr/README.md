# ADR — журнал архитектурных решений

Записи «что решили и почему» — по одной на существенную развилку. ADR отвечает на
«почему так»; текущее состояние системы описывают живые документы:
[`CONTEXT.md`](../../CONTEXT.md) (доменный глоссарий + сведённая текущая истина) и
`CLAUDE.md` (операционная карта). Живые документы ссылаются на номера ADR и не
пересказывают их; при расхождении любых доков с кодом доверять коду.

## Что достойно ADR

Решение фиксируется, если верно хотя бы одно: дорого обратить; привязка к
поставщику/платформе/протоколу; режет несколько подсистем; выбран один из нескольких
жизнеспособных вариантов с явным trade-off. Тривиальные и реализационные детали —
не сюда (их дом — код, планы, живые документы).

## Жизненный цикл

- **Статус-строка** — blockquote сразу под заголовком: статус · дата ·
  поправки/связи · состояние реализации (снимок в теле ADR; сводный roadmap
  незавершённого — в `docs/backlog.md`, не в индексе). Статусы: **предложен → принят**; далее
  «поправлен ADR N» (курс скорректирован) или «заменён ADR N» (решение отменено).
- **Тело принятого ADR неизменно.** Смена курса = новый ADR со ссылками в обе
  стороны: новый пишет «Поправляет: ADR N», в старом обновляется только
  статус-строка. Без нового ADR допустимы: опечатки, битые ссылки, дописанные с
  датой наблюдения в Consequences.
- **Пишем в момент решения**, не задним числом; рассмотренные и отвергнутые
  варианты (Considered options) — обязательная часть: без них запись со временем
  теряет ценность.
- **Числа-настройки в ADR не фиксируются** — ADR держит структуру, не значения;
  дом чисел — код (константы) и таблица в
  [`docs/plans/auto-flow.md`](../plans/auto-flow.md).

Шаблон новой записи — [`_template.md`](_template.md).
