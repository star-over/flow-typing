# Тренировка требует входа; полу-режим гостя похоронен — возврат гостя только анонимным профилем

> Статус: принят · 2026-07-03 · Поправляет: ADR 0005 (гостевая часть) · Реализация: в коде (план `docs/plans/2026-07-03-adr-0012-auth-required.md`, ветка `feat/adr-0012-auth-required`)

Канон держал три несовместимые позиции о неавторизованном доступе одновременно
(аудит `docs/plans/2026-07-03-adr-canon-audit.md` §4.1): auth.md — «гостевой режим
поддерживается (soft-progression)»; ADR 0005 — гость = анонимная identity
(запланированный этап); рабочее намерение — снять доступ совсем (ADR 0011,
Consequences). Фактическое состояние было четвёртым: гость печатает, но профиль не
растёт (`drillRecord` бросает 'Not authenticated' и молча гасится) — он навсегда
застревает на ступени 1 со случайной выдачей. Такой полу-режим нарушает
«soft-progression» по построению и является худшей витриной для продукта, чья суть —
адаптация.

Решение: **неавторизованный доступ к тренировке снимается**. Все `drill*`-функции —
auth-required; гостевых веток в контрактах нет. Полу-режим «печатает, но не растёт»
похоронен как класс решений — навсегда. Если гость возвращается (драйвер — публичная
альфа / выход на PMF-фазу), он возвращается **только** как анонимный профиль с полной
петлёй (этап «Анонимный профиль», ADR 0005): ленивая identity при первом старте,
регистрация = явный перенос профиля.

## Considered options

- **Достроить анонимный профиль сейчас** — идеальное конечное состояние сразу, но
  заметный кусок работы (Anonymous-провайдер, перенос профиля, уборка брошенных
  анонимных аккаунтов) не на критическом пути проверки гипотезы (важнее этапы
  «Фокус»/«Термостат»); публичных пользователей нет — выгода сегодня нулевая.
  Отложен, не отвергнут: остаётся единственным санкционированным путём возврата гостя.
- **Легализовать полу-режим** (печатает без роста + честный UI «войди, чтобы расти») —
  ноль серверной работы, но это и есть источник противоречия: вечный налог гостевой
  ветки на каждый будущий серверный контракт (двойная семантика `drillNext`,
  silent-catch, cold-start-ветки) ради витрины, показывающей продукт без его сути.
  Отвергнут навсегда.
- **Снять доступ без пути возврата** — та же простота, но решение о госте для
  публичного запуска пришлось бы принимать заново с нуля; путь возврата уже
  спроектирован (ADR 0005), выбрасывать его нет причин. Отвергнут в пользу связки
  «снять сейчас + анонимный профиль потом».

## Consequences

- `/train` для гостя показывает приглашение войти со ссылкой на `/signin` (по
  образцу `/stats` — один приём на всё приложение, без redirect-магии);
  лендинг `/` и `/signin` остаются публичными.
- **Инструментальный dev-вход** — обязательная пара к снятию доступа: без него
  ИИ-агент (E2E, `/verify`, дизайн-прогоны в чистом браузерном контексте) не может
  войти — OAuth для агента непроходим (внешние формы, 2FA). Механизм: стоковый
  Password-провайдер `@convex-dev/auth`, регистрируемый в `convex/auth.ts` только
  при выставленном env-флаге Convex (на production-deployment флага нет — провайдера
  физически не существует); форма на `/signin` — за клиентским флагом из
  `.env.local` через `$env/dynamic/public` (для `adapter-static` он подставляется на
  этапе `vite build` из наличного `.env.local`: CI/production собирает без флагов →
  кнопки и данных входа в артефакте нет; локальный `make build` с dev-`.env.local` их
  запекает — но `build/` gitignored и по сети не отдаётся); один стабильный тестовый
  пользователь;
  существующая мутация `resetMyProfile` легализуется как инструмент «чистого листа»
  прогонов. Это инструмент разработки, не продуктовый режим — клиентских обходов
  auth-семантики нет. Отвергнутый вариант Anonymous-за-флагом плодил бы юзера на
  каждый чистый контекст и воскрешал отложенный анонимный профиль чёрным ходом.
- **Порядок сноса:** сначала dev-вход, затем закрытие `/train` и auth-required
  контракты — прогоны агентов не ломаются ни на минуту.
- `drillNext` становится auth-required и бросает 'Not authenticated' симметрично
  `drillRecord`/`upsertMine`; ветка «нет пользователя → cold-start ступень»
  (`DEFAULT_OPENED_STEPS` для `userId === null`) умирает; снята двусмысленность
  «`drillNext` без auth, `drillRecord` с auth» (ADR 0007, Consequences).
- Silent-catch «гость/офлайн» в `session-impl.ts` сужается до одного офлайн-случая —
  гостевая половина причины исчезает (связанный открытый вопрос об outbox — аудит §4.2,
  отдельное решение).
- Строка auth.md «Гостевой режим — Поддерживается» — superseded этим ADR; CONTEXT.md
  (Relationships) и docs/05 §5.7 обновляются при принятии.
- Этап «Анонимный профиль» в `docs/plans/auto-flow.md` остаётся запланированным; при
  его реализации этот ADR будет поправлен (доступ вернётся — но только в полной петле).
- Тесты гостевых путей `drillNext` (cold-start без пользователя) переписываются на
  auth-required семантику.
- Дыра «qwerty без серверных данных» (аудит §4.3) этим решением не закрывается —
  вырожденная сессия остаётся доступной авторизованному пользователю; независимое
  решение.
