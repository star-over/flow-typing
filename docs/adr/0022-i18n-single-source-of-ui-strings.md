# i18n — единое хранилище всех UI-строк; в разметке литералов нет

> Статус: принят 2026-07-09 · Реализация: в коде (auth-хром через i18n, P0-6); правило — CLAUDE.md «Conventions»

Разворот на EN-first (ADR 0021) вскрыл класс дефекта: дефолт интерфейса уже `en`,
но экран входа (`SignInScreen`) и меню юзера (`UserMenu`) выводили русские строки
вшитыми в разметку — англоязычный пользователь видел русский вход независимо от
`interfaceLanguage`. Корень — не в этих трёх строках, а в отсутствии правила: ничто
не запрещало компоненту вставить строковый литерал прямо в разметку. Починка
отдельных строк лечит симптом; правило закрывает класс.

**Решение.** i18n (`dictionaries/{en,ru}.json` через derived-store `dictionary`) —
**единственное хранилище всех строк, видимых пользователю**. Любая UI-строка попадает
в DOM из i18n: компонент либо читает `$dictionary.<секция>.<ключ>`, либо получает
**уже локализованную** строку в качестве параметра (локализует родитель — как надпись кнопки паузы
у `Header`). Правило держится **даже для строк, физически одинаковых во всех языках**
— бренд (`FlowTyping`), имена провайдеров (GitHub/Google/Yandex), технические ярлыки:
их значение просто совпадает в `en` и `ru`, но живут они в словаре. Строкового
литерала, предназначенного для показа, в разметке компонента нет — включая
accessibility-текст (`sr-only`, `aria-label` с понятным текстом).

Вне правила — то, что не является UI-копией: динамические значения из внешних
источников (текст ошибки провайдера, пользовательский ввод), не отображаемые
атрибуты-идентификаторы (`class`, технические `aria-*`), dev-only логи в консоль.

## Considered options

- **A — i18n только для переводимого, непереводимое (бренд, имена провайдеров)
  литералом.** Отвергнут: суждение «это локализуется?» принимается по каждой строке
  в момент написания и дрейфует; проверяющий не отличает намеренный литерал от забытого;
  ровно этот баг (русский литерал в EN-сборке) возвращается. И у бренда/термина нет
  шва, если позже понадобится по-локальная подгонка (транслитерация, регистр).
- **B (выбран) — всё через i18n, включая строки с одинаковым значением во всех языках.**
  Цена: в словарях появляются записи, чьи `en` и `ru` совпадают, и микро-трение при
  добавлении строки. Выигрыш: единый перечень всей UI-копии в одном месте; «голый
  строковый литерал в разметке = дефект» становится инвариантом без исключений
  (grep/линт, а не список «намеренных литералов» для сверки); тривиально добавить
  локаль или переопределить термин.

## Consequences

- Аудит вшитой UI-копии — механический: любой литерал в разметке — нарушение, без
  allowlist'а «намеренных».
- Словари несут записи с совпадающими `en`/`ru` (напр. `sign_in.github` = «Sign in
  with GitHub» / «Войти через GitHub» одинаков в части бренда; чисто-брендовые
  фрагменты совпадают целиком) — принятая дупликация, цена правила без исключений.
- Чек-лист нового компонента получает шаг: строки → секция i18n (гранулярность
  по компоненту/области, как у существующих секций).
- Тестом сейчас **не** enforced: проверка — grep кириллицы/латиницы в разметке +
  ревью. Линт-правило (запрет строковых литералов-детей в `.svelte`-разметке) —
  возможное усиление, **не строим** до потребности (нет поспешно добавленной инфраструктуры).
- Применено к auth-хрому (`SignInScreen`, `UserMenu` — P0-6); остальное приложение
  уже соответствовало (аудит нашёл литералы только в этих двух).
