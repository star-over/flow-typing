# ui/Button — примитив кнопки и модель вариантов

> Статус: принят · 2026-07-21 · поправлен 0035 (реализация вариантов переехала на shadcn-svelte + Tailwind) · Реализация: в работе (грань #3 CSS-аудита)

## Context

Аудит CSS (грань #3) нашёл дублирование кнопочных стилей: один визуальный
словарь (`surface + border`, плотная заливка, ghost-контур, brand-accent)
скопирован в независимые блоки `FooterActions` / `SettingsPage` / `stats` /
`SurveyPrompt` / `MessageScreen` / `LandingScreen` / `SignInScreen`. `MessageScreen`
(ADR-less примитив) уже вынес экран-сообщение, но ghost-кнопку держит вшитой.

Введение примитива кнопки с **системой вариантов** — это не применение
существующей конвенции (как было с `MessageScreen`), а новое структурное решение
о дизайн-системе: таксономия вариантов, способ полиморфизма элемента, где
проходит граница «стиль компонента ↔ макет владельца». Спецификация `MessageScreen`
явно отложила это решение сюда.

## Decision

1. **`src/components/ui/Button.svelte`** — доменно-нейтральный интерактивный
   примитив, покрывает только интерактивные элементы управления.
2. **Полиморфизм элемента:** параметр `href` задан → `<a href>` (навигация); иначе →
   `<button type>` (действие). Различие «навигация ↔ действие» — реальное
   (доступность, новая вкладка, preload, семантика), моделируется явно.
3. **Шесть вариантов, семантические имена в одном параметре** `variant`:
   `neutral` / `primary` / `success` / `danger` / `ghost` / `accent`. Матрица
   `style × tone` отвергнута (даёт несуществующие комбинации).
4. **Foreground плотных вариантов** (`primary`/`success`/`danger`/`accent`)
   стандартизуется на `--color-on-dense` (роль «текст на плотной заливке»),
   устраняя дрейф `--color-background` в `FooterActions`.
5. **Ось размера** `size` (`md` / `lg`), отдельная от цвета.
6. **Граница стиль ↔ макет:** компонент владеет внутренним видом (padding,
   radius, цвета, состояния, `inline-flex`+gap); позиционирование (margin,
   align, width, flex) — у родителя. В Svelte scoped-стиль родителя до корня
   дочернего компонента не дотягивается, поэтому граница ещё и техническая.
7. **Контент — фрагмент `children`** (композиция `KeyHint`), не закрытый контракт.

## Considered options

- **Готовая библиотека (shadcn/аналоги)** — на момент 0033 отложена: тогда её брали
  как React-only и вне стека проекта, а доменные вопросы (CTA-ссылка, KeyHint, i18n,
  темы) она не закрывает в любом случае. Позже пересмотрено: `shadcn-svelte`
  (Bits UI + Tailwind v4) работает на Svelte 5 без React — хром переведён на неё
  (ADR 0035), доменная модель вариантов этого ADR остаётся в силе.
- **Button-only без полиморфизма** — отвергнут: свёл бы CTA к кнопке, потеряв
  свойства ссылки на главной точке входа лендинга.
- **Матрица `style × tone`** — отвергнута в пользу семантических имён (YAGNI).
- **Втягивание сегмента/каретки** — отвергнуто: сегмент = будущий
  `SegmentedControl`, каретка = глиф; делят с кнопкой только токен цвета.

## Consequences

- Появляется общий словарь кнопок; семь мест перестают дублировать стили.
- `MessageScreen` теряет вшитую ghost-кнопку — переходит на `Button variant="ghost"`.
- Foreground плотных кнопок футера получает крошечный сдвиг тона (осознанно).
- `ui/Surface` (плашки) и `ui/SegmentedControl` (сегмент) — отдельные будущие
  примитивы; Button делит с ними роль-токены, но не код.
- Accent-вариант красит текст ролью `--color-cursor-foreground` (роль под brand-accent — «тёмное на янтаре», как каретка логотипа и CTA), а не `--color-on-dense`; Decision #4 группирует его с плотными вариантами по механике заливки, а не с текстовым токеном.
