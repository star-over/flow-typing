# KeyLadder переезжает в символьную раскладку: шаг открытия — на символе (`ladderStep`), не на клавише

> Статус: принят · 2026-07-08 · Поправляет ADR 0002 (представление лестницы) · Реализация: в работе (рефакторинг 1:1 — перенос без смены нарезки)

KeyLadder (ADR 0002) нарезала на шаги **клавиши** (`keyCapId → step`), а шаг символа
выводился как `max` по шагам его клавиш. Нарезка тем самым уже per-symbol — но
**неявно и невыразимо**: все заглавные делят одну клавишу `Shift` (единственный
шаг 6), поэтому `max(шаг_буквы, 6)` сводит почти все заглавные в шаг 6 — они
открываются одним залпом вместе с пунктуацией на Shift. Для тренажёра «расти по
одному движению» это провал ровно на границе Shift, и `key→step` его починить **не
может в принципе** (заглавные делят одну клавишу с одним шагом). Решение: перенести
шаг открытия с клавиши на **символ** — хранить `ladderStep` прямо в записи
символьной раскладки (`{ symbol, keyCaps, ladderStep }`); отдельного артефакта
`KeyLadder` больше нет.

`symbol→step` строго выразительнее `key→step`: воспроизводит любое поведение
`key→step` (дать каждому символу его прежний `max`-шаг) **и** выражает то, что
`key→step` невыразимо (методичный ввод заглавных по одной). «Денормализация» — не
возражение: шаг и сейчас per-symbol через `max`, просто скрытый; хранить явно —
перевод скрытого в управляемое, это достоинство, не цена.

## Considered options

- **Одна общая геометрическая лестница на все раскладки** — отвергнута: нарезка
  привязана к раскладке (ADR 0002, per Layout Context), общий шаг не выразить.
- **Оставить `key→step` (статус-кво)** — отвергнут: не выражает методичные
  заглавные (Shift-взрыв сводит их в один шаг); при этом шаг символа всё равно
  вычисляется как `max` по клавишам — модель уже per-symbol, но в неудобном,
  невыразимом представлении.
- **`symbol→step` с файловым `version` лестницы** — отклонён как преждевременный:
  `version` не читается ни строкой кода, а машины миграции (пересчёт `openedSteps`
  юзеров по версии) не существует — см. Consequences.
- **`symbol→step` без `version` (выбран)** — `ladderStep` в записи раскладки;
  отдельный `KeyLadder`, его проверка (`validateKeyLadder`) и двойные карты (`keyStepMap` +
  `symbolToKeyCaps` → `keyToStep`) уходят.

## Consequences

- `shared/key-ladder/` растворяется: `jcuken.ts`, `types.ts`
  (`KeyLadder`/`KeyLadderEntry`), `validate.ts` удаляются. `computeStepLevel`
  упрощается до `max(ladderStep)` по символам drill'а (уходит `keyToStep`);
  `symbolsAtStep`/`maxLadderStep` становятся тривиальными операциями над
  `ladderStep` и переезжают к `shared/symbol-layout.ts`.
- `validateKeyLadder` (сверка «лестница ↔ набор клавиш раскладки») больше не нужна —
  источник один. Заменяется тонкой проверкой над самим файлом: шаги `0..max` без
  дыр, `Space` на шаге 0, у каждого символа целый `ladderStep ≥ 0`.
- Новая обязанность: где открывается каждая заглавная буква и символ на Shift — теперь **много**
  решений вместо одного «Shift на шаге 6». Выразительность = обязанность задать
  порядок осмысленно (авторская работа, стыкуется с частотной настройкой под язык).
  **1:1-перенос сохраняет текущее поведение** (заглавные по-прежнему сведены на
  шаге 6); постепенные заглавные — отдельный шаг позже, на нём же вернётся version.
- Граница слоёв (ADR 0016) **не нарушена**: `ladderStep` — «учебная программа», шаг
  не путает физ/символ/палец. Но файл раскладки теперь «символ → клавиши **+ шаг
  программы**» — осознанное расширение символьного слоя новой сущностью, принято
  здесь.
- `version` лестницы **снят пока**. Его не читает ни строки кода, а машины миграции
  (пересчёт `openedSteps` юзеров по версии) не существует — это была
  задокументированная политика, не механизм; tripwire без читателя защищает слабо.
  Рабочее правило вместо него: правка `ladderStep` → `make rebuild-selection-index`
  (идемпотентный пересчёт `stepLevel`, версией и раньше не управлялся); смена
  **нумерации** шагов (ре-нарезка, не 1:1) → пересчёт `openedSteps` юзеров.
  `version` вернётся вместе с кодом, который его читает (первая реальная ре-нарезка),
  тогда его форма определится нуждами того кода, а не гаданием сейчас.
- Клиент не затронут: `ladderStep` — серверная/инструментальная забота; клиентская
  zod-схема (`src/lib/layouts.ts`) отбрасывает лишнее поле, клиентский тип раскладки
  его не несёт (ADR 0016).
- JSON раскладки остаётся **массивом** (`ladderStep` добавлен в запись) — импорт-
  сайты не трогаются, физ/палец/символ единообразны. `SymbolEntry.ladderStep`
  опционален: qwerty пока без нарезки (регистрация qwerty ждёт своей лестницы,
  P0-9), а зарегистрированные раскладки обязаны иметь `ladderStep` у каждого символа
  (проверяет `validateSymbolLadder`).
- Связь с ADR 0002: принцип «ручные шаги по геометрии, не частотность» **в силе** —
  меняется только **представление** лестницы (единица клавиша → символ). Тело 0002
  неизменно; его статус-строка отмечает поправку представления.
