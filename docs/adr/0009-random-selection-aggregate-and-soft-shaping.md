# Случайный отбор drill'ов — агрегат для кандидатов, форма распределения в soft-слое; `drillNext` — query, не mutation

Развивает разделение ADR 0003 (жёсткие требования — индексной фильтрацией, мягкие — ранжированием) до конкретного механизма выдачи на большом корпусе. Отбор порции расщепляется на два физически разных слоя — **где брать кандидатов** (storage, индекс) и **как их упорядочить** (app, чистая функция) — и случайность делится между ними по роли. Это снимает техническую причину, по которой `drillNext` был mutation (ADR 0006/0008), и держит выдачу открытой к будущим срезам фильтрации без переписывания горячего пути.

1. **Двухслойный отбор: candidate generation ↔ soft shaping.** Storage-слой отвечает только на «какие строки прочитать» — жёсткая пред-фильтрация плюс случайный доступ. App-слой (чистая `rankBySoftSignals`) отвечает на «как упорядочить прочитанное». Эти роли не смешиваются: индекс не знает о профиле и предпочтениях, ранжирование не ходит в базу.

2. **Hard = `(namespace, bounds)` агрегата = индексная пред-фильтрация ADR 0003.** Агрегат `@convex-dev/aggregate` над `drillSelectionIndex` с `Namespace = symbolLayoutId`, `Key = stepLevel`. Жёсткий фильтр «раскладка + потолок репертуара» выражается как `namespace = layout` (зеркало `eq('symbolLayoutId', …)`) + `bounds = { upper: { key: openedSteps, inclusive: false } }` (зеркало `lt('stepLevel', openedSteps)`). Поверх среза агрегат даёт `count()` и `at(offset)` за O(log n) — единственное, чего нативный индекс не умеет дёшево. Фильтр считается hard ⟺ он выразим как `равенство(X) + один диапазон(Y)`; иначе он soft.

3. **Случайность разделена по роли.** *Breadth* (равномерность + масштаб + обход кэша) рождается на storage-слое: seeded избыточная выборка (over-sample) `M` кандидатов через `atBatch` по случайным позициям `[0, count)`. *Shaping* (вес, recency, смещение к слабому, тай-брейк, в том числе взвешенный random) задаёт **только** soft-функция над этой выборкой. Равномерный отбор — частный случай нулевых весов. Унести storage-random в soft нельзя: тасовать весь пул — O(pool) (ломает масштаб), тасовать детерминированную страницу — нулевая вероятность у части корпуса (мёртвый контент).

4. **`drillNext` — query, не mutation; свежесть из `seed`.** Снимается обходной манёвр ADR 0006/0008 (mutation ради несвежей случайной выборки). Контракт: `drillNext(symbolLayoutId, budgetChars, seed)`. Кэш Convex замораживает результат query целиком — поэтому свежесть даёт `seed`-аргумент, генерируемый клиентом на каждый поход (разный ключ кэша → промах → честный рандом), а **не** то, в каком слое сидит `Math.random`. `random()` агрегата без seed и «random в soft без seed» застывают одинаково.

5. **Граница hard↔soft и повышение.** `M` — бюджет soft-слоя (ручка «чтения ↔ сила адаптивности»): сильное взвешивание требует `M ≫ k`, при пределе скатывается к O(pool). Будущий фильтр остаётся soft (rejection/вес над over-sample), пока не измерено, что он **и горячий, и селективный**; тогда он повышается в `namespace`/`key` через существующий `selectionIndex.rebuild` (пересборка + backfill агрегата). Per-user сигналы (слабость по профилю) в глобальный индекс не запекаются в принципе — остаются в soft.

## Considered options

- **`mutation → query` напрямую** (или `random()` без seed) — отвергнут: query кэшируется как единое целое, выборка застывает на весь сеанс между записями в namespace. Не решает задачу без seed.
- **Чистый клиентский рандом** (слать eligible-пул на клиент, тасовать локально) — отвергнут: O(pool) на проводе, ломает масштаб (req «большой корпус»).
- **`randomKey`-индекс без компонента** (поле случайного ключа + range-скан) — отвергнут: `stepLevel < openedSteps` и range по `randomKey` — две независимые оси, в один индекс не ложатся; каждая новая жёсткая ось ломает схему. Примитивы `count/at/bounds/namespace` агрегата чище под будущие срезы.
- **Random полностью в soft** (убрать storage-draw) — отвергнут: вынуждает либо O(pool), либо смещённое покрытие, и всё равно требует seed против кэша. Жизнеспособен только как *добавочный* слой взвешивания поверх over-sample (зафиксировано в решении 3), не как замена.
- **Запекать per-user слабость в индекс** — невозможен: индекс глобален и user-agnostic (как `drillSelectionIndex` по ADR 0005); персонализация живёт в soft над over-sample.

## Consequences

- Новый компонент + инвариант синхронизации: агрегат держится в синхроне с
  `drillSelectionIndex` на уровне оркестрации `rebuild` (единственный писатель таблицы):
  `insertBatch` отражает вставку (`insertIfDoesNotExist`), а namespace сбрасывается одним
  `clear()` в начале `rebuild` (`resetLayoutAggregate`). `clearLayoutPage` — только таблица.
  Операции идемпотентны (rebuild прогоняется повторно).
- Контракт меняется: `drillNext(symbolLayoutId, budgetChars, seed)` — добавляется `seed`; функция перестаёт быть mutation. `session-impl.ts` зовёт `convex.query` со свежим seed на каждый поход вместо `convex.mutation`. Доменно `drillNext` остаётся читателем (ADR 0008) — теперь и технически.
- `rankBySoftSignals` — единая точка будущей политики выдачи: фокус-паттерны (ADR 0004), анти-повтор/recency, длина слова (мягкие требования ADR 0003), взвешенный random. Чистая, тестируется без заглушек над фиктивным over-sample.
- На текущих ~501 drill выигрыша по производительности нет — `.collect()` копеечный. Внедряем сейчас, пока корпус мал и backfill агрегата бесплатен, чтобы не переписывать горячий путь на большом корпусе позже.
- `M` (over-sample factor) — провизорная ручка, как числовые пороги ADR 0008: ADR фиксирует структуру (storage даёт равномерный seeded over-sample, форму задаёт soft), не значение.
- Реализовано планом `docs/plans/2026-06-25-drill-random-selection-aggregate.md` (v1 поведение-сохраняющий: равномерный seeded-отбор, soft-слой/over-sample отложены).
