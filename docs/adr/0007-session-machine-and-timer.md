# Таймерная сессия — отдельной машиной над движком печати; таймер охватывает running+refilling

ADR 0005/0006 положили подбор на сервер и контракт в символах, но не сказали, **как клиент устроен**. Рабочая петля («визуализация движения» + подача порций под уровень) требует слоя, который держит таймер сессии, очередь drill'ов, дозагрузку, чекпоинт-сводки и склейку drill'ов в непрерывный поток. Это решение фиксирует архитектуру этого слоя. Реализация — `docs/plans/2026-06-17-session-machine.md`.

## Решение 1 — три машины, раскол по двум осям

`appMachine` → invoke `sessionMachine` (в состоянии `training`) → invoke `trainingMachine`.

- **`appMachine` ↔ `sessionMachine` — по времени жизни.** `appMachine` — долгоживущая оболочка: навигация (`menu`/`training`/`sessionComplete`), кросс-режимная клавиатура (`keyboardMachine`), singleton-актор, переживает SvelteKit-навигацию и HMR. `sessionMachine` — одноразовый прогон: таймер, очередь, `previousCheckpoint`, недопечатанный поток существуют **только во время тренировки**. Рождается на входе в `training` (`invoke`), уничтожается на выходе — сбросить нечего, чего по построению не существует вне прогона. Слить session в appMachine = держать изменяемое состояние прогона (таймер!) в HMR-singleton'е и руками обнулять на каждом `menu → training`; один забытый сброс = состояние прошлого прогона течёт в следующий.
- **`sessionMachine` ↔ `trainingMachine` — по ответственности (один писатель).** `trainingMachine` — чистый классификатор: сравнивает нажатие с целью, пишет `attempts`, двигает курсор; **единственный писатель** канонического `TypingStream`. `sessionMachine` — доставка и жизненный цикл (async-сеть, таймер). Смешать сеть+таймер в классификатор = взять его чистоту/детерминированность (нужную для повтора и инструмента захвата) в заложники грязной части; домены отказа (сеть падает, классификация — нет) сцепляются в произведение состояний.

`completed[]` в `sessionMachine` — **event-sourced проекция** напечатанного (из `TYPING.ADVANCED`), не второй писатель: канонический поток пишет только `trainingMachine` (свои `attempts` + полученный `APPEND_SYMBOLS`).

## Решение 2 — таймер охватывает running+refilling через родительское состояние `timing`

Таймер сессии (переживающий паузу, по настенным часам) обязан: тикать и в `running`, И в `refilling` (юзер печатает во время дозагрузки), **не сбрасываться** на bounce `running↔refilling`, и иметь возможность **истечь** в любой активный момент, включая refilling.

Достигается обёрткой `running`+`refilling` в общее подсостояние **`timing`**, которое держит тикер (`invoke`), `markSegmentStart` (`entry`), `accumulateElapsed` (`exit`) и общие обработчики `TICK`/`TIMER_EXPIRED`/`PAUSE_TIMER`/`KEY_PRESS`. Внутренний переход `running↔refilling` **не выходит** из `timing` → тикер не перезапускается, `segmentStartedAt` не сбрасывается, время идёт непрерывно, истечение ловится. `paused`/`draining` — сёстры `timing`; выход из `timing` (на паузу или в draining) фиксирует сегмент один раз через `exit`.

Опирается на документированную семантику XState v5: `invoke` живёт на своём узле и переживает переходы между его детьми; `entry`/`exit` компаунд-состояния срабатывают только на его вход/выход, не на внутренних переходах; `after` не умеет паузу (поэтому таймер — на аккумуляторе `elapsedMs`+`segmentStartedAt`+`fromCallback`-тикере, не на `after`). Подтверждено исследованием (docs XState v5 + community-практики «таймер invoke'ить на родителе» + независимый разбор кода).

## Considered options

- **Плоский `refilling` рядом с `running`** (первая версия) — `markSegmentStart`/тикер на `running`. Bounce `refilling → running` заново входит в `running` → `markSegmentStart` сбрасывает `segmentStartedAt`, **не зафиксировав** `elapsedMs` → отсчёт откатывается к ~0 на каждом refill → **сессия не истекает никогда**. Плюс `TIMER_EXPIRED` теряется, пока машина в `refilling`. Это и был баг, породивший ADR.
- **Плоская структура + ручной commit на каждом переходе** `running↔refilling` — чинит баг, но дублирует таймер-концерн (тикер, `accumulateElapsed`, обработчики) по двум состояниям; именно это дублирование и забыли при вводе `refilling`. Хрупко.
- **Refill без отдельного состояния** (`spawnChild`-Promise, не покидая `running`) — таймер не видит смены подсостояния, но теряется наблюдаемый `refilling`, `onDone`/`onError` уезжают в ручную обработку, и нужен ручной guard от гонок (которые иерархия даёт бесплатно).
- **`timing`-обёртка** (выбрано) — таймер-концерн живёт в одном месте; bounce не трогает его структурно; `onError` декларативен; цена — пути состояний `{active:'running'}` → `{active:{timing:'running'}}`.

## Consequences

- Иерархия состояний: `app.training` → `session.active` (invoke `training`) → `{ timing: { running, refilling }, paused, draining }` → `done`.
- Таймер: `elapsedMs` (аккумулятор завершённых сегментов) + `segmentStartedAt` (на входе в `timing`) + `displayElapsedMs` (живое, обновляется тиком); `isExpired` по `displayElapsedMs`. Дисплей таймера (целые секунды) UI берёт из контекста session — «глупый UI».
- Печатаемое — непрерывный `TypingStream`; **`drill` не называет печатаемое** (drill = строка контента + единица дозагрузки; границы drill'ов в потоке невидимы). Сводка — на **чекпоинтах** (перед дозагрузкой и в конце сессии), не per-drill: свёртка профиля аддитивна, EWMA сохраняет порядок → детализация не важна, а свежий профиль виден следующей порции.
- Конец сессии решает **таймер**, не длина потока: истечение → `draining` (дозагрузки нет, очередь допечатывается; страховочный таймаут `DRAIN_CAP_MS` не даёт зависнуть) → `done` (финальный чекпоинт + `SESSION.COMPLETE` родителю). `lessonComplete` снят.
- Пауза замораживает И таймер, И печать: `KEY_PRESS`/`TYPING.ADVANCED` объявлены только в печатающих подсостояниях, у `paused` их нет.
- Гость: `drillNext` авторизации не требует (печатает), `drillRecord` auth-required → на чекпоинте гостя сводка считается, запись профиля пропускается.
