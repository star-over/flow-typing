# Подбор drill'ов — заявкой требований, а не оценкой сложности

Классическая схема адаптивных тренажёров — посчитать сложность каждого упражнения и выбирать ближайшие к уровню пользователя. Отвергнута: сложность layout-зависима (одна и та же биграмма «но» — same-finger на йцукен и лёгкое движение в других парах) и персональна (труден drill, бьющий в слабости конкретного пользователя) — любой хранимый скаляр лжёт. Вместо этого Skill Profile компилируется в **DrillRequest** — декларативные требования к порции: жёсткие (символьная раскладка, потолок репертуара) исполняются индексной предварительной фильтрацией, мягкие (фокус-паттерны, длина слова, анти-повтор) — ранжированием по близости. Сложность нигде не хранится и не вычисляется как число.

## Considered options

- **Score-and-rank** — `difficulty(drill, layout, profile)` по всему корпусу: не ложится на индексы БД, необъяснима («сложность 7.3» не отвечает на «почему этот drill»), каждая правка формулы — пересчёт мира.
- **Сложность-колонка в drill** — лжёт для всех раскладок, кроме одной, и не может быть персональной в принципе.

## Consequences

- Заявка — интерфейс на вырост: сегодня её исполняет подбор по корпусу, завтра ту же заявку может исполнять генератор контента («нет drill'ов под заявку — сгенерируй и положи»).
- Корпус остаётся layout-нейтральным (символьное пространство); моторное знание компилируется в символьные множества per раскладка (справочники классификации), а не в мету drill'ов.
- Думать о трудности всё равно нужно — но в двух интерпретируемых местах: установка значений ручек (термостат, ADR 0004) и мера близости при подборе (политика ослабления мягких требований).
