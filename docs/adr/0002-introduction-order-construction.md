# Канонический порядок ввода строится по эталонному словарю языка, не по живому корпусу

Порядок ввода дорог в изменении (orderVersion → миграция профилей пользователей), а Drill Corpus — живой и растущий; привязывать стабильное-дорогое к изменчивому-дешёвому нельзя. Решение: содержимое `INTRODUCTION_ORDER` вычисляется скриптом — домашний ряд целиком (методическая постановка рук) → далее жадная максимизация прироста доступного словаря по стабильному частотному словарю языка → ручные моторные поправки → запекание в статический справочник с версией. Покрытие живого корпуса меряется против готового порядка («доступно drill'ов на префиксе k») как контентная метрика и на порядок не влияет.

## Considered options

- **Чисто методический (по рядам)** — частотность языка игнорируется, доступный словарь растёт медленно, новичок долго сидит на словах домашнего ряда.
- **Чисто частотный по буквам (keybr-style)** — статичное ранжирование не видит, что слово доступно только при введённых *всех* его буквах: маржинальная ценность буквы зависит от уже введённого префикса. Плюс раскидывает новичка по рядам с первых же букв, до постановки рук.
- **Жадный по живому корпусу** — каждое пополнение корпуса намекало бы на пересчёт порядка; зависимость перевёрнута.

## Consequences

- Нужен внешний эталон: частотный словарь top-N слов per язык (выбор источника — на этапе реализации).
- Скрипт построения порядка — одноразовый инструмент с ручной валидацией результата, не runtime-компонент.
- Появляется «контентный радар»: кривая `доступно drill'ов на префиксе k` показывает дыры корпуса до того, как в них упрётся пользователь.
