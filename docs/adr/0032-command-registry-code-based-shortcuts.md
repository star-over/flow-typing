# Пользовательские действия: единый реестр, распознавание по event.code, триггер 1:1

> Статус: принят · 2026-07-20 · Реализация: src/lib/user-actions/, src/lib/platform.ts, src/components/ui/KeyHint.svelte

## Context

Приложению нужны клавиатурные действия (не печать) двух видов: **аккорды-команды**
(`Cmd+,` → настройки, `Cmd+.` → статистика) и **навигационные клавиши** управления
тренировкой (`Escape` → пауза/возобновление, `Enter` → рестарт), плюс визуальные подсказки
сочетаний на кнопках и в меню, чтобы пользователь видел доступность клавиатуры. До
этого глобальный keydown в `+layout.svelte` отправлял ВСЕ клавиши в
`keyboardMachine` без фильтра модификаторов — `Cmd+K` во время тренировки
засчитывался как опечатка.

Первая итерация развела эти два вида по разным механизмам: аккорды — через новый
диспетчер, а навигационные клавиши — через канал `NAVIGATION_KEY` внутри машин
(`keyboardMachine` распознавал Escape/Enter, `appMachine` guard'ами `isEscape`/`isEnter`
разводил их в `PAUSE`/`RESUME`/`START_TRAINING`). Два механизма для одного понятия —
«нажатие клавиши запускает действие приложения» — размазывают карту клавиш между
машинами и диспетчером: подсказка и поведение могут разойтись, а добавление действия
трогает два места.

Проект мультиязычный: пользователь постоянно переключает системные раскладки
(qwerty/йцукен). `KeyboardEvent.key` зависит от раскладки и модификаторов
(Cmd+, на русской раскладке даёт key='ю'; Shift/Alt мутируют значение key).

## Decision

Один **реестр пользовательских действий** (`src/lib/user-actions/user-actions.ts`) —
источник истины для действий обоих видов: из него читают диспетчер keydown и компонент
`KeyHint` (визуальная подсказка); будущая палитра (Cmd+K) подключится к тому же
реестру, её поля (group/keywords/titleKey) добавятся вместе с ней, не заранее.
Действие = `{ id, trigger, when, run(context) }`; `UserActionId` — SCREAMING_SNAKE по
аналогии с событиями XState.

**Триггер строго 1:1.** У действия ровно один триггер — аккорд с модификатором
(`binding`) ИЛИ одна голая клавиша (`key`), без альтернатив и массивов. KeyHint
рендерится из того же триггера: альтернативы дали бы подсказке соврать о том, что
слушает диспетчер. Тип `UserActionTrigger` кодирует «одно из двух» объединением с
`never`, поэтому «и binding, и key» невыразимо.

**Единый канал.** Диспетчер (`dispatch.ts`, врезка в `handleKeyDown` до
`appActor.send`) сопоставляет и аккорды, и голые Escape/Enter — навигационные клавиши
больше не отдельный канал. Канал `NAVIGATION_KEY` из машин упразднён: `keyboardMachine`
и `appMachine` не знают Escape/Enter, а получают уже готовые семантические события
`PAUSE`/`RESUME`/`START_TRAINING`. Голая навигационная клавиша гасится глобально даже
без открытого действия (`isUserActionKey`), чтобы не протечь в `keyboardMachine`
опечаткой.

**Гейт по состоянию.** `when: 'always' | StateValue | StateValue[]` сверяется с
состоянием `appMachine` (через `context.isActive` → `inState`). Одна физическая клавиша
ведёт к разным действиям по состоянию: `Escape` в `{training:'running'}` → пауза, в
`{training:'paused'}` → возобновление; `Enter` в `{training:'paused'}`/`sessionComplete`/
`sessionError` → рестарт. Диспетчер берёт первое действие, чей гейт открыт и триггер
совпал.

Распознаватель — собственный, без зависимостей, **по `event.code`** (физическая
позиция), не по `key`: сочетания инвариантны к активной раскладке, тип аккорда —
`KeyCapId`. `mod` распознаётся кроссплатформенно по `metaKey || ctrlKey`
(«Mod»-приём): на Mac срабатывает и `Ctrl+,` — теневой вариант сочетания,
которого нет в подсказке; принимаем как цену простоты. Тип `KeyBinding` требует хотя бы один
командный модификатор (mod или alt): binding из одного shift запрещён типом, чтобы
подсказка не показывала сочетание, которое не слушаем.

Исключения фильтра аккордов: `event.repeat` (auto-repeat не дёргает действие),
`ctrl+alt` (AltGr на Windows — канал печати), keydown самого модификатора. Действия не
перехватываются из input/textarea/select/contenteditable (input-focus guard).

KeyHint (`src/components/ui/`) и `aria-keyshortcuts` читают триггер из реестра через
общие функции (`keyHintPropsForTrigger`, `formatAriaTrigger` в `platform.ts`),
разбирающие форму триггера один раз: аккорд → глифы/`Meta+,`, голая клавиша →
глиф/`Escape`. Подсказка `aria-hidden` и скрывается на touch-only устройствах;
`aria-keyshortcuts` ставится на сам триггер (кнопку/пункт меню).

## Considered options

- **Два механизма (аккорды в диспетчере, навигация через `NAVIGATION_KEY` в машинах)**:
  статус-кво до унификации. Отклонён — карта клавиш размазана между машинами и
  диспетчером, подсказка и поведение могут разойтись, добавление действия трогает два
  места; после унификации машины — чистые состояния, вся карта в одном файле.
- **Массив триггеров на действие** (несколько клавиш-синонимов): отклонён — KeyHint
  показывает один триггер; синонимы либо соврут в подсказке, либо потребуют выбирать
  «главный» триггер, что и есть 1:1, только неявно.
- **Распознавание по `event.key`** (рекомендация статей по i18n-shortcuts):
  отклонено — key нестабилен на пунктуации и под Shift/Alt, а флагманские
  сочетания проекта пунктуационные; безопасное множество key (A-Z, 0-9) не
  покрывает Cmd+,. Для qwerty/йцукен (обе физически QWERTY) code позиционно
  стабилен.
- **tinykeys**: живой микро-пакет, умеет code-распознавание. Отклонён: реестр
  обязателен в любом случае, а распознаватель аккордов тривиален. Остаётся
  сменной деталью, если появятся vim-последовательности (g затем d).
- **mousetrap / hotkeys-js**: отклонены — построены на deprecated which/keyCode,
  некорректны вне US-раскладок.
- **Переиспользование `KeyCap.svelte` для подсказки**: отклонено — сценический
  компонент (finger-раскраска, центр-якорь рук, press-result), мёртвый груз для
  инлайн-подсказки; заведён доменно-нейтральный `KeyHint`.
- **Зависящее от платформы распознавание mod** (Cmd строго на macOS, Ctrl строго
  на остальных): отклонено — требует передачи платформы в диспетчер ради
  отсечения теневого варианта, вреда от которого нет.

## Consequences

- Новое действие = одна запись в реестре (аккорд или голая клавиша); визуал,
  aria и поведение не расходятся структурно.
- Навигационные клавиши и аккорды делят один канал; карта клавиш живёт в одном
  файле. Машины лишились канала `NAVIGATION_KEY` (guard'ы `isEscape`/`isEnter`,
  состояние `recognizedNavigation`) — они чистые состояния, ведомые только
  семантическими событиями `PAUSE`/`RESUME`/`START_TRAINING`.
- Триггер 1:1 запрещает действию два входа; если понадобятся синонимы клавиш —
  пересмотреть новым ADR (структурную цену платим ради честности подсказки).
- Печать меняется в одной точке: аккорды Cmd/Ctrl/Alt больше не доходят до
  `keyboardMachine` (это и фикс бага-опечатки). Печать через AltGr (ctrl+alt)
  сохранена; **Alt-only считается командным каналом** — при появлении раскладок
  с Alt-only-символами (конфликт с моделью ADR 0017) политику пересмотреть новым ADR.
- На macOS у mod-действий есть не показанный в подсказке вариант через Ctrl —
  осознанная цена Mod-конвенции.
- Браузер-зарезервированные сочетания (Cmd+W/T/N/Q) недоступны приложению —
  выбирать аккорды вне этого множества; конфликт чинится в одном месте.
- Глифы ⌘⌥⇧ не входят в unicode-range Geist Mono — KeyHint рендерит --font-sans.
- Ярлыки клавиш (`Ctrl`, `Alt`, `Shift`, `Esc`, `,`, `K`) рендерятся как данные
  физической раскладки, минуя словарь, — осознанное исключение из ADR 0022
  (по прецеденту подписей клавиш в `KeyCap.svelte`); не UI-копия.
- Глиф KeyHint рендерится `text-primary` на `surface-accent` (решение 2026-07-20):
  на размере sm приглушённый глиф нечитаем (точка и запятая сливаются), а
  `text-primary` — единственная существующая роль, проходящая WCAG 4.5:1 на
  `surface-accent` во всех темах; новая роль `--color-kbd-*` не заводится.
