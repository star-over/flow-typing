# Репертуар — префикс канонического порядка ввода, не множество символов

> Статус: принят · 2026-06-13, уточнён под модель KeyLadder 2026-06-15 · Реализация: в коде

Адаптивной выдаче drill'ов нужен жёсткий фильтр «в drill'е только открытые пользователю клавиши», но проверка подмножества не индексируется в Convex. Решение: фиксируем порядок открытия клавиш — **KeyLadder** (рукотворный справочник с версией per Layout Context, в коде; шаги по геометрии — ADR 0002), репертуар пользователя кодируем одним числом — `openedSteps` (число открытых шагов), а каждому drill'у пакетно считаем производное поле `stepLevel` (макс. шаг среди его клавиш) в отдельной таблице отбора. Subset-проверка вырождается в индексируемое скалярное сравнение `stepLevel < openedSteps`.

## Considered options

- **Репертуар-множество** (массив символов в профиле, subset-фильтр в теле запроса) — гибче (допускает «дыры» и персональный порядок ввода), но не индексируется, раздувает профиль и не даёт инварианта префикса. Отвергнуто, потому что гибкость мнимая: персонализировать порядок *незнакомых* символов невозможно в принципе (по ещё не набранным буквам нет данных), а качество уже введённых — зона ответственности Weakness Map, не репертуара.

## Consequences

- KeyLadder глобальна per Layout Context; персонален только темп (`openedSteps`).
- Смена KeyLadder после запуска — миграция: `version`, пересчёт `openedSteps` с сохранением фактически введённого, пакетный пересчёт таблицы отбора (`stepLevel`).
- Конвейер корпуса параметризован символьной раскладкой и проверяет членством: все символы drill'а (`uniqueSymbols`) принадлежат набору символов раскладки (drill с символом вне набора отклоняется). Это и даёт семантику «drill совместим с раскладкой», заменяя определение языка точной проверкой (закрывает корень known-issue #1).
- У опытного пользователя `openedSteps` сразу на максимуме — фильтр вырождается без специальных режимов в коде.
