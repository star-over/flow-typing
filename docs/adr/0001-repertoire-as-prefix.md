# Репертуар — префикс канонического порядка ввода, не множество символов

Адаптивной выдаче drill'ов нужен жёсткий фильтр «в drill'е только введённые пользователю символы», но проверка подмножества не индексируется в Convex. Решение: фиксируем канонический порядок ввода символов (статический справочник с версией per Layout Context, в коде), репертуар пользователя кодируем одним числом — длиной префикса этого порядка, а каждому drill'у пакетно считаем производное поле `minRepertoireIndex` (позиция его самого «позднего» символа) в отдельной таблице отбора. Subset-проверка вырождается в индексируемое скалярное сравнение `minRepertoireIndex ≤ repertoireSize`.

## Considered options

- **Репертуар-множество** (массив символов в профиле, subset-фильтр в теле запроса) — гибче (допускает «дыры» и персональный порядок ввода), но не индексируется, раздувает профиль и не даёт инварианта префикса. Отвергнуто, потому что гибкость мнимая: персонализировать порядок *незнакомых* символов невозможно в принципе (по ещё не набранным буквам нет данных), а качество уже введённых — зона ответственности Weakness Map, не репертуара.

## Consequences

- Порядок ввода глобален per Layout Context; персонален только темп расширения репертуара.
- Смена порядка после запуска — миграция: `orderVersion` в профиле, пересчёт префиксов с сохранением фактического набора символов пользователя, пакетный пересчёт таблицы отбора.
- Конвейер корпуса валидирует на входе: все `unique_symbols` drill'а присутствуют в порядке его языка (drill с символом вне порядка отклоняется) — это же даёт семантику «drill совместим с раскладкой» (закрывает корень known-issue #1).
- У опытного пользователя префикс сразу полный — фильтр вырождается без специальных режимов в коде.
