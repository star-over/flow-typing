### Phase 9 — Apple provider

**Что:** Sign in with Apple через `@auth/core/providers/apple`. Profile нормализуется в стандартный `{ id, name, email, image }` shape, поэтому `createOrUpdateUserHandler` (см. Phase 2/4/8) подойдёт без правок.

**Почему отложено:** Технические предпосылки вне нашего контроля:
- Apple Developer Program — **$99/год**, требует Apple ID.
- **Domain verification** — Apple требует `apple-developer-domain-association.txt` на верифицированном домене. Два варианта:
  - (a) production cloud Convex deployment + собственный production-домен с DNS-контролем (classical path);
  - (b) попытка верифицировать сам `*.convex.site` через Convex HTTP routes — не подтверждено, нужна отдельная research-задача перед коммитом к Phase 9.

**Driver:** Бизнес-need в iOS-аудитории ИЛИ user request ИЛИ кто-то закрывает Apple-domain-verification research как Phase 9.0.

**Шаблон:** Phase 8 (Yandex) — после `OAuthProviderId` extract'нут в shared module, добавление 4-го провайдера сводится к:
- расширить union `'github' | 'google' | 'yandex' | 'apple'` в `src/lib/auth/auth.types.ts` (одна строка);
- `import Apple from '@auth/core/providers/apple'` + добавить в `providers` array;
- кнопка + 4 theme tokens × 5 файлов.

Final reviewer Phase 8 отметил, что 4 идентичных CSS-блока в SignInScreen — natural checkpoint решать про `--sign-in-screen-btn-base-*` extract. На 4-м провайдере это становится разумнее.

**Размер:** ~Phase 8 (1 день) + Apple Developer enrollment overhead.

### Phase 10 — Account linking V2

**Что:** UI в `/settings` + mutation для добровольного слияния двух `users`-строк одного человека (e.g. «у меня GitHub-аккаунт и Google-аккаунт с одним email — хочу объединить»). Миграция связанных `userSettings` / `sessions`. Conflict resolution (что делать при конфликте email / settings / sessions между двумя записями).

**Почему отложено:** Текущий инвариант «провайдер = аккаунт» (явно НЕ link-by-email) защищается на handler-уровне (`convex/auth.test.ts:44-61`) и принят пользователем как by-design. Account linking — противоположная семантика; не делать без явного запроса пользователей, иначе ломаем безопасность ради функции, которую никто не просил.

**Driver:** Поступление >1 запроса от пользователей «у меня два аккаунта, можно их объединить?». До этого — не трогать.

**Размер:** большая фаза (несколько подзадач: merge mutation, settings/sessions migration, UI, edge cases). Не минимальная.

### Phase 12 — Telegram

**Что:** Sign-in через Telegram Login Widget. Технически **не OAuth 2.0**: собственный HMAC-flow Telegram'а (виджет → query-параметры `{id, first_name, last_name, username, photo_url, auth_date, hash}` → HMAC-SHA256-проверка через bot token как secret). Стандартного провайдера в `@auth/core/providers/*` нет (проверено через ctx7 на 2026-06-11).

**Почему отложено:**
- Нет user request'а и подтверждённой бизнес-потребности.
- Технически — не «ещё один OAuth-провайдер»: flow не ложится на абстракцию `@auth/core` (нет `client_id`/`client_secret`/`authorization_code`).
- Требует schema-правки: Telegram **не отдаёт email**, поэтому `users.email` должен стать `v.optional(v.string())` и `createOrUpdateUserHandler` должен корректно обрабатывать email-less аккаунты. Это первый провайдер, который ломает инвариант «email всегда есть».
- Bot token — отдельный класс секрета (не client secret OAuth-приложения, а identity самого бота). Хранение строго через `npx convex env set AUTH_TELEGRAM_BOT_TOKEN` — никогда в client-bundle.

**Driver:** Бизнес-need в Telegram-аудитории (потенциально большая в РФ/СНГ) ИЛИ user request.

**Не шаблон Phase 4/8.** Один из трёх возможных подходов (выбрать в момент планирования):
1. **`Credentials`-провайдер `@auth/core`** — `authorize(credentials)` принимает Telegram payload, делает HMAC-проверку, возвращает `{ id, name, image, email: null }`. Самый лёгкий путь, но `Credentials` в Auth.js считается «advanced» и плохо комбинируется с другими session-стратегиями.
2. **Custom OAuth-провайдер для `@auth/core`** — насилуем Telegram-flow в OAuth-абстракцию. Избыточно и плохо читается.
3. **Convex HTTP-route напрямую** — endpoint в `convex/http.ts` принимает Telegram callback, проверяет HMAC, вручную создаёт сессию через internal API Convex Auth. Самый низкоуровневый, но честный — flow не насилуется.

UI: вместо нашей кнопки `<button>Войти через ...</button>` рендерится `<script src="https://telegram.org/js/telegram-widget.js">` с data-атрибутами bot username и redirect URL. Это отдельный UX-кейс (внешний скрипт от Telegram, не наша вёрстка) — не вписывается в текущий `SignInScreen.svelte` без рефакторинга.

Готовых community-пакетов уровня `@convex-dev/auth-telegram` на 2026-06-11 не найдено (WebSearch не показал релевантных).

**Размер:** существенно больше Phase 4/Phase 8. Сопоставим с Phase 11 (SberID) по сложности custom-интеграции, плюс уникальные осложнения (schema-правка для nullable email, внешний `<script>` в UI). Перед стартом — отдельная research-задача Phase 12.0 на выбор подхода (1/2/3 выше) + проверка, появился ли community-пакет за прошедшее время.

### Переключатель языка в шапке — заменить нативный select на фирменное выпадающее меню (вариант A)

**Что:** Переключатель языка интерфейса в шапке реализуется как компактный нативный `Select` (вариант B: повторно использован `Select.svelte` из настроек). Возможная замена — **вариант A**: своё компактное выпадающее меню (иконка глобуса + код языка «EN», по клику — всплывающий список языков в стиле меню пользователя `UserMenu`). Требует собственного компонента + контракт-токенов темы (`--header-lang-*` × 4 темы + `_template.css`); поведение всплывающего меню снимается с `UserMenu` (`<details>` + закрытие по клику-вне/Escape).

**Почему отложено:** Нативный select закрывает задачу минимальным кодом (готовые токены, доступность из коробки), а языков всего два. Фирменное всплывающее меню оправдано, когда (а) нативное ОС-меню начинает спорить с «тихим точным» хромом визуально, либо (б) языков становится 3+, где компактный триггер-код важнее полной надписи. Решение по замене принимаем отдельно при необходимости (осознанно зафиксировано на проектной сессии 2026-07-09).

**Driver:** Появление 3-го языка интерфейса ИЛИ решение довести хром до фирменного всплывающего меню (визуальный дискомфорт от нативного меню).

**Размер:** маленький-средний (компонент dropdown + контракт-токены × 5 файлов; логика всплывающего меню повторно используется из `UserMenu`).

### Session vs drill — ✅ реализовано (2026-06-17)

**Реализовано** как `sessionMachine` — отдельный слой между `appMachine` и `trainingMachine` (**ADR 0007**, план `docs/plans/2026-06-17-session-machine.md`). Печатаемое — непрерывный `TypingStream` (drill'ы склеены через пробел, границы невидимы); завершение решает **таймер сессии**, а не длина потока; `lessonComplete` снят. `trainingMachine` стал чистым классификатором-внуком и шлёт `TYPING.ADVANCED` вверх. Агрегат уровня сессии для Phase 6 (`/stats`) получил рамку (таймерная сессия); consumer Phase 6/7 реализован (см. выше).

### Keyboard events на `/settings` и `/stats` — фильтр по pathname

**Что:** `src/routes/+layout.svelte` навешивает глобальный keyboard listener (`<svelte:window onkeydown/up/blur>`), который шлёт `KEY_DOWN`/`KEY_UP`/`PAUSE` в `appActor` независимо от текущего роута. На `/settings` и `/stats` FSM сидит в `idle`-состоянии, эти события безвредны (keyboardMachine их корректно отбрасывает). Если когда-то понадобится — например, для горячих клавиш на `/settings`, которые не должны просачиваться в `training` — добавить фильтр по `$page.url.pathname` в layout-listener.

**Почему отложено:** Нечего фильтровать. Нет горячих клавиш, конфликтующих с FSM. Защита на уровне `keyboardMachine.isTextKeyGuard` уже работает.

**Driver:** Появление любого keyboard-handling требования специфичного для `/settings` или `/stats` (например, «Esc возвращает на главную»).

**Размер:** маленький (3-5 строк в `+layout.svelte`).

### Интеграция Madge для анализа зависимостей и модулей

**Что:** Добавить в проект [madge](https://github.com/pahen/madge) — инструмент для визуализации и анализа зависимостей между модулями TypeScript/JavaScript. Настроить скрипты в `package.json` и/или `Makefile` для генерации отчётов:
- список циклических зависимостей (`madge --circular src/`);
- граф импортов для ключевых директорий (`src/machines/`, `src/lib/`, `convex/`);

**Почему отложено:** Нет острой технической проблемы прямо сейчас. Проект пока небольшой, циклических зависимостей не выявлено вручную, архитектурные слои ещё стабилизируются. Преждевременное внедрение анализатора без чётких правил "что считать ошибкой" даст шум, а не пользу.

**Driver:** Любой из:
- рост числа модулей и появление подозрений на циклические зависимости;
- рефакторинг архитектуры (например, выделение `shared/`, разделение `src/lib/` на подслои);
- подготовка к введению архитектурных линтов / CI-проверок (dependency-cruiser, ESLint-boundaries и т.п.);
- необходимость документировать модульную структуру для новых участников.

**Размер:** маленький — установка пакета, 1–2 npm-скрипта, опционально конфиг-файл `.madgerc` с игнорированием `*.svelte`/`*.css` и прочих не-JS импортов, если madge не поддерживает их из коробки.

**Примечания:**
- Madge анализирует AST и не исполняет код; для SvelteKit/Vite-проекта может потребоваться фильтрация или трансформация путей.
- Рекомендуется запускать локально и в CI только как **информационный** отчёт на первом этапе, а не как жёсткий gate, пока не выработаны правила.

### Параметры адаптивности в `/settings`

**Что:** `docs/04-settings-management-system.md` §4.1.4 и §4.3 явно резервируют место в `/settings` под «параметры адаптивности (стратегия выбора, длительность сессии и т.п.)» и «профиль пользователя». Сейчас `/settings` содержит только `interfaceLanguage` и `theme`; адаптивные параметры физически некуда положить.

**Почему отложено:** Адаптивная система реализована частично (рабочая петля + рост репертуара); термостата и ручек трудности ещё нет (этапы «Фокус»/«Термостат» в `docs/plans/auto-flow.md`). Элементы управления без backing-логики были бы плацебо.

**Driver:** Начало работ над «Dynamic Flow» (адаптивная подача drill'ов). UI-параметры подключаются вместе с движком, не раньше.

**Первый кандидат** (зафиксирован на сессии проектирования адаптивной выдачи, 2026-06-12): предпочтение длины слов — «короче / нейтрально / длиннее» как *смещение* поверх термостата, не сырое значение (ручной выбор не должен воевать с адаптивным движком). Обоснование «длина слова = длина моторной программы между пробелами-вдохами» — см. CONTEXT.md (DrillRequest).

**Размер:** зависит от scope адаптивной системы. Чисто UI — маленький; вместе с движком — отдельный umbrella-план уровня auth.

**Driver:** Запрос en-аудитории ИЛИ плановый заход в qwerty-этап.

**Размер:** средний (лестница — кураторская работа + конвейер + rebuild + тесты).

До этапа дефолт настроек `en`/`qwerty` ведёт нового пользователя в пустую сессию — принято осознанно (решение 2026-07-03: дефолт не меняем; публичных пользователей нет, вход скоро обязателен — ADR 0012; закроется самим этапом).

### Чередование большого пальца для `Space` — исправить расчёт пальца-виновника

**Что:** В `src/lib/hands-scene.ts` (`determineTypingContext`, ветка `keyId === 'Space'`) палец, которому приписывается ошибочное нажатие пробела, считается сырым первым срезом: целевая рука левая → винится `R1`, иначе → `L1` (большой палец противоположной руки). Это неверно. По-правильному целевой большой палец для пробела должен **чередоваться** в зависимости от того, каким пальцем нажата предыдущая клавиша. Пока логика не кристаллизована, на текущем этапе берётся `L1`/`R1` без чередования.

**Почему отложено:** Чередование пробела — нетривиальный тюнинг, не препятствие. Требует целостного решения (закон выбора большого пальца + протяжка контекста предыдущего нажатия в `determineTypingContext` + фикстуры на оба направления), а не точечной заплатки. Сознательно не смешано с типовой переработкой типа ViewModel (кандидат 3 аудита `docs/plans/2026-06-28-architecture-deepening-audit.md`), чтобы не запутывать сломанную область и саму переработку в одном коммите.

**Driver:** Заход в тюнинг визуализации ошибок пробела ИЛИ жалоба на неверную подсветку пальца при промахе по `Space`.

**Размер:** средний — не одна строка: закон чередования + контекст предыдущего нажатия + фикстуры на оба направления.

**Причина (breakdown по функциям, 1.06 ГБ за цикл, dashboard 2026-07-09):**
- `selectionIndex.insertBatch` — **973 МБ (92%)**: вставка 10–14k строк `drillSelectionIndex` + поддержка B-дерева компонента `@convex-dev/aggregate` (на каждую вставку — чтение+запись узлов дерева). **Доминирует.**
- `selectionIndex.drillsPage` — 53 МБ (5%): постраничное чтение всех drills при пересборке.
- `clearLayoutPage` + `btree.deleteTreeNodes` — ~16 МБ: очистка старого индекса и дерева.
- `drill.drillNext` — 21 МБ: пользовательская выдача (норма, идёт из кэша).

Дорого не чтение, а **запись индекса + поддержка `aggregate`**, помноженные на десятки `rebuild` за сессию (паритет лестницы, билингва, оживление qwerty, проверки).

**Что (варианты, по убыванию эффекта):**
1. **Реже вызывать `rebuild`** — главный драйвер всплеска (за сессию — десятки прогонов ради проверок). Ноль кода.
2. **Инкрементальный индекс** вместо full clear+rebuild: при добавлении drills дописывать только новые строки. (Смена нарезки `ladderStep` всё равно трогает все → инкремент помогает при доливке корпуса, не при смене лестницы.)
3. **Пакетное население `aggregate`** — если компонент даёт путь заполнить дерево bulk-ом без per-row операций (проверить API `@convex-dev/aggregate`); именно per-row btree и есть 92%.
4. **Локальный расчёт `stepLevel`** (исходная идея) — `computeStepLevel`/`symbolToStep` чистые (`shared/`), раскладки локальны (`src/layouts/`), считать можно в `build-corpus` и импортировать индекс напрямую. Но экономит лишь ~5% (`drillsPage`): 92% (`insertBatch`+aggregate) — серверная запись, локально не убирается. Полезно как часть п.2/3, не само по себе.

**Оговорка:** `drills` — layout-agnostic (`stepLevel` разный по раскладкам → индекс на раскладку); `aggregate` серверный by design (даёт O(log n) случайный отбор, ADR 0009).

**Почему отложено:** Оптимизация **dev-стоимости**, не корректности. Лимит I/O сбрасывается помесячно; на production `rebuild` — редкое событие (смена корпуса/лестницы), своя квота. В установившемся режиме `drillNext` идёт из кэша (ADR 0009), расход копеечный.

**Driver:** Повторяющаяся боль с dev-квотой I/O ИЛИ регулярное обновление корпуса.

**Размер:** п.1 — ноль; п.2/3 — средний (инкремент + путь `aggregate`); п.4 — маленький, но малоэффективный в одиночку.

### Тихий сбой дозагрузки в середине сессии — сделать деградацию видимой

**Что:** `src/machines/session.machine.ts`, состояние `refilling`: `onError: { target: 'running' }`. Когда дозагрузка (`fetchDrills` при низкой воде очереди) падает — сеть оборвалась уже в середине сессии — сессия молча продолжает тем, что в буфере, без сигнала пользователю. При повторяющихся сбоях хвост потока иссякнет раньше таймера: печатать станет нечего, а обратный отсчёт продолжит идти. P0-5 сделал видимым сбой **старта** сессии (первый fetch → состояние `error` → экран `sessionError`), а mid-session refill сознательно оставлен как есть.

**Почему отложено:** Это разумная деградация, а не тихий обрыв — пользователь продолжает печатать буфер, единичный сбой refill безвреден (порог `REFILL_THRESHOLD_SYMBOLS` даёт запас). Расширять error-UX на середину сессии в рамках P0-5 = расползание scope. Полноценное решение нетривиально: нужно отличить единичный сбой (продолжаем) от устойчивого (буфер иссякает → показать), не ломая идущий таймер и инварианты ADR 0007 (таймер охватывает running+refilling, `exit timing` фиксирует сегмент один раз).

**Возможный подход (при возврате):** отдельное подсостояние в `sessionMachine` — при исчерпании буфера на фоне повторяющихся refill-сбоев заморозить таймер (как `paused`) и показать индикатор «переподключение…», с автоповтором дозагрузки и продолжением при успехе. Подсостояние-сестра `timing` (морозит таймер, но не теряет набранное и не ломает коммит сегмента). Симметрично тому, как старт-сбой стал первоклассным состоянием `error` (P0-5, вариант А владельца).

**Driver:** жалоба «счётчик тикает, а печатать нечего» на нестабильной сети ИЛИ заход в offline/устойчивость-к-сбоям (пересекается с P2 «Offline / PWA / persistent-очередь записи» плана запуска).

**Размер:** средний — новое подсостояние `sessionMachine` + guard устойчивого сбоя + индикатор + фикстуры (юнит-тест через `provideSession` с последовательностью сбоев дозагрузки, уже поддержан флагом `fetchRejects`).

### Шифрование и управление версиями env-файлов (research 2026-07-13)

**Что:** локальные env-файлы (ключи, токены, конфиг «для разных целей») лежат на диске, в git их класть нельзя — но нужно хранить в системе контроля версий и держать как источник правды. Внедрить один из подходов. Разбор семейств и таблица — в истории чата research-сессии; ключевая развилка: git-история (зашифровано в репозитории) vs история сервиса (rollback + audit внутри платформы).

**Три кандидата (выбор в момент старта):**
1. **SOPS + age** — де-факто стандарт «зашифровать и в git». Шифрует **значения**, ключи оставляет открытыми → diff'ы осмысленные («поменялся `DB_PASSWORD`»). Офлайн, без привязки к поставщику, без оплаты; управление версиями = обычный git. `sops exec-env` подставляет переменные в дочерний процесс, на диск ничего не пишет. Несколько recipients (`age1…`) → масштабируется на команду/CI без переезда. Корневой секрет — приватный `age`-ключ (хранить в OS keychain). **Наиболее вероятный выбор** под соло + минимализм.
2. **dotenvx** — «dotenv с шифрованием» от автора `dotenv`. `dotenvx encrypt` → пара `DOTENV_PUBLIC_KEY` (в файле) + `DOTENV_PRIVATE_KEY` (в `.env.keys`, в `.gitignore`), ECIES. Проще для тех, кто уже на `dotenv`; минус — шифрует значения → git-diff'ы нечитаемые.
3. **1Password (`op` CLI)** — если 1Password уже в ходу. Secret references `op://vault/item/field` прямо в шаблоне `.env`, `op run -- …` подставляет в рантайм. Версии — item history (в сервисе, не в git). Меньше git-native, но ноль нового инструмента.

**Смежное (учесть при заходе):** backend-секреты Convex-функций логично держать не в этих файлах, а через `npx convex env set` (своя история/audit по развёртыванию) — тогда набор локальных env-файлов под шифрование становится меньше. Пересекается с «CI-автоматизация развёртывания» (GitHub secrets `CONVEX_DEPLOY_KEY` / `CLOUDFLARE_*`) и с bot-token хранением Phase 12.

**Почему отложено:** не препятствие — файлы работают локально. Это гигиена/устойчивость (потеря машины = потеря невосстановимых секретов; нет истории изменений). Driver'а «прямо сейчас» нет.

**Driver:** второй разработчик / вторая машина ИЛИ появление секрета, который больно потерять/невозможно перевыпустить ИЛИ заход в CI-развёртывание (там всё равно нужен упорядоченный источник секретов).

**Размер:** SOPS+age / dotenvx — маленький (настройка `.sops.yaml`/ключ + `.gitignore` + инструкция расшифровки в dev); 1Password — маленький, если аккаунт уже есть.

### Аудит тестов на привязку к жёстко зашитым константам (magic numbers/strings)

**Что:** пройти тестовый набор и найти проверки, сверяющие результат с **литералом, дублирующим значение из кода** (дефолт, порог, enum-строка). Такой тест — детектор изменений (change-detector): ломается на *осознанной* смене значения, ничего не выявляя (тавтология «значение равно самому себе»), и требует механической правки теста на каждое изменение константы. Для каждой находки разнести две роли: (а) проверка **поведения** → сверять с источником истины (импортировать константу/дефолт, не вшивать литерал); (б) защита от *опечатки* в самой константе → одна явная «канарейка», названная так, роль не подмешана в поведенческие тесты. Литералы оставлять лишь там, где значение — не дубль константы: осмысленный вход (пропуск валидного значения) или производное.

**Прецедент:** `src/lib/settings.test.ts` (2026-07-12) — смена дефолта длительности сессии `300→60` (`user-settings.ts`) сломала 3 теста, сверявших с литералом `300`, хотя баг отсутствовал. Разобрано: `76e8df3` (поведенческие проверки → ссылка на `DEFAULT_USER_SETTINGS`) + `b14594d` (одна явная канарейка `user-settings.test.ts`). Точечно закрыто; системный проход по остальным тестам — эта задача.

**Почему отложено:** не баг — тесты зелёные; это устойчивость/гигиена набора. Болезненный случай (settings) уже закрыт, острого драйвера сейчас нет.

**Driver:** повторное «поменял значение в коде → упал тест, не выявив баг» в другом месте ИЛИ плановая ревизия тестового набора. На заметку: `app.machine.test.ts`/`session.machine.test.ts` содержат много `durationSeconds: 300`, но там значение передаётся **явно в событии** (вход, не дубль дефолта) — при аудите отличать такие от настоящих сверок с константой.

**Размер:** аудит — маленький-средний (прогнать grep по числовым/строковым литералам в `**/*.test.ts`, классифицировать: дубль-константы vs осмысленный вход/производное); правки — по числу находок, механические.

## Прогрессия: отложенные направления (проработаны 2026-07-05)

Две крупные линии из исследовательской сессии 2026-07-05. Реализация отложена, но материал проработан — при возврате основная часть готова. Ветки — независимые, с чистого листа от master; exposure-фронтир рекомендуется первым (переопределяет, *что* такое readiness).

### Exposure-фронтир — рост репертуара запросом, `drillRecord` тонким писателем

**Что:** текущую ступень (`openedSteps`) не хранить и не двигать записью `drillRecord`, а **вычислять запросом** из ячеек профиля. `drillRecord` худеет до записи ячеек; решение о росте перестаёт быть его заботой.

**Принятые решения:**
- Рубеж = `max(step)` по символам с `exposures > 0` (глубочайшая реально тронутая ступень). Монотонен by construction: предъявления только растут → множество тронутых только растёт → max только растёт. Хранимая метка и запись роста **не нужны**.
- Качество (точность/латентность) **не** решает рубеж-пол, только **край** — открывать ли следующую ступень (локальная проверка символов текущего шага). Просадка внизу рубеж не роняет: его держат предъявления, не качество.
- Суть — разделение: «где стоишь» = монотонный факт из данных; «идти ли дальше» = проверка качества на краю.
- Роль readiness меняется: из механизма роста → в сигнал качества для проверки открыть-следующий и показа `/stats`. Не выбрасывается.
- Согласуется с ADR 0001 (репертуар = факт ввода) и ADR 0008 (введённое не пропадает) — вернее нынешнего дизайна, где рубеж определяется качеством через readiness.

**Ключевой принцип (почему только монотонное):** запрос read-only → память не хранит → считает рубеж заново каждый раз → результат идёт за своим числом; упадёт число — упадёт рубеж (ступень закроется, символы вылетят из пула). Значит основа — только не убывающее число. Счётчики (`exposures`, `clean`) монотонны; доли/средние (точность, латентность-EWMA) — нет, а ячейки хранят кумулятивные счётчики, не пики долей.

**Препятствия/ограничения:**
- Монотонность — продуктовое требование (ADR 0008 §1): просевший символ не ронять из пула — забытое надо тренировать больше. Exposure-рубеж это держит.
- Исполнение: `drillNext` — query (ADR 0009), писать не может; `drillRecord` — единственная запись цикла. Exposure-рубежу запись не нужна вовсе.
- Стоимость **не** препятствие: история уже свёрнута в ячейки (per-symbol); рубеж = 1 read документа + O(символы) проход (это уже делает `computeRepertoireSnapshot`), а не «агрегировать всю историю».

**Открытые точки (заострить при возврате):** точная формула рубежа (случай «тронул верхний символ, не все нижние»); что именно решает открыть-следующий (readiness + debtLimit?); «мигание» ещё не тронутого следующего шага до первого нажатия (само-разрешается); педагогика (продвижение по практике vs по мастерству — подтвердить); cold-start дефолт (шаг 1).

**Проверки перед реализацией:** грилл против ADR 0001/0008 (`grill-with-docs`); **новый ADR поверх 0008** — локус роста writer-owned → query-derived, монотонность через exposures.

**Driver:** заход в переработку роста репертуара. **Размер:** средний (перестройка роста + новый ADR + правки `drillRecord`/`drillNext`/`/stats`).

### Платформа правил — создание правил в GoRules JDM с компиляцией в TS

**Что:** правила прогрессии (семья §6) создаются визуально в GoRules JDM (decision tables), компилируются **offline** в чистый TS, сверяются с нативным Zen как эталоном, исполняются как TS в V8. Движка правил в V8-пути нет. Решение зафиксировано в **ADR 0018**; здесь — проработанный материал сверх тела ADR.

**Принятые решения (кратко; полно — ADR 0018):** исполнение — генерённый TS, движка правил нет; цель компиляции — TS (не JsonLogic — тот отвергнут как промежуточный толкователь). Граница подмножества (сердце): только `decisionTable` + hit policy **First**, граф связанных таблиц (ценность DRD); подмножество ZEN в ячейках (сравнения, `and`/`or`/`not`, арифметика, поля, диапазоны, условные выражения). **Без** function-нод. Отложено: switch/expression-ноды, Collect, экзотические ZEN-функции.

**Препятствия / проверенные живьём факты (эталон — не терять):**
- native `@gorules/zen-engine` в V8 невозможен. napi, только в `"use node"` actions. Проверено: у `MutationCtx` нет `ctx.runAction` (ошибка исполнения `TypeError`, Request ID `37f0d4294ba1231c`); query в `"use node"` сервер отбивает дословно — «Only actions can be defined in Node.js». Standalone use-node action с Zen работает (`{action:"open"}`, 751µs). → native Zen годен только offline (node): как эталон и для создания правил, не в горячем V8-пути (query/mutation).
- `dmn-eval-js` в V8 падает: `moddle-xml` зовёт `setTimeout` (запрещён в queries/mutations). Проверено.
- `zen-engine-wasm` — требует SharedArrayBuffer/COOP-COEP, экспериментальный, в изолированной среде Convex-V8 не доказан.
- Готового JDM→JS/TS компилятора **нет** (проверены все репозитории org GoRules — интерпретаторы + редакторы). Zen — интерпретатор (встраивает QuickJS). Компилятор — наш.
- Function-ноды исключены: произвольный JS в QuickJS (`@gorules/quickjs`, napi) с HTTP/async/side-effects → в TS не переносится (среда), тащит недетерминизм в V8, убивает декларативность. Вычисления — в TS feature-extraction, не в правиле.

**Оценка сложности компилятора:** ограниченное подмножество (наши правила) — подъёмно: парсер подмножества ZEN + генерация кода графа таблиц ≈ **1.5–2k строк, недели** + хвост сопровождения (ZEN не наш, расширения на нас); «приемлемое качество» достижимо **благодаря сверке с Zen**. Полный JDM — месяцы + риск по точности (Zen считает в десятичных/`big.js` → float64 TS даёт другие числа), датам, null — **вне scope**. Дисциплина границы важнее самого кода.

**Проверки / стратегия качества:** differential-тест против нативного Zen (`zen(input) === generatedTs(input)` на per-rule корпусе, в node-vitest-проекте, в `make check-all`) — костяк, корректность через сверку с эталоном, не формальным доказательством. Первая проверка компилятора — readiness как JDM обязан дать те же результаты, что golden-тест `shared/repertoire/readiness.test.ts`. Первая ценность — DrillWindow (§6.2, зоны flow/boredom/frustration).

**Справка:** `zen-expression` (Rust-крейт) — авторитетная спецификация семантики ZEN (стековая VM); схема JDM документирована (`docs.gorules.io/reference/json-decision-model-jdm`); нативный Zen — эталон; готового кода к переносу нет.

**Driver:** коммит в визуальную систему правил принят (вариант «платформа»); реализация — при заходе в работу, оправдана масштабом семьи §6, не одним правилом. **Размер:** большая линия (компилятор + сверка + первое правило) — отдельный umbrella-план уровня auth.

### Тесты трёхслойной схемы тем — запереть слои, когда схема устоится

**Что:** Схема тем переводится на три слоя (решение 2026-07-17, обсуждение с владельцем): **слой 1** — приватное ядро темы, имена и количество произвольные, только абсолютные значения; **слой 2** — фиксированный словарь семантических ролей, прямой маппинг на слой 1 или калькуляция от него (`oklch(from …)`), новых абсолютов нет; **слой 3** — контракт компонента, только `var()` на слой 2, без литералов и формул. Правила держатся руками, пока не закрыты тестами:

- слой 2 и слой 3 не содержат абсолютных `oklch(<число> …)`;
- слой 3 — только `var(…)`: ни `oklch(from …)`, ни `calc`;
- слой 1 ни на что не ссылается (`var()` внутри него нет);
- каждый `var(--…)` в теме разрешается в объявление (сейчас **битую ссылку не ловит ничто** — ни сборка, ни `svelte-check`, ни контракт-тест: цвет молча исчезает).

Охват теста — список уже нормализованных тем (`const NORMALIZED = ['sepia']`), пополняется по мере миграции. Так тест зелёный с первого дня и не блокирует незавершённые темы.

**Почему отложено:** Схема кристаллизуется на одной теме (`sepia`), границу между слоями предстоит двигать; правила, запертые тестом до того как форма устоялась, будут держать за руки. Решение владельца — сперва довести схему на sepia, затем подтянуть light/dark/nord, тесты в конце, чтобы не терять фокус на фронтенд-части.

**Почему не забыть:** Без замка эталон откатывается — так уже произошло: `src/themes/CLAUDE.md` разрешает «литералы, формулы, ссылки — любая смесь», и sepia обросла копиями (`--sign-in-screen-background` дублирует `--color-surface` дословно; `--user-menu-authenticated-name-color` разъехался с `--color-text-primary` на 0.02). Всего таких дрейфующих токенов было 30 из 228 (аудит 2026-07-17).

**Индикатор готовности слоя 2:** различия слоя 3 между темами схлопнулись. Каждое различие = недостающая роль в слое 2 (`--keycap-home-ring`: light считает его подписью, sepia — маршрутом → нет роли «якорь дома»). На 2026-07-17: 52 различия, из них 30 — дрейф, 22 — реальные (10 × `--movement-path-*-marker` (ADR 0026 обещал свести в `--movement-path-route`), 10 × `--keycap-*-target-ring`, `--keycap-home-ring`, `--keycap-path-ring`). Ноль различий → слой 3 запекается в код (176 из 228 токенов уже сейчас дословно одинаковы во всех темах).

**Driver:** все четыре темы переведены на трёхслойную схему.

**Размер:** малый — четыре проверки регулярными выражениями поверх `src/themes/*.css` + список нормализованных тем; шов такой же, как в `src/themes/contract.test.ts`.

## Темизация — отложено до растворения контракта (L3)

Находки аудита тем (2026-07-17/18), сознательно **не** правленные сейчас: слой 3 (контракт компонента) — транзитные леса, которые растворяет будущая миграция (ADR 0029). Точечно чинить то, что уйдёт целиком — работа впустую; фиксируем, чтобы учесть при растворении L3 и миграции `light`/`dark`/`nord`.

- **П.2 — ссылка из L3 в ядро мимо ролей.** `--movement-path-marker-core: var(--ink)` тянет из слоя 3 прямо в ядро, минуя слой ролей. Завести роль `--color-marker-core` при растворении L3.
- **П.6 — асимметрии пар контракта.** Нет `--keycap-error-ring` (есть только `--keycap-correct-ring`); hover primary/success-кнопок зашит `opacity: 0.9` в `FooterActions.svelte` и не темизируется; комментарий в `KeyCap.contract.ts` про «зелёное кольцо» у correct-ring неточен — это `var(--color-gap)`, геометрия без цвета.
- **Тесты слоёв.** Нет абсолютов в L2/L3; L3 — только `var()` + структурные литералы; L1 ни на что не ссылается. Включить при стабилизации схемы — см. отдельную запись «Тесты трёхслойной схемы тем» выше.
- **П.9 мелочи.** Разнобой идиом калькуляций (`calc(l * .90)` против `calc(l - 0.08)`); семантика имени `--color-path-highlight` («подсветка» против «тихой направляющей»); `--radius-sm` используется без такого примитива (спасает fallback); `--user-menu-loading-color: var(--color-symbol-pending)` — семантическая натяжка; префиксы контракта `--hands-*`/`--landing-*` не по §6.3.
- **Контраст метки correct-keycap.** `--color-keycap-correct-background` несёт α 0.8; над пергаментом эффективный контраст метки ~2.85:1 — ниже 3:1 для крупного символа. Это пред-существующее свойство альфы заливки, не регрессия консолидации; проверить при доводке.
- **П.10 — первый шаг миграции `light`/`dark`/`nord`.** 28 самодельных токенов «все 4 разные» свести к существующим ролям sepia — отдельная сессия.
