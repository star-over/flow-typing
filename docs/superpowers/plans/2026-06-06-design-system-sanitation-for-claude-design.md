# Design System Sanitation for Claude Design Ingest

**Date:** 2026-06-06
**Status:** Plan (not yet executed)
**Owner:** @FFire
**Context:** Подготовка существующей дизайн-системы FlowTyping к ingest в Claude Design (Anthropic Labs, launched 2026-04-17). Дизайн ранее был сделан для проверки концепта — теперь нужна санация до уровня, на котором AI-инструмент сможет читать систему и работать с ней последовательно.

## Umbrella

**Цель**: дать Claude Design **читаемую** дизайн-систему за ≤ 1 рабочий день, не трогая существующие компоненты и не запуская token pipeline. Принцип — **только добавлять и группировать**, ничего не переименовывать (иначе ломаются потребители). Работаем в `src/app.css` и в одном новом `src/design-system.json` рядом с ним; всё.

Что Claude Design ищет при первичной настройке системы:
1. Структурированный набор токенов с явной иерархией
2. Доменную семантику в комментариях
3. Что-то одно, во что можно ткнуть и сказать «вот наша система»

Сейчас в проекте есть только (1) частично. Пять фаз ниже закрывают всё это.

## Текущее состояние (baseline)

- `src/app.css` — палитра в OKLCH с маппингом на Tailwind-имена в комментариях, масштабы `--spacing`/`--radius` производные от базовой единицы. Семантика расщеплена (surfaces / text / outcomes / finger / keycap-group / keycap-role / press-result), **но слои `semantic` и `domain` перемешаны без меток**.
- Storybook покрывает UI-компоненты, но не токены.
- Документации дизайн-системы как отдельной сущности нет.

### Что отсутствует и блокирует ingest

1. Нет явных меток слоёв (primitives / semantic / domain) — для AI они выглядят однородно.
2. Нет typography-шкалы (`--font-size-*`, `--line-height-*`, `--font-weight-*`).
3. Нет shadow/elevation токенов.
4. Нет motion-токенов (`--motion-duration-*`, `--motion-ease-*`) — критично для продукта про «визуализацию движения».
5. Доменные токены (`--color-finger-*`, `--color-keycap-*`) описаны через происхождение цвета, а не через **смысл** — AI не знает, что `--color-finger-1` = большой палец и используется только в TARGET state.
6. Нет single source of truth — манифеста системы как одного файла.
7. Нет dark theme.

## Roadmap — 5 фаз, ~6 часов суммарно

Приоритеты: **MUST** делаем точно, **SHOULD** делаем если есть время, **NICE** — потом.

---

### Phase 0 · Послойный порядок в `app.css` — **MUST** (~45 мин)

**Что делаем**: переставляем существующие секции в три явно подписанных слоя, не меняя значений и не переименовывая токены.

```css
/* === LAYER 1 · PRIMITIVES — raw values, no semantics === */
/* --color-neutral-0..950, --color-blue-500, --font-sans, --spacing, --radius */

/* === LAYER 2 · SEMANTIC — role tokens, refer to primitives === */
/* --color-bg, --color-surface, --color-text-primary, --color-success, --color-error */

/* === LAYER 3 · DOMAIN — FlowTyping-specific === */
/* --color-finger-*, --color-keycap-*, --color-path-highlight, --color-cursor-* */
```

**Acceptance**:
- `make check` зелёный, визуально ничего не изменилось.
- В `app.css` три явных секции с большими разделителями.
- Любой токен можно отнести однозначно к одному из трёх слоёв.

**Почему это работает**: сейчас слои 2 и 3 перемешаны. После санации AI и человек сразу видят: «домен — пятый палец, не трогать; surface — можно переосмыслить».

**Cost saving трюк**: значения сейчас уже разумные, можно почти ничего не менять — главное метки и порядок.

---

### Phase 1 · Заполнить недостающие шкалы — **MUST** (~1.5 ч)

**Что делаем**: добавляем три раздела в LAYER 1 (primitives):

```css
/* Typography — производна от --font-size-base, как у вас уже spacing/radius */
--font-size-base: 1rem;
--font-size-xs: …; --font-size-sm: …; --font-size-md: …;
--font-size-lg: …; --font-size-xl: …; --font-size-2xl: …;
--line-height-tight: 1.2; --line-height-normal: 1.5; --line-height-loose: 1.75;
--font-weight-regular: 400; --font-weight-medium: 500;
--font-weight-semibold: 600; --font-weight-bold: 700;

/* Elevation — 4 уровня */
--shadow-sm: …; --shadow-md: …; --shadow-lg: …; --shadow-xl: …;

/* Motion — критично для продукта про «визуализацию движения» */
--motion-duration-fast: 120ms;
--motion-duration-base: 200ms;
--motion-duration-slow: 400ms;
--motion-ease-standard: cubic-bezier(0.2, 0, 0, 1);
--motion-ease-decelerate: cubic-bezier(0, 0, 0, 1);
--motion-ease-accelerate: cubic-bezier(0.3, 0, 1, 1);
```

**Acceptance**:
- Все три раздела присутствуют в `app.css`.
- Значения **не подгоняются** под текущие inline-числа в компонентах (пока). Это база, к которой компоненты будут постепенно мигрировать.

**Почему это работает**: сейчас каждый компонент жонглирует размерами шрифта руками — Claude Design при генерации придумает свою шкалу и она разойдётся с проектной. Без motion-токенов нет языка обсуждать анимацию пальца — а это ядро продукта.

**Cost saving трюк**: берём готовые ladders (Tailwind v4 type scale, Material 3 motion) — не сочиняем.

---

### Phase 2 · JSDoc-семантика на доменные токены — **MUST** (~45 мин)

**Что делаем**: перед каждой доменной секцией добавляем 2-3 строки комментария, написанные **для AI и человека одновременно**.

```css
/* --- Finger palette (per FingerId numeric index) ---
 * Hand has 5 fingers: 1=thumb, 2=index, 3=middle, 4=ring, 5=pinky.
 * Used ONLY for TARGET state — see docs/03-ui-viewmodel-contract.md.
 * The hue cycles by anatomy: warm (thumb/index) → cool (ring/pinky). */
--color-finger-1: …
```

Покрыть нужно как минимум:
- `--color-finger-*` — что такое FingerId, когда применяется
- `--color-keycap-group-*` — почему пять цветовых групп, как они сопоставлены с пальцами
- `--color-keycap-role-target-*` — что такое navigation role, когда активируется
- `--color-keycap-correct/error-*` — жизненный цикл press result
- `--color-path-highlight`, `--color-target-marker`, `--color-cursor-*` — что такое «navigation visualization»
- `--color-symbol-*` — статусы TypingStream

**Acceptance**:
- Каждая доменная секция в LAYER 3 имеет JSDoc-блок до 5 строк, объясняющий **зачем** этот токен, **когда** используется и **что нельзя** с ним делать.
- Не описываем «откуда взялся hex» — это есть в существующих inline-комментариях справа.

**Почему это работает**: сейчас комментарии описывают «откуда взялся hex» — это для рефакторинга палитры. Нужна противоположная информация — **зачем этот токен**. Claude Design тогда не подменит `--color-finger-1` градиентом «потому что красиво» в новом экране Settings.

---

### Phase 3 · Single-file manifest `src/design-system.json` — **SHOULD** (~1 ч)

**Что делаем**: один JSON-файл в упрощённом формате (вдохновлён W3C Design Tokens, но без полной спецификации):

```jsonc
{
  "$layers": ["primitives", "semantic", "domain"],
  "primitives": {
    "color": {
      "neutral-0": { "value": "oklch(1 0 0)" },
      "yellow-400": { "value": "oklch(0.86 0.17 92)" }
    }
  },
  "semantic": {
    "color": {
      "bg": { "value": "{primitives.color.neutral-0}" }
    }
  },
  "domain": {
    "finger": {
      "1": {
        "value": "{primitives.color.yellow-400}",
        "description": "thumb / большой палец, TARGET state only"
      }
    }
  }
}
```

**Acceptance**:
- Файл существует в `src/design-system.json`.
- Покрывает все токены из `app.css`.
- Иерархия слоёв соответствует Phase 0.
- Доменные токены имеют поле `description` (короткий перевод из JSDoc Phase 2).
- **НЕ** подключается к сборке — это декларативный snapshot, читаемый AI и людьми.

**Почему это работает**: это не подменяет `app.css` (его правит время выполнения), а **дублирует** его как декларацию. Claude Design прочитает его за секунды, без эвристик. Не вводим Style Dictionary / token pipeline — это и есть «дёшево»: руками синхронизируем дважды в полугодие, если меняем токены.

**Альтернатива дешевле**: оставить только `app.css` с очень подробным header-комментарием. Хуже для AI, лучше для DX. Решение по вкусу при выполнении.

---

### Phase 4 · Лёгкая dark-theme заготовка — **NICE** (~1.5 ч)

**Что делаем**: второй блок `:root[data-theme="dark"] { … }` с инвертированными surfaces/text и подстроенными chroma для домена (finger/keycap чуть менее насыщенные, не белёсые). Переключатель в Settings можно не делать сразу — достаточно того, что тема существует и переключается через DevTools.

**Acceptance**:
- В `app.css` есть `:root[data-theme="dark"]` блок.
- Все компоненты используют **только** токены (если где-то остался inline-цвет — исправляем).
- Переключение через `<html data-theme="dark">` в DevTools даёт связную тёмную тему.
- Не обязательно красиво — обязательно работоспособно.

**Почему это работает**:
- Для Claude Design — при генерации новых экранов он автоматически выдаст обе темы.
- Для пользователя — 30-минутные сессии печати в светлой теме это глаза.

**Не делаем в этой фазе**: переключатель темы в UI, persistence через `preferences.ts`, авто-определение `prefers-color-scheme`. Это отдельная задача.

---

### Phase 5 · Один `design-system.html` для web capture — **NICE** (~45 мин)

**Что делаем**: статичная страница, показывающая **все** токены и **все** компоненты во **всех** состояниях. Не Storybook — голый HTML с импортом `app.css`. Открывается в браузере, публикуется туннелем / разворачивается в Pages, скармливается Claude Design как «вот наш style guide, прочитай и встрой в свой контекст».

**Acceptance**:
- Файл `src/design-system.html` (или `static/design-system.html`) на ~200 строк.
- Витрина: палитра primitives, palette semantic, palette domain, typography ladder, spacing ladder, shadow ladder, motion sample (анимированный квадрат), все базовые компоненты (KeyCap во всех state, Finger в TARGET/IDLE/ERROR, FlowLine с пятью статусами символов).
- Открывается напрямую через `file://` или `localhost:5173/design-system.html`.

**Почему это работает**: у вас уже есть Storybook со stories — он покрывает компоненты, но не токены и не «всё на одном экране». Эта страница — буквально один файл, генерируется один раз, потом Claude Design её прочитает за один заход.

## Что специально НЕ делаем

| Не делаем | Почему |
|---|---|
| Style Dictionary / token transformers / W3C tokens спецификация целиком | Дорого для одного человека и одного приложения. |
| Wholesale refactor scoped CSS в компонентах | Слом ради «чистоты». Компоненты работают через CSS-vars — больше ничего не нужно. |
| Переименование существующих токенов | Имя — это API. Переименование = поиск/замена по всему проекту, регрессионные риски. |
| Полноценная dark theme + переключатель + persistence | Чистая UX-работа, не нужна для ingest Claude Design. После фаз 0-3 он сам предложит варианты. |
| Отдельный design-system.md | Семантика уже в JSDoc токенов. Документация в двух местах = устаревает в обоих. |

## Sequencing

- **Полдня (~3 ч)**: Phase **0 → 1 → 2**. Этого достаточно для приличного ingest.
- **День (~6 ч)**: **0 → 1 → 2 → 3 → 5**. Полная санация под Claude Design.
- **Час (минимум)**: только Phase **2** — JSDoc-семантика. AI получает **смысл** токенов, что важнее структуры.

Phase 4 (dark theme) делается параллельно с любым из этих наборов или отдельно — она не блокирует ingest.

## Acceptance для всей санации

- `make check-all` зелёный.
- Визуальная регрессия равна нулю (Storybook stories выглядят так же).
- `app.css` имеет три явных слоя с разделителями.
- Все доменные токены имеют JSDoc-комментарий с семантикой.
- `src/design-system.json` существует и валиден.
- (опционально) `design-system.html` открывается и показывает витрину.

## Выход в Claude Design после санации

1. `make build` → static SPA в `build/`.
2. Разворачиваем на Cloudflare Pages / Vercel / Netlify (5 минут).
3. В Claude Design первичная настройка: указываем production URL + git-репозиторий (или zip-снимок).
4. Claude Design читает `app.css` (видит слои), `design-system.json` (видит манифест), `design-system.html` (видит витрину через web capture).
5. Первый prompt — берётся **полностью** из секции «Brief для Claude Design» ниже, дальше дописываем конкретную задачу (например: «сделай редизайн экрана Settings, сохраняя tokens и не трогая ядро hands-scene»).

## Brief для Claude Design (продуктовый контекст)

Этот раздел — **готовый текст**, который копируется в первый prompt при работе с Claude Design (см. шаг 5 предыдущей секции). Цель — встроить в AI представление о том, **что за продукт** ему предстоит дорабатывать. Любая последующая задача оценивается через призму этой философии, а не «сделать красиво вообще».

Brief нужно держать **синхронным с `docs/01-philosophy-and-vision.md`** — это его сжатая выжимка для AI, не отдельный источник правды.

### Elevator pitch (одно предложение)

> FlowTyping — клиентский SPA-тренажёр слепой печати, в котором главный элемент интерфейса рисует **путь, который должен проделать палец** от домашней позиции до целевой клавиши и обратно. Это **не** виртуальная клавиатура с подсветкой — это **визуализация движения**.

### Философия — три принципа, которые должны просвечивать через дизайн

1. **Рефлексивно-адаптивное обучение, не зубрёжка.** Цель — мышечная память и проприоцепция, не запоминание. Дизайн должен **уводить взгляд от клавиатуры** к руке. Любые подсказки, которые провоцируют пользователя «подглядеть на keycap», работают против продукта.
2. **Точность важнее скорости.** Алгоритм не повышает сложность, пока не стабилизирована точность. UI должен ощущаться как **внимательный коуч**, а не как arcade-игра. Спокойствие, не возбуждение. Скорость — естественное следствие точности, не самоцель.
3. **Ритм и возврат в исходную позицию — священны.** Каждый палец возвращается на «домашнюю» клавишу (ASDF / JKL; / ФЫВА / ОЛДЖ) после нажатия. Это краеугольный камень слепого метода. Должно визуально пульсировать через интерфейс: motion, easing, дыхание состояний.

### Tone & mood

- **Близко к**: Linear, Things, Arc Browser, Notion для разработчиков. Технологичный минимализм с тёплыми акцентами.
- **Далеко от**: Duolingo, Mavis Beacon, любые тренажёры в эстетике «казино / военная подача / 80-е». Никаких confetti, badges, level-up с фанфарами, streak-эмодзи.
- **Палитра**: уже выбран тёплый янтарный target-маркер + холодный синий path-highlight. Это намёк на «живая рука + механический контракт клавиатуры». Сохраняем дуальность.
- **Типографика**: Geist (sans) + Geist Mono. Mono — для FlowLine и KeyCap label, потому что monospace это **функциональное требование**, не стилистика. Не подменять proportional-шрифтом.
- **Motion — первый класс**. Это **продукт про движение**. UI должен дышать в темпе ритмичной печати, а не дёргаться скачками. Любое state-transition с резким snap'ом — провал контракта.

### Целевая аудитория

Старшие школьники и студенты, молодые специалисты и офисные работники, начинающие программисты. Объединяет — желание перейти от двупальцевой печати к слепому методу, осознанная мотивация, готовность вкладывать 20-40 минут в день. **Не** игроки в casual mobile games. **Не** дети, нуждающиеся в геймификации.

### Что **нельзя** трогать (semantic locks)

| Зона | Почему |
|---|---|
| SVG-геометрия руки (`src/data/finger-paths.ts`, `HandsScene.svelte`, `Finger.svelte`) | Привязана к `FINGER_CENTER_POINTS` — на этих координатах стоят кластеры клавиш. Сломаешь геометрию — сломаешь всю hands-scene. |
| Контракт ViewModel (`docs/03-ui-viewmodel-contract.md`) | Какие клавиши, в каком состоянии, когда горят — это бизнес-логика, не дизайн. Правило «Полного Кластера» не пересматривается. |
| Domain-токены (`--color-finger-1..5`, `--color-keycap-group-*`, `--color-keycap-role-target-*`) | Пятипальцевая семантика отражает анатомию руки. Не подменять «одноцветным» решением «потому что чище». |
| Mono-шрифт для FlowLine и KeyCap label | Требование, не стиль. |
| FingerId-семантика (1=thumb, 2=index, 3=middle, 4=ring, 5=pinky) | Доменная модель. Цветовое назначение по индексу — её отражение. |

### Что **можно и нужно** доработать

- **Окружение руки**: фон сцены, освещение, мягкие тени, материалы, композиция вертикальной и горизонтальной осей.
- **Периферийные экраны**: Header, FooterActions, MainContent layout, Menu, Settings, Stats, TrainingComplete. Эти экраны были сделаны для проверки концепта — там простор для редизайна.
- **Состояния и переходы**: motion от IDLE → TARGET, от TARGET → CORRECT/ERROR. Сейчас переходы есть, но не выверены по тайминги и easing'у.
- **Типографическая иерархия** в окружении (не во FlowLine и не в KeyCap label).
- **Семантические токены** (surfaces, text, outcomes) — можно переосмыслить, сохраняя имена.
- **Dark theme** — отсутствует, нужна.
- **Микро-анимации поверх SVG-руки** — opacity/transform на active finger, лёгкая «жизнь» рукой в idle. Не геометрия SVG-путей, а CSS-слой поверх.

```text
- Правило «Полного Кластера» из docs/03-ui-viewmodel-contract.md

Свободно дорабатывать:
- Окружение руки (фон, тени, материалы, композиция)
- Периферийные экраны (Settings, Stats, Menu, TrainingComplete, Header)
- Motion: длительности, easing, переходы состояний
- Типографическую иерархию в окружении
- Семантические токены (surfaces, text, outcomes)

Принципы, которые должны просвечивать через результат:
1. Точность важнее скорости — никаких подгоняющих элементов
2. Ритм и возврат в исходную позицию — визуально пульсирует через motion
3. Уводить взгляд к руке, не к клавиатуре

Задача: <твоя конкретная задача>

Предложи 2-3 варианта в разных направлениях, сохраняя tokens и
domain-семантику. Объясни, в чём принципиальная разница между вариантами.
```

### Как использовать brief дальше

- **Первая сессия**: копируешь весь Prompt template целиком, дописываешь конкретную задачу — Claude Design получает полный контекст.
- **Последующие сессии в том же проекте**: контекст уже в памяти Claude Design (после первичной настройки проекта). Достаточно ссылаться: «помни брифинг проекта, сделай X». Если AI «забыл» (после долгого простоя или смены модели) — прикладываешь brief снова.
- **Когда добавляешь новый экран в Figma-плагин Claude Design**: разделы «Не трогать» и «Свободно дорабатывать» — главные. Они снимают 80% «гениальных» предложений типа «давайте уберём цвет с пальцев, будет минимализм».
