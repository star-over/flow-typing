# Dependency Upgrade Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bump every dependency reported by `ncu` to its latest version (including TypeScript 6, Vite 8 + vite-plugin-svelte 7, ESLint 10), one commit per phase, with verification gates between phases.

**Architecture:** Five sequential phases ordered by risk (low → high). Each phase modifies `package.json` + `package-lock.json` and at most one config file, runs the project's existing verification commands, then commits. Phases are largely independent — if one phase fails, revert it and continue.

**Tech Stack:** SvelteKit 2.61 (SPA, static adapter), Svelte 5, npm, vitest, ESLint flat config, TypeScript strict, Storybook 10. Local Node v24.9.0.

---

## Spec reference

See `docs/superpowers/specs/2026-06-02-dependency-upgrade-design.md` for the goals, non-goals, and risk research that drove this plan.

**Discovery during planning:** `eslint-plugin-vitest@^0.5.4` is declared in `devDependencies` but is **not imported anywhere** in `eslint.config.mjs` — it is a dead dep. The spec called for migrating it to `@vitest/eslint-plugin`; this plan instead removes it (no replacement), which is simpler and equally unblocks ESLint 10. If linting test files for Vitest patterns is desired later, that's a separate change.

## File structure

Files modified by this plan (no new source files):

| File | Phases that touch it | Reason |
|---|---|---|
| `package.json` | 1–5 | Every phase bumps deps. |
| `package-lock.json` | 1–5 | Always travels with `package.json`. |
| `eslint.config.mjs` | 3 (only if a new recommended rule fires broadly) | Add an `overrides` block to disable a specific rule with explanation. |
| `tsconfig.json` | 4 (only as escape hatch) | Add `"ignoreDeprecations": "6.0"` if TS 6 surfaces deprecations from generated `.svelte-kit/tsconfig.json`. |

No changes expected in `vite.config.ts`, `svelte.config.js`, source files, or test files. If a verification step reveals required source edits beyond the scope above, stop and report — do not invent fixes.

## Task 4: Phase 3 — ESLint 10 + remove dead `eslint-plugin-vitest`

**Files:**
- Modify: `package.json` (remove one line, bump two).
- Modify: `package-lock.json` (regenerated).
- Modify: `eslint.config.mjs` (only if a new recommended rule fires — see Step 5).

**Why we remove `eslint-plugin-vitest` instead of migrating:** the plugin is declared in `devDependencies` but never imported in `eslint.config.mjs`. Removing it eliminates the only peer-dep blocker for ESLint 10. If Vitest linting is desired in the future, add `@vitest/eslint-plugin@^1.6.19` then — that's a separate change.

- [ ] **Step 1: Edit `package.json`**

`devDependencies` edits:
- Remove the line entirely: `"eslint-plugin-vitest": "^0.5.4",`
- Replace: `"@eslint/js": "^9.39.4"` → `"@eslint/js": "^10.0.1"`
- Replace: `"eslint": "^9.39.2"` → `"eslint": "^10.4.1"`

- [ ] **Step 2: Install**

Run: `npm install`
Expected: exits 0. Peer warnings about `eslint-plugin-vitest` should disappear. If `typescript-eslint` or `eslint-plugin-svelte` print peer warnings about ESLint 10, stop and report — both have v10 in their peer range per the spec's research; a warning would indicate the published `package.json` has changed since.

- [ ] **Step 3: Run lint to surface ESLint 10's new recommended rules**

Run: `make lint`
Expected: either exits 0, or fails with diagnostics from one or more of: `no-unassigned-vars`, `no-useless-assignment`, `preserve-caught-error`.

- [ ] **Step 4: If lint passes, skip to Step 6. If lint fails, classify the findings**

For each new rule that fires, decide:
- **Mechanical fix in ≤ 3 lines per file:** apply the fix. Examples — declaring a variable with `let` and never assigning (`no-unassigned-vars`), unused intermediate `const x = ...; x = ...` (`no-useless-assignment`), `catch (e)` where the caught error is dropped instead of re-thrown or preserved (`preserve-caught-error`).
- **Fires broadly (10+ sites) or fix isn't mechanical:** disable in `eslint.config.mjs` — see Step 5.

- [ ] **Step 5: If a rule must be disabled, add an override block to `eslint.config.mjs`**

Open `eslint.config.mjs`. Add a new config block before the final `ignores` block. Example for disabling `preserve-caught-error`:

```javascript
  {
    rules: {
      // Disabled because [REPLACE WITH the count + brief reason from the lint output].
      // Re-enable in a dedicated cleanup pass.
      'preserve-caught-error': 'off',
    },
  },
```

Do NOT leave the `[REPLACE WITH ...]` placeholder — fill it in with the actual rationale from the lint output (e.g., "fires at 14 call sites across `src/machines/`, fixes require non-trivial control-flow changes"). If you cannot articulate a real reason, the rule probably should be fixed instead of disabled.

- [ ] **Step 6: Re-run lint to confirm green**

Run: `make lint`
Expected: exits 0.

- [ ] **Step 7: Run the rest of the suite**

Run: `make check && make test`
Expected: both exit 0.

- [ ] **Step 8: Commit**

If only `package.json` and `package-lock.json` changed:

```bash
git add package.json package-lock.json
git commit -m "$(cat <<'EOF'
chore: bump eslint to v10 and drop unused eslint-plugin-vitest

ESLint 10 requires Node 20.19+/22.13+/24+ (have v24.9). The old
.eslintrc API was removed but the project already uses flat config
(eslint.config.mjs). eslint-plugin-vitest@0.5.4 was declared in
devDependencies but never imported — removing it eliminates the only
peer-dep blocker against ESLint 10.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
EOF

```

If `eslint.config.mjs` also changed in Step 5:

## Cross-phase notes

- **Pre-flight per phase:** `git status --short` must be empty and `make check-all` must be green before starting any phase after the first. If a phase commits but verification re-run is red, that's a bug in the phase — revert and investigate.
- **One commit per phase**, always with `package.json` and `package-lock.json` together. Other files (`eslint.config.mjs`, `tsconfig.json`) only appear if a specific step adds them, and only with the explicit rationale baked into the commit message.
- **Failure handling:** if a phase fails irreparably with the steps above, run `git restore package.json package-lock.json` (plus any other changed file) and skip to the next phase. The five phases are independent — order is risk-based, not dependency-based.
- **Out of scope:** introducing `@vitest/eslint-plugin` (no current Vitest linting), upgrading SvelteKit (not needed for any phase), refactoring application code beyond what an upgrade strictly requires, any change to `vite.config.ts`, `svelte.config.js`, or source files unless specifically called out in a step.
