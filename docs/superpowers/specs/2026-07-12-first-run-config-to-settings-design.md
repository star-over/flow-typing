# Дизайн: убрать конфиг-стену с пути первого нажатия

> Дата: 2026-07-12. Задача плана: `docs/plans/2026-07-05-mvp-launch.md` §2 **P0-11 (a)**.
> Статус: согласовано с владельцем (Вариант A).

## Проблема

После входа `App.svelte` нормализует тренажёр в состояние `menu`, и `MainContent`
рендерит `MenuScreen` — **форму из пяти `Select`-списков** (язык текста, символьная
раскладка, пальцевая раскладка, тип курсора, канал ритма) + длительность + ещё один
клик «Начать тренировку». CTA лендинга обещает тренировку, а отдаёт конфиг-форму с
доменным жаргоном, на который новичок не может осмысленно ответить. Это крупнейшая из
трёх преград между «Начать тренировку» и первым нажатием (P0-11), и §9 go/no-go
обязан отличать немотивированный отвал на ней от «движение не помогает».

Дефолты уже рабочие (`en`/`qwerty`/`asdf` + курсор + ритм + 300 c; у `en+qwerty`
есть корпус) — данные не мешают пустить первый забег сразу. Все шесть элементов управления
default-able: ни один не обязателен новичку перед первым нажатием.

## Решение (Вариант A)

Тихий стартовый экран + конфиг переезжает на `/settings`.

1. **`MenuScreen` становится тихим стартовым экраном** — одиночная CTA «Начать
   тренировку», без единого выпадающего списка.
2. **Шесть тренировочных элементов управления переезжают на `/settings`** новой секцией
   «Тренировка» (рядом с именем/темой/danger-зоной).
3. **FSM не меняется** — состояние `menu` остаётся, меняется только его UI. Это
   «глупый UI» + переработка структуры `/settings`, **новый ADR не требуется**.

Отвергнутые альтернативы:
- **Progressive disclosure на `/train`** (элементы управления свёрнуты под «Настроить ▸»):
  форма всё равно живёт на входе в тренировку и дублирует естественный дом `/settings`.
- **Автостарт на первом забеге** (first-run → сразу в сессию): трогает автостарт
  машины + требует определения первого запуска → новый ADR; внезапный старт без «ready»-такта
  ощущается резко. Одиночная спокойная CTA решает жалобу без правки машины.

Жалоба — про **конфиг-форму с жаргоном**, не про существование кнопки старта.
Спокойная CTA, повторяющая лендинг, — подтверждающий такт, а не преграда. Один тихий
экран обслуживает и новичка, и хаб-возврат между сессиями — определение первого запуска не нужно.

## Изменения по компонентам

### `src/components/train/MenuScreen.svelte` — свести к CTA

- Удалить пять `Select`-полей, `SessionDurationSelector`, все производные списки
  опций (`textLanguages`, `layoutOptions`, `fingerLayouts`, `cursorTypes`,
  `rhythmChannelOptions`, `sessionDurationOptions`) и связанные импорты
  (`Select`, `SessionDurationSelector`, `getCompatibleSymbolLayoutsForTextLanguage`,
  лишние типы).
- Остаётся `.start-btn` с CTA (`--landing-cta-*` токены сохраняются) + `onStart`.
- Чистый экран: только CTA, без подзаголовка. Ссылка на настройки уже есть в шапке —
  второй линк под кнопкой не добавляем.
- Tech-debt-заметка «нет своего `MenuScreen.contract.ts`» остаётся верной.

### `src/components/ui/SettingsPage.svelte` — новая секция «Тренировка»

Добавить секцию с шестью элементами управления, порядок на странице:
**Имя → Тема → Тренировка → «Назад» → danger-зона** (danger остаётся последней).

| Элемент управления | Компонент | i18n-ключ (существует) |
|---|---|---|
| Язык текста | `Select` | `settings.text_language_label` |
| Символьная раскладка | `Select` (+ фильтр `getCompatibleSymbolLayoutsForTextLanguage`) | `settings.symbol_layout_label` |
| Пальцевая раскладка | `Select` | `settings.finger_layout_label` |
| Тип курсора | `Select` | `settings.cursor_type_label` |
| Канал ритма | `Select` | `settings.rhythm_channel_label` |
| Длительность | `SessionDurationSelector` (переезжает `train/` → `ui/`, см. ниже) | `settings.session_duration_label` |

- Производные списки опций и логика совместимости «язык текста → доступные раскладки»
  переезжают из `MenuScreen` **как есть** — поведение сохраняется точь-в-точь. Уточнение
  (исправлено после аудита): при смене «Язык текста» на несовместимый `normalizeSettings`
  (`src/lib/settings.ts`, гоняется на каждом `update`) приводит символьную раскладку к
  дефолту языка — существующее поведение, оно же делает пустой корпус `ru`+`qwerty`
  недостижимым из UI. Переносим без изменений.
- Поля используют существующие `.field`-идиому и `--settings-page-*` токены;
  `SessionDurationSelector` и `Select` — свои существующие токены. **Новых
  контракт-токенов нет.**
- **`SessionDurationSelector` переезжает `src/components/train/` → `src/components/ui/`**
  (правка после аудита). Это generic-примитив без train-зависимостей, а после переноса
  элементов управления его единственным потребителем становится `ui/SettingsPage`. Оставить его в
  `train/` завело бы первое в проекте ребро `ui → train` (сейчас есть только `train → ui`);
  переселение убирает инверсию и кладёт компонент рядом с потребителем.
- Подзаголовок секции — новый i18n-ключ `settings.training_group` в оба словаря.
- **Гость видит секцию** (без auth-барьера): это preferences, как тема (тоже видна
  гостю). Гость может настроиться до входа; при первом входе cloud пуст → локальные
  настройки пушатся в cloud (`decideSyncOnLogin`) и переживают вход. **Оговорка
  (после аудита):** секция расширяет гость-редактируемую поверхность с одной темы до
  шести настроек; при входе в **существующий** аккаунт с облачными настройками
  стратегия «cloud wins» (`decideSyncOnLogin` при непустом cloud → pull) **тихо
  перезапишет** локальные правки гостя облаком. Это существующее by-design поведение
  Phase 5 (не баг), просто теперь оно касается большего числа полей.

### `src/components/app/MainContent.svelte` — без содержательных правок

`onStart` для `MenuScreen` продолжает слать `START_TRAINING` с параметрами из
`$settings` (единственный источник истины; так уже работает Enter-путь в `App.svelte`).
Контракт события не меняется.

## Что НЕ трогаем

- **FSM / машину** (`app.machine.ts`) — `menu`/`training`/автостарт без изменений.
- **ADR** — новый не нужен (нет смены зафиксированного решения о жизненном цикле).
- **Контракт-токены / темы** — переезжающие элементы управления уже используют
  `--settings-page-*` / `--select-*` / `--landing-cta-*`; контракт-тест зелёный без правок.
- **Sync (Phase 5)** — тот же `$settings`-стор, тот же `upsertMine`; ноль backend-работы.
- **Дефолт длительности 300 c** — отдельный пункт (P0-11 d), за владельцем; молча не менять.

## i18n

Единственная новая строка — подзаголовок секции `settings.training_group`
(например, «Тренировка» / «Training») в `dictionaries/{en,ru}.json`. Все ключи полей
и опций уже существуют (ADR 0022).

## Проверка

- Storybook: у `MenuScreen` истории нет (не заводим); история `SettingsPage`
  (`Guest` / `Authenticated`) правок не требует — секция «Тренировка» не за auth-барьером,
  отобразится в существующих историях автоматически после добавления ключа.
- FSM не меняется → новых тестов машины не требуется; чистая логика совместимости
  раскладок уже покрыта. Паритет ключей `en/ru` тестом не покрыт (`ru.json` не читается
  ни одним тестом; `Dictionary = typeof en`) → держать оба edit'а `training_group`
  (ADR 0022 требует обе языковые версии).
- `make check-all` зелёным перед коммитом (включая `make spell`).
- **Live-verify** упирается в известное препятствие dev-auth (лимит записи dev Convex,
  `signUp` → `InvalidAccountId`). Обходы: секция «Тренировка» на `/settings` доступна
  и гостю → переезд проверяется на гостевом `/settings` без входа; тихий экран
  `/train` — за гостевым барьером → Storybook + при возможности вход в существующий
  dev-аккаунт. Каким путём проверено — отметить в отчёте.
