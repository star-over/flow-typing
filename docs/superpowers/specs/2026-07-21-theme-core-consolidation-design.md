# Дизайн: сведение ядра тем `light` / `dark` / `nord`

Дата: 2026-07-21 · Ветка: `refactor/theme-core-consolidation`

Последняя незакрытая грань CSS-аудита (#6). Доводит ADR 0029 до конца.

## Контекст

Тема — два слоя (ADR 0029): **ядро** (приватная палитра, только абсолютные
`oklch`, голые имена) и **роли** (`--color-*`, фиксированный словарь из 73 имён,
значение выводится из ядра или другой роли). Замысел: смысл живёт в ролях,
ядро — маленькая палитра, названная по содержимому. Тогда тема перекрашивается
правкой палитры, а не двух сотен строк.

`sepia` этому следует, три остальные темы — нет:

| тема | ядро | роли | выведено расчётом | плоских `var()` 1:1 |
| --- | --- | --- | --- | --- |
| `sepia` | **11** | 73 | **53** | 20 |
| `light` | 35 | 73 | 28 | 45 |
| `dark` | 40 | 73 | 29 | 44 |
| `nord` | 35 | 73 | 26 | 47 |

У `light` / `dark` / `nord` смысл осел в ядре, а роли выродились в
переименование. Шапки этих трёх файлов признают это прямым текстом: «палитра
перенесена из прежнего … 1:1 (механика, без пересмотра значений) — отсюда её
размер; сокращение — отдельная задача».

Три следствия, которые видно в коде:

- **Ядро содержит лестницы.** В `nord` `--polar-night-1..4` — это `0.32 / 0.38 /
  0.44 / 0.50` при общих `c 0.02 h 254`: одна база и шаг `+0.06`, записанные
  четырьмя независимыми абсолютами. То же у `light` с `--paper` (`1 / 0.98 /
  0.95 / 0.9 / 0.85`) и `--ink` (`0.14 / 0.44 / 0.56`).
- **Ядро содержит токены, названные по роли.** `--primary-hover` во всех трёх;
  в `nord` вдобавок `--cursor-background`, `--cursor-foreground`,
  `--keycap-label`, `--keycap-marker`, `--symbol-pending`. У каждого ровно один
  потребитель — одноимённая роль. Это пустая пересылка, и она ломает саму цель:
  токен палитры не может называться по роли, иначе перекрашивать нечего.
- **Имена ядра расходятся между темами.** Источник оттенка пальца зовётся
  `--route-N` в `sepia` и `--finger-N` в трёх остальных.

Осознанные различия тем (`--color-finger-error`, `--color-cursor-background`
определены по-разному) — это пер-темная доводка по ADR 0028 / 0031, она вне
работы и не трогается.

## Решение

Переписать запись тем `light` / `dark` / `nord` так, чтобы смысл переехал из
ядра в роли. Словарь ролей (73 имени), семантика контракт-теста и компоненты
(`src/components/**`) не меняются вовсе — работа целиком внутри четырёх
CSS-файлов.

### Критерий честного вывода

Роль записывается как `oklch(from var(--ядро) …)` только если выполнены **оба**
условия:

1. **Фактическое наследование.** Минимум один из трёх компонентов записан как
   `l` / `c` / `h` либо `calc(l …)` / `calc(c …)` / `calc(h …)` — то есть при
   перекраске ядра он реально поедет.
2. **Смысловая связь.** Источник — тот токен, чьей ступенью или отзвуком роль
   является по смыслу: ступень лестницы поверхностей от базы фона, отзвук
   оттенка пальца, `hover` от своей базы. Не «ближайший подходящий токен».

Запись всех трёх компонентов литералами (`oklch(from var(--green) 0.45 0.11
151)`) **запрещена**: она даёт то же число, но лжёт про связь — это абсолют в
чужой одежде, и он хуже честного ядрового токена.

Не выполнено — значение остаётся в ядре, но обязано носить имя по содержимому.
Ядро из 22 честных токенов лучше ядра из 11 с выдуманными связями. Ожидаемый
итог 18–22 токена на тему — следствие критерия, а не цель.

Отсюда же лечатся токены, названные по роли: `nord --cursor-background:
oklch(0.77 0.16 70)` ни из чего в `nord` честно не выводится, поэтому он
остаётся абсолютом и **переименовывается** в `--amber`. Дефект снимается
переименованием, а не натянутым выводом.

### Сведение близких значений

Где два значения различаются незначимо, они сводятся к одному. Правило:

- **Порог.** ΔE ≤ 0.05, где ΔE — евклидово расстояние в OKLab
  (`a = C·cos H`, `b = C·sin H`, `ΔE = √(ΔL² + Δa² + Δb²)`).
- **Альфа.** Значения с разной альфой не сводятся ни при каком ΔE — альфа в ΔE
  не входит.
- **Контраст — жёсткий блокиратор.** Для каждого предполагаемого сведения
  контраст изменяемой роли считается до и после ко всем ролям-подложкам темы.
  Список подложек фиксирован: `--color-background`, `--color-surface`,
  `--color-surface-raised`, `--color-surface-hover`, `--color-surface-accent`,
  `--color-primary-background`, `--color-error`, `--color-success`,
  `--color-target-1..5`, `--color-cursor-background`.
  Если хоть одно отношение пересекает границу WCAG вниз (4.5 для текста,
  3.0 для крупного символа и не-текста) — сведение отклоняется независимо от
  ΔE. Правило намеренно консервативно: оно не требует знать, на каком фоне роль
  лежит на самом деле, и ошибается в безопасную сторону. Числа контраста в
  комментариях `sepia` и `light` — обоснование значений, а не украшение; ронять
  их молча нельзя.

Каждое принятое сведение получает комментарий в CSS с числами (ΔE и затронутый
контраст) — в том же стиле, в каком обоснования уже записаны в `sepia`.

### Вычислитель ролей и снапшот

Гейт всей работы — расчёт, а не осмотр. `src/themes/resolve-color.ts` —
чистый модуль без зависимостей, разбирающий узкую грамматику тем:

- ядровой токен → `oklch(L C H)` либо `oklch(L C H / A)`, всегда абсолют (это
  уже гарантирует `contract.test.ts`);
- роль → `var(--x)` либо `oklch(from var(--x) ‹c1› ‹c2› ‹c3› [/ ‹a›])`;
- компонент → число, либо `l` / `c` / `h`, либо `calc(‹l|c|h› ‹+ - * /›
  ‹число›)`;
- ссылки разрешаются рекурсивно (роль на роль встречается:
  `--color-primary-background: var(--color-text-primary)`), циклов нет — их
  ловит `contract.test.ts`.

Цветовой науки для основного гейта не нужно: сравниваются **координаты OKLCH** —
ровно то, что вычисляет браузер. Перевод в sRGB, отсечение по охвату и
отрисовка одинаковы для одинаковых координат. Округление до 6 знаков снимает
шум плавающей точки (`0.93 - 0.01 = 0.9199999999999999`).

Модуль дополнительно отдаёт перевод OKLCH → sRGB и относительную яркость — это
нужно только правилу контраста при сведении значений.

`src/themes/resolved.test.ts` разрешает 4 темы × 73 роли в снапшот.
`_template.css` исключён: там `unset`, цветов нет.

**Главный риск гейта — не сдвиг цвета, а ошибка в самом вычислителе:** неверный
разбор даст одинаково неверный результат до и после и покажет ложную зелень.
Поэтому вычислитель получает собственные тесты на каждую форму грамматики со
значениями, посчитанными вручную. Вторая страховка встроена в снапшот: у
плоских ролей (`var(--ядро)`) разрешённое значение обязано совпадать с
литералом ядра дословно, и это сверяется при чтении первого снапшота.

## Порядок работ

Снапшот фиксируется **до** любой правки тем, отдельным коммитом. Тогда каждый
следующий коммит доказывает отсутствие случайного сдвига самим фактом: у
коммита точного перевода снапшот в изменениях отсутствует, а у коммита со
сведением значений меняются ровно заявленные строки.

1. вычислитель + его тесты + снапшот 4 × 73; темы не трогаются;
2. `sepia`: `--route-N` → `--finger-N`; снапшот не меняется;
3. `light`;
4. `dark`;
5. `nord`;
6. доки: критерий честного вывода и правило сведения в `src/themes/CLAUDE.md`,
   шапки трёх тем (снять «сокращение — отдельная задача»), ADR 0034.

Шаги 3–5 независимы друг от друга.

Про имя `--finger-N`: оно совпадает с именем роли `--color-finger-N`, но это не
тот же дефект, что `--primary-hover`. `finger` — слово предметной области
(`FingerId` в глоссарии), а не роль CSS: в ядре лежит оттенок пальца-владельца,
а `route` / `target` / `keycap-group` — его ролевые проявления. Совпадение
осознанное и записывается в `src/themes/CLAUDE.md`.

## Проверка

Гейт каждого шага: `make check-dev` зелёный **и** снапшот не потребовал
обновления сверх заявленного в шаге. Перед финалом — `make check-all`
(включает `make spell`).

Визуальная проверка в живом приложении в гейт не входит: правила выше
машинные, а исполнители витрину и приложение не видят (dumb UI, ADR 0013).
Снапшот остаётся журналом — по изменениям коммитов видно каждый сдвинутый
цвет и его обоснование.

## ADR

Заводится **0034**. Не поправка к 0029: тот установил два слоя, а здесь
вводятся новое правило (критерий честного вывода роли) и новый механизм
(вычислитель ролей как гейт цветового равенства). Это решение, а не уточнение
формулировки. ADR 0028 / 0031 (цвет маршрута, оттенки пальцев) не затрагиваются.

## Что решено НЕ делать

- **Не менять словарь ролей.** 73 имени остаются как есть; работа только со
  слоем ядра и записью значений ролей.
- **Не трогать компоненты.** Ни одной правки в `src/components/**` — они
  ссылаются на роли, а роли не переименовываются.
- **Не пересматривать значения `sepia`,** кроме переименования `--route-N`.
- **Не вводить постоянный тест контраста.** Он потребовал бы сопоставления
  «роль → её фон», то есть новой сущности на поддержке. Контраст считается по
  месту, при разборе конкретного сведения значений.
- **Не сводить значения через границу WCAG** — см. правило выше.

## Область неопределённости

- Сколько именно пар пройдут порог ΔE ≤ 0.05 и не упрутся в контраст, заранее
  неизвестно: список составится при разборе каждой темы. Итоговый размер ядра
  (18–22) — оценка по осмотру, не обязательство.
- `dark` осмотрен только по списку имён ядра (40 токенов, самое большое из
  четырёх). Возможно, там найдутся связи, не встречавшиеся в `light` и `nord`;
  критерий на этот случай уже сформулирован, но объём шага 4 может оказаться
  больше соседних.
