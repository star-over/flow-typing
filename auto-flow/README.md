# auto-flow/ — мастерская Auto-Flow

«Мастерская» подготовки данных для адаптивной системы **Auto-Flow**: всё, что **готовит данные**, но не входит в рантайм приложения. Тренажёр в `src/` это не импортирует — здесь живут инструменты сборки данных, запускаемые отдельно.

Зачем и каким языком — [`CONTEXT.md`](../CONTEXT.md) (глоссарий) и [`docs/adr/`](../docs/adr). Что и в каком порядке строить — [`docs/plans/auto-flow.md`](../docs/plans/auto-flow.md).

## Структура

```
auto-flow/
├── corpus/                  конвейер корпуса (чистые шаги + тесты)
│   ├── pipeline.ts          сборка: clean → segment → фильтры → дедупликация → meta → coverage
│   ├── clean.ts segment.ts filters.ts meta.ts coverage.ts
│   ├── types.ts             DrillRecord (форма таблицы drills) + CoverageReport
│   └── corpus.test.ts
├── scripts/                 запускаемые скрипты (build-corpus и далее)
├── data/                    тексты-источники и выгрузка корпуса — gitignored
├── string-normalization.ts  нормализация текста (общая для мастерской)
└── README.md
```

## Что НЕ здесь (прибито фреймворками)

- **Схема и серверные функции** — в `convex/` (Convex включает в сборку только этот каталог): таблица `drills`, выдача порции, приём сводки.
- **Рантайм-клиент тренировки** — в `src/` (SvelteKit строит приложение из `src/`).
- **Данные раскладок** — в `src/data/layouts/*.jsonl` (их грузит сам тренажёр). Конвейер их **читает по пути**, не импортирует.

## Конвейер корпуса

Чистая функция `тексты + набор символов раскладки → записи drills + отчёт покрытия`. Раскладка — **параметр**: единица годится, только если все её символы входят в набор символов раскладки (членство заменяет определение языка). Подробности шагов — в заголовках файлов `corpus/`.

Запуск (скрипт-оболочка — следующий шаг): читает XML-источник, гонит через конвейер, пишет выгрузку в `data/`, заливает в Convex через `convex import`.
