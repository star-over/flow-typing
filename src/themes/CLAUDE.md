# Темы и роли

Полный гид и алгоритм добавления компонента — `docs/06-component-contracts-and-themes.md`.

Тема — **два слоя**: приватное **ядро** (L1) + семантические **роли** (L2, `--color-*`). Компонент ссылается на роли напрямую (`var(--color-*)` в своём `<style>`), а геометрию (рамка/кольцо/тень: числа width/offset/blur/spread, `solid`/`dashed`, `inset`, единицы) держит литералом прямо в CSS. Отдельного слоя компонентных контрактных токенов больше нет — растворён (ADR 0029).

Источник истины словаря ролей — `src/themes/roles.ts → ROLE_DICTIONARY` (73 имени, эталон — `sepia`). Контракт-тест `src/themes/contract.test.ts` enforce-ит двухслойную модель: каждая тема (`src/themes/<id>.css`) и `_template.css` декларируют каждую роль словаря (значения свободны), роли ссылаются только на роли/ядро без дублей имён и циклов `var()`, а компоненты — только на объявленные `--color-*`.

Темы в `src/themes/`:
- `light` / `sepia` (colorScheme=light), `dark` / `nord` (colorScheme=dark). Каталог — `THEMES` в `src/themes/registry.ts`.
- **Двухслойная схема (эталон — `sepia.css`).** Тема делится на две секции, граница между ними — правило, а не оформление. **Слой читается из префикса имени для каждого токена** (инвариант, ADR 0029): голое имя = ядро (L1); `--color-*` = роль (L2). Слоя контракта (`--<компонент>-*`) больше нет. Индустриальный маппинг: ЯДРО = primitive/option tokens · РОЛИ = semantic tokens.
  1. **Ядро** — приватная палитра темы. Имена и количество произвольны (в `sepia` — `--parchment`, `--ink`, `--amber`…), значения **только абсолютные**. Единственное место, где в теме появляется новый цвет. Имена цветовые и это честно: внутри темы они описывают содержимое, а не роль.
  2. **Роли** — семантика, **фиксированный словарь имён** (`--color-*`, общий для всех тем). Каждая роль — маппинг/калькуляция от ядра **или от другой роли** (`oklch(from …)` — тюнинг светлоты/хромы/оттенка); новых абсолютов нет (ADR 0029). Нужна калькуляция цвета — значит роли не хватает; единственный whitelist цветового литерала — `transparent`.
- **Критерий честного вывода (ADR 0034).** `oklch(from var(--ядро) …)` легален, только если выполнены оба условия: (1) минимум один компонент реально наследуется — записан как `l` / `c` / `h` либо `calc(l …)` / `calc(c …)` / `calc(h …)`, а не литералом; (2) источник — тот токен, чьей ступенью или отзвуком роль является по смыслу (ступень лестницы поверхностей, отзвук оттенка пальца, `hover` от своей базы), не «ближайший подходящий». Запись всех трёх компонентов литералами запрещена — она даёт то же число, но лжёт про связь. Не прошедшее критерий значение остаётся в ядре под именем по содержимому. **Вывод раньше сведения:** где точный вывод возможен, применяется он, а не сведение близких значений (сведение стирает различие безвозвратно, вывод — нет).
- **Суффикс-словарь ролей** (источник истины именования, ADR 0029): `-foreground` — текст/символ на плотной заливке; `-hover` — hover-состояние роли; `-dim` — цвет приглушённой полупрозрачной поверхностью (альфа над фоном); `-strong` / `-muted` — усиленная / приглушённая ступень; `-raised` — приподнятая поверхность; `-accent` — тёплый / акцентный вариант; `-border` — граница.
- **Правило именования (ADR 0029).** Роль именует **семантику** (`background` / `foreground`), а не CSS-свойство элемента — слой при этом несёт префикс `--color-*`; геометрия/структура живёт литералом в CSS компонента, не токеном. Аббревиатуры раскрыты: `bg`→`background`, `fg`→`foreground`.
- **Идиома правки светлоты в `oklch(from …)`.** Сейчас в коде смешаны мультипликативная (`calc(l * .90)`) и аддитивная (`calc(l - 0.08)`) формы. Для **новых** ролей — аддитивная дельта (`calc(l - X)`) как более читаемый дефолт. Свод ядра (задачи консолидации, ADR 0034) уже переписал существующие значения в `light`/`dark`/`nord` там, где сведение или честный вывод того требовали — оговорка «задним числом не переписываются» больше не действует для этих трёх тем; дальнейшая унификация формы (мультипликативная vs аддитивная) по-прежнему вопрос визуальной доводки, не консолидации.
- **Гейт цветового равенства (ADR 0034).** `src/themes/resolved.test.ts` держит **значение** — координаты OKLCH всех 73 ролей × 4 тем в снапшот-файле; `contract.test.ts` держит **форму** — каждая роль объявлена, ссылки только на роли/ядро, без циклов. Вместе они закрывают обе половины: рефакторинг ядра, структурно корректный, но незаметно сдвинувший цвет, ловит снапшот. При любой правке ядра — снять `make theme-report THEME=<id>` (координаты, близкие значения с ΔE, контраст ролей к подложкам, включая композитные) до и после, приложить числа в комментарий рядом со значением.
- **Свод ядра завершён:** все темы держат честное ядро (ADR 0034) — `sepia` 11 токенов (эталон, только переименование), `light` 17, `dark` 17, `nord` 15 (было 35/40/35 механически перенесённых); словарь ролей выровнен до 73 имён (`ROLE_DICTIONARY`); форму слоёв держит `contract.test.ts`, значение — `resolved.test.ts`.
- **Визуальная модель ADR 0028 перенесена во все темы.** Все темы несут маршрут и цель по пальцу; оттенки пальцев едины (ADR 0031, база `sepia`: h72/200/250/295/340), тема выводит только светлоту/хрому и архитектуру маршрут/цель. На тёмных темах (`dark`/`nord`) архитектура инверсна — цель это светлая заливка с тёмным символом (устраняет прежнюю просадку контраста мид-индиго-цели). Числа — расчётом (охват sRGB, WCAG, ΔE), не на глаз.
- **Имя ядрового источника оттенка пальца — `--finger-N` во всех темах** (в `sepia` было `--route-N` до переименования). Совпадение с именем роли `--color-finger-N` — единственное осознанное исключение из запрета «ядро не называется по роли»: `finger` — слово предметной области (`FingerId` в глоссарии `CONTEXT.md`), а не роль CSS. В ядре лежит оттенок пальца-владельца сам по себе; `route` / `target` / `keycap-group-N` — его ролевые проявления на разных элементах сцены.
- **Слой контракта растворён (ADR 0029) — сделано.** Компоненты привязаны к ролям L2 напрямую; файлы `*.contract.ts`, `src/themes/contract.ts` (`THEME_CONTRACT`) и слой `--<компонент>-*`-токенов в темах удалены. Дальнейшая работа по темам — визуальная доводка пер-темных цветов (см. `docs/backlog.md`, «Темизация — визуальная доводка»).
- `_template.css` — скелет для новой темы: каждая роль задана как `unset`, нужно только заполнить ядро и значения ролей.
- Bootstrap синхронный — inline-script в `src/app.html` выставляет `data-theme` до первой отрисовки сцены. View Transitions API даёт crossfade при смене темы (`src/themes/registry.ts → setTheme`).
