# Компоненты (dumb UI)

Механику токенов/тем НЕ дублируем — `src/themes/CLAUDE.md`; ViewModel — `src/lib/CLAUDE.md`; полный алгоритм — `docs/06-component-contracts-and-themes.md`.

- **Duo-конвенция.** Компонент = `X.svelte` + опциональный сосед `X.stories.svelte` (svelte-csf). Отдельного `X.contract.ts` больше нет — темизация идёт через роли `--color-*` напрямую (ADR 0029). Duo неполное — норма, а не пробел: не каждому компоненту нужны stories.
- **Проверка — Storybook, не vitest (ADR 0013).** В `src/components/**` намеренно **0** `*.test.ts`: dumb UI логики не содержит — рендерит ViewModel как есть. Любая логика выносится за чистые швы в `.ts` (`src/lib`, `shared`) и тестируется там. **Не вводить** jsdom / browser-mode / `@storybook/test` play-функции; `make check-all` Storybook не гоняет. Это зафиксированное решение, не пробел — авто-проверки компонентов не пере-предлагать.
- **i18n (ADR 0022).** Ни одного строкового литерала-надписи в разметке — только `$dictionary.<секция>.<ключ>` или пришедшая параметром локализованная строка (непереводимое — бренд, имена провайдеров — тоже из словаря).
- **Добавить темизируемый компонент** — пошаговый алгоритм в `docs/06` §6.7: перечислить элементы/состояния → на каждое подобрать существующую роль в `ROLE_DICTIONARY` (`src/themes/roles.ts`) и сослаться из `<style>` через `var(--color-*)` (геометрию — литералом здесь же); нет подходящей роли — завести новую (имя в `ROLE_DICTIONARY` по §6.3 → значение в каждой теме `src/themes/registry.ts` + `unset` в `_template.css`) и сослаться на неё → `make check-all` (contract-test обязан пройти).
- Подпапки: `ui/` — generic-примитивы (доменно-нейтральные листья: `Avatar`, `KeyHint`, `MessageScreen`, `Select`, `Wordmark` — сюда не класть фичи экрана); `app/` — оболочка приложения (корень `App` + `MainContent` + `FooterActions`); `hands-scene/` несёт render-геометрию; остальные — по областям экрана (`settings/`, `auth/`, `train/`, `header/` …).
