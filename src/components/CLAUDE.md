# Компоненты (dumb UI)

Механику токенов/тем НЕ дублируем — `src/themes/CLAUDE.md`; ViewModel — `src/lib/CLAUDE.md`; полный алгоритм — `docs/06-component-contracts-and-themes.md`.

- **Trio-конвенция.** Темизируемый компонент = три соседа: `X.svelte` + `X.contract.ts` (массив имён `--токенов` как `as const` + экспортируемый тип `XContractToken`) + `X.stories.svelte` (svelte-csf). Trio неполное — норма, а не пробел: не каждый компонент темизируем и не каждому нужны stories.
- **Проверка — Storybook, не vitest (ADR 0013).** В `src/components/**` намеренно **0** `*.test.ts`: dumb UI логики не содержит — рендерит ViewModel как есть. Любая логика выносится за чистые швы в `.ts` (`src/lib`, `shared`) и тестируется там. **Не вводить** jsdom / browser-mode / `@storybook/test` play-функции; `make check-all` Storybook не гоняет. Это зафиксированное решение, не пробел — авто-проверки компонентов не пере-предлагать.
- **i18n (ADR 0022).** Ни одного строкового литерала-надписи в разметке — только `$dictionary.<секция>.<ключ>` или пришедшая параметром локализованная строка (непереводимое — бренд, имена провайдеров — тоже из словаря).
- **Добавить темизируемый компонент** — пошаговый алгоритм в `docs/06` §6.7: перечислить элементы/состояния → назвать токены (§6.3, визуальные роли) → создать `X.contract.ts` → влить в `src/themes/contract.ts` (`THEME_CONTRACT`) → объявить токены в каждой теме (каталог — `src/themes/registry.ts` + `_template.css`) → потреблять через `var()` → `make check-all` (contract-test обязан пройти).
- Подпапки: `ui/` — generic-примитивы (доменно-нейтральные листья: `Avatar`, `Select`, `Wordmark` — сюда не класть фичи экрана); `app/` — оболочка приложения (корень `App` + `MainContent` + `FooterActions`); `hands-scene/` несёт render-геометрию; остальные — по областям экрана (`settings/`, `auth/`, `train/`, `header/` …).
